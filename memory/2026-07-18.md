## The lesson's spine

Events answer "what happened"; gameplay usually asks "what is". The lesson opens with the
failure: movement driven by `SDL_EVENT_KEY_DOWN` inherits the OS **key repeat** — one move,
~0.5 s of nothing, then lurching. Worked out numerically: 16 movements in the first second
versus 60 when polling, and since repeat rate is a user preference (down to ~2/s), two players
on identical hardware get an 8× speed difference.

Resolution: **poll levels, derive edges.**

```
down     =  keys_[k];
pressed  =  keys_[k] && !keys_prev_[k];
released = !keys_[k] &&  keys_prev_[k];
```

One source of truth, two snapshots, three answers — so `key_down` and `key_pressed` cannot
contradict each other, and queries are pure (asking does not consume an edge).

Also covered: scancode vs keycode (the AZERTY bug, with a keyboard diagram showing WASD's
positions spelling ZQSD); the drain-then-sample ordering rule; and the honest limits — a tap
between samples is invisible, and the wheel/text have no pollable state at all.

## Verified, not assumed

Checked against SDL headers at the pinned `release-3.4.12`:

- `const bool *SDL_GetKeyboardState(int*)` — **bool** in SDL3, not SDL2's `Uint8`.
- `SDL_SCANCODE_COUNT = 512`; letters run from `A = 4`, so `W = 26` (used in the worked example).
- `SDL_GetMouseState` returns `SDL_MouseButtonFlags`, writes `float` x/y; `SDL_BUTTON_MASK(n)`.
- `event.key.{down,repeat}` are `bool`; `event.wheel.{x,y}` are `float` + a `direction` flag.
- `SDL_PollEvent` → `SDL_WaitEventTimeoutNS(e,0)` → `SDL_PumpEventsInternal`, i.e. polling pumps.
- Traced `SDL_SetKeyboardFocus` → `SDL_ResetKeyboard` in `src/events/SDL_keyboard.c:350`: the
  stuck-key-after-alt-tab fix is delivered **as key-up events**, so it only reaches you if you
  drain before sampling. That turned an ordering convention into a demonstrable consequence.

## Two traps found while reconciling — both silent

1. **A naive list union would have been a 12-page regression.** `01-02` filed
   `bool char int long unsigned void` under `CPP_KEYWORDS`; the template filed them under
   `CPP_TYPES`. The tokeniser checks `kw` before `ty`, so merging both lists verbatim recolours
   every fundamental type from `.tok-t` to `.tok-k`. Kept the template's classification, folded in
   only what was genuinely missing (`defined`, `override`, `final`, the SDL `Uint*`/`Sint*` types).
   Verified in-browser: `int`/`bool`/`Uint8`/`size_t` → `tok-t`, `struct`/`constexpr` → `tok-k`.
2. **`01-02`'s `::` shell-comment rule was unanchored** and matched mid-line, so
   `cmake --build . --target SDL3::SDL3 --config Release` rendered as a comment from `::` onward.
   `SDL3::SDL3` is *the* link target in this course. Re-anchored to `(^|\n)(\s*)(::[^\n]*)` and
   renumbered the capture groups; unit-tested the tokeniser in `node` (8 cases, invariant = tags
   stripped must reproduce the input exactly) before stamping it into 12 files.

Net effect beyond de-duplication: every page now gets the HLSL keyword set and the
`lang-hlsl`/`lang-c`/`lang-sh` dispatch that only the template had — which Module 4 needs — and
`00-03`'s Windows-batch `::` lines highlight correctly for the first time.

`SHARED-SCRIPT` markers (I added them) but **never had `SHARED-CSS` markers** — those had been

## The lesson's spine

1.2's box moved 6 px per *frame*, so its speed was a property of the monitor (180 / 360 / 864 px/s
at 30 / 60 / 144 fps — the 144 Hz player's game is 4.8× faster). The fix is one multiplication,
and the lesson is mostly about the fix's limits.

Three measurement decisions first: **monotonic** not wall (a wall clock corrected by NTP hands you
a *negative* dt), **nanoseconds** not milliseconds (±20% measurement error at 300 fps; truncates
to zero above 1000 fps), and **`Uint64` not `float`** for absolute time.

Then the payoff and its boundary. `Σ(v·dtᵢ) = v·T` is exact — but only because `v` factors out of
the sum. Anything accelerating is integrated with explicit Euler, whose error is `½·g·T·h`:
**proportional to the step**, so the frame rate remains an input to the physics. That single fact
is the entire argument for 1.4, and the lesson derives it rather than asserting it.

Closes by pricing each naive patch (clamp / floor / smooth / cap) — none restores determinism —
and naming the spiral of death (cost per step ÷ time simulated per step > 1).

## Verified, not assumed

- Traced `SDL_GetTicksNS` → `SDL_GetPerformanceCounter` → `CLOCK_MONOTONIC_RAW` (the Unix source
  comments that it chose that clock as "not subject to adjustment by NTP") / `mach_absolute_time` /
  `QueryPerformanceCounter`. The header never says "monotonic"; the implementation does.
- `SDL_Delay`/`SDL_DelayNS` wait *at least* the requested time; `SDL_DelayPrecise` busy-waits to
  get closer. Measured `SDL_Delay(10)` ≈ 11.8 ms in our own harness, which is the doc comment
  demonstrating itself.
- `SDL_SetRenderVSync`, `SDL_RENDERER_VSYNC_DISABLED`, `SDL_RenderDebugTextFormat` (8×8 ASCII
  bitmap font, current draw colour), `SDL_SetRenderScale`.
- **The float-freeze claim was compiled and run**, not argued: `86400.0f + (1.0f/500)` evaluates to
  exactly `86400.0f`. ulp(86400.0f) = 7.8 ms.
- Every worked number checked in Python first: the 30/60/144 fps table, the dt-cancellation table,
  and the Euler error table (0, 2.5, 3.75, 4.375, 4.9167 m) against the closed form `g·T·(T−h)/2`.

## The lesson's spine — three broken attempts, in order

Deliberately structured as a derivation rather than a presentation, because each naive fix breaks
something new and that sequence *is* the understanding:

1. **Constant step, no accumulator.** Physics is now identical everywhere — and the game runs at
   (fps ÷ 60)× speed. Same 4.8× spread between 30 and 144 fps that 1.3 opened with, just relocated.
2. **The accumulator.** Bank real time, spend whole steps, carry the change. Correct speed *and*
   identical physics. Invariant: after the loop, `0 <= accumulator < h`.
3. **…and it judders.** The render is stale by the accumulator, which *varies* 0→h. Derived
   numerically: at 100 fps / 60 Hz the staleness sawtooths 10.0, 3.33, 13.33, 6.67, 0.0 ms with
   period 5 (`0,1,0,1,1` steps per frame).
4. **Interpolation.** `alpha = accumulator / h`; draw `lerp(previous, current, alpha)`. Proof that
   this renders at exactly `T − h`: the accumulator cancels. A lag swinging 0→h becomes a constant
   h. **Smoothness is consistency, not immediacy** — on average interpolation is *further* behind.
5. **The spiral guard**, plus honesty about what it costs (permanent slow motion, reported).

Closes with choosing `h`, and a warning that fixed-timestep determinism is same-binary/same-machine
only — FMA contraction, x87 intermediates, `libm`, vectorisation. Necessary for lockstep, nowhere
near sufficient.

## Verification

`fixed_step` driven through a 15-assertion harness: the 7-steps-over-12-frames pattern; `alpha`
inside `[0,1)` across 700 irregular frames including zero-length ones; sim time trailing real time
by under one step after 6 s; a worst-case 0.25 s frame capping at 8 steps and reporting exactly
116.67 ms dropped; 100 consecutive one-second frames still terminating at 8 steps each; `set_rate`
preserving banked time; rate 0 and NaN clamped rather than producing an unterminating loop.

The demo's payoff modelled before coding: with a 60 Hz step the ball's apexes are **236, 204, 171,
137 at 2000, 240, 60 and 20 fps — identical**. Changing the *sim* rate does change them (10 Hz: 56,
−196), which is the integrator error, now under our control.

Build warning-clean at `-Wall -Wextra`; runs and shuts down cleanly.

## Widget note worth keeping

The interpolation slider reported `raw lag: 0.0–0.0 ms` at 15/30/60 fps, which looks broken and is
correct: a render rate dividing evenly into the sim rate has no judder at all. That is *precisely
why this bug survives testing at 60 fps*, so the widget now says so instead of looking wrong. Also
clamped a `-0.0` produced by float error in a quantity the invariant says is non-negative.

## Decisions

- **`fixed_step` is a class, not four lines in the loop.** The accumulator has an invariant
  maintained by exactly one subtraction; inline, that subtraction is one `continue` from being
  skipped and the failure is an infinite loop *inside a frame*.
- **On hitting the step cap, drain the excess** rather than just returning — otherwise `alpha > 1`
  and the renderer silently extrapolates exactly when the machine is struggling.
- **`clamp_rate` written as `if (!(rate >= min))`** so NaN is caught (every NaN comparison is false).
- **The demo box bounces, never wraps.** Wrapping is a teleport, and interpolating a teleport draws
  the box sliding backwards across the screen. Made it Exercise 1.4.4 instead.
- **Demo state moved into a `sim_state` struct** — interpolation forces it, since "the previous
  value of everything" is only expressible if "everything" has a name. Flagged in the lesson as a
  preview of the pressure that produces the ECS in Module 5.
