## Verified against primary sources (not recalled)

§10 forbids guessing SDL details, so these were checked against the SDL3 wiki and
`raw.githubusercontent.com/libsdl-org/SDL/main/include/SDL3/*.h`. **SDL on `main` is 3.5.0.**

| Fact | Value |
|---|---|
| SDL_GPU NDC | lower-left `(−1,−1)`, upper-right `(1,1)` → **+Y up** |
| SDL_GPU depth | **z ∈ [0,1]**, 0 = near |
| Viewport / UV | top-left `(0,0)`, **+Y down** (not a contradiction — different spaces) |
| Vulkan Y-flip | SDL converts behind the scenes; no shader flip |
| `SDL_GPUFrontFace` | `{COUNTER_CLOCKWISE = 0, CLOCKWISE}` |
| `SDL_GPUCullMode` | `{NONE = 0, FRONT, BACK}` |
| `SDL_CreateWindow` | `(const char *title, int w, int h, SDL_WindowFlags flags)` — no x/y |
| SDL_shadercross | real and maintained, but **3.0.0-preview** — flag its pre-1.0 status |

**Key finding:** winding is *not* an SDL-wide convention — it is per-pipeline state, and both
enums default to zero, so a zero-init rasterizer state means "CCW front, cull nothing." A
forgotten `cull_mode` therefore shows up as *no culling*, silently. All recorded in LEARNINGS.md.

## Bugs found and fixed during verification

1. **Theme toggle was dead.** The JS set `data-theme` but the CSS only had
   `@media (prefers-color-scheme: dark)`. Fixed by duplicating the dark token block under
   `:root[data-theme="dark"]` and scoping the media query to `:not([data-theme="light"])`.
   *The in-app preview pane could not detect this* — it reports impossible computed styles in
   its static-snapshot mode. Confirmed the fix in real Chromium via Playwright over
   `python3 -m http.server`, since Playwright blocks `file:`.
2. **CSS silently killed every diagram colour.** `figure.dia svg text { fill: … }` is CSS and
   therefore beats any inline `fill="var(--dia-hi)"` presentation attribute on a `<text>`. Would
   have drained the colour from every diagram in all 94 lessons. Fixed with `.t-hi`/`.t-ok`/
   `.t-bad` utility classes (class selectors outrank the bare element selector). Shapes are
   unaffected — the rule only targets text.
3. Label collisions in the axis diagram; a broken `&#8407;` glyph in the matrix figure.

## The design problem in 0.1, and how it was resolved

§3.9 demands every lesson end runnable, but 0.1 sits *before* the toolchain (0.3) and CMake
(0.4) — the student has no compiler. Inventing C++ would have broken §8 ("every listing compiles
at its point in the course") far more seriously than having no C++.

**Resolution:** 0.1's runnable artifact is an inline **frame-budget explorer** (vanilla JS, in
the page), and the lesson states plainly why there is no C++ yet. The Complete Code Listings
section honestly says "no files, and here is why". This should happen exactly once.

## The lesson's spine

The whole anatomy is *derived* from one number rather than listed:

- Frame budget = `1000 / refresh`. 60 Hz → **16.67 ms**.
- **The vsync cliff:** performance is quantized. `periods = ceil(work / budget)`,
  `effective_fps = refresh / periods`. So 17 ms at 60 Hz → 30 FPS, not 58. Being 2% over costs
  50% of the framerate.
- The six subsystems fall out of six questions a frame must answer.
- **The boundary rule:** a boundary belongs where a *rate* changes or an *owner* changes. Nothing
  else. This rule is reused for the rest of the course (it predicts Godot's `servers/`, and it
  settles the scene-vs-renderer merge argument in exercise 0.1.4).
- Thesis, stated in lesson 1 and repaid through Module 8: every strange thing in engine
  architecture is an answer to "how do we finish in 16.67 ms, *every* time?"

## Errors caught by verification (all were mine, in prose I'd written)

Checked every numeric claim with a Python script rather than by eye. Three real bugs:

1. **"the difference between 60 and 30 FPS is 8.3 ms"** — flatly wrong; it is **16.67 ms**.
   Replaced with a correct and stronger version: a 30-FPS drop costs 16.67 ms at 60→30 but only
   0.37 ms at 300→270 — same headline, 45× less work.
2. **"average to 16 ms — '60 FPS'"** — 16 ms is **62.5 FPS**. Corrected to 62.
3. **"nudge renderer up by a single millisecond and watch the cliff"** — +1 ms gives 16.5 ms,
   which still *fits*. The flip needs +1.2 ms (render 8.4 → total 16.70). Rewrote to name the
   exact notch, then verified in-browser: 16.60 → 60 FPS, 16.70 → 30 FPS.
4. Visual: `.budget-bar { overflow: hidden }` clipped the budget line's "BUDGET" label. Fixed
   with an inner `.budget-track` that clips while the outer box stays open.

**Lesson for future sessions: verify worked examples computationally, and drive the widget to
confirm it reproduces the hand-worked exercise.** The widget's defaults deliberately *are*
exercise 0.1.2's numbers (15.5 ms), so the two can be cross-checked — and were.
