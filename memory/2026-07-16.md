## The lesson's spine

The whole anatomy is *derived* from one number rather than listed:

- Frame budget = `1000 / refresh`. 60 Hz → **16.67 ms**.
- **The vsync cliff:** performance is quantized. `periods = ceil(work / budget)`,
  `effective_fps = refresh / periods`. So 17 ms at 60 Hz → 30 FPS, not 58. Being 2% over costs
  50% of the framerate.
- The six subsystems fall out of six questions a frame must answer.
- **The boundary rule:** a boundary belongs where a *rate* changes or an *owner* changes. Nothing
  else. This rule is reused for the rest of the course (it predicts Godot's `servers/`, and it
  settles the scene-vs-renderer merge argument in exercise 0.1.4).
- Thesis, stated in lesson 1 and repaid through Module 8: every strange thing in engine
  architecture is an answer to "how do we finish in 16.67 ms, *every* time?"

## Errors caught by verification (all were mine, in prose I'd written)

Checked every numeric claim with a Python script rather than by eye. Three real bugs:

1. **"the difference between 60 and 30 FPS is 8.3 ms"** — flatly wrong; it is **16.67 ms**.
   Replaced with a correct and stronger version: a 30-FPS drop costs 16.67 ms at 60→30 but only
   0.37 ms at 300→270 — same headline, 45× less work.
2. **"average to 16 ms — '60 FPS'"** — 16 ms is **62.5 FPS**. Corrected to 62.
3. **"nudge renderer up by a single millisecond and watch the cliff"** — +1 ms gives 16.5 ms,
   which still *fits*. The flip needs +1.2 ms (render 8.4 → total 16.70). Rewrote to name the
   exact notch, then verified in-browser: 16.60 → 60 FPS, 16.70 → 30 FPS.
4. Visual: `.budget-bar { overflow: hidden }` clipped the budget line's "BUDGET" label. Fixed
   with an inner `.budget-track` that clips while the outer box stays open.

**Lesson for future sessions: verify worked examples computationally, and drive the widget to
confirm it reproduces the hand-worked exercise.** The widget's defaults deliberately *are*
exercise 0.1.2's numbers (15.5 ms), so the two can be cross-checked — and were.

## The MSVC `__cplusplus` trap (worth remembering when SDL arrives)

MSVC reports `__cplusplus == 199711L` (C++98) by **default**, regardless of `/std:c++20`. Two
fixes, both needed: pass `/Zc:__cplusplus` (libraries incl. SDL sniff the macro), AND read
`_MSVC_LANG` in your own code (always honest). `hello.cpp` uses `_MSVC_LANG` and prints both so the
gap is visible.

## SDL3 API verified against release-3.4.12 headers (added to LEARNINGS.md)

All the SDL2→SDL3 traps, confirmed by fetching the actual headers (NOT guessed):
- `bool SDL_Init(SDL_InitFlags)` and `bool SDL_PollEvent(SDL_Event*)` — **bool**, not SDL2's int.
- `SDL_CreateWindow(const char*, int w, int h, SDL_WindowFlags)` — **no x/y**.
- `SDL_CreateRenderer(SDL_Window*, const char* name)` — name=nullptr for default (SDL2 was index+flags).
- Keyboard: **`event.key.key`** (SDL_Keycode) / `event.key.scancode` / `.down` / `.repeat` —
  SDL2's `event.key.keysym.sym` is GONE.
- `<SDL3/SDL.h>` does **NOT** include `<SDL3/SDL_main.h>` — must include separately, once, in the
  main file (or Windows link fails on WinMain). Standard entry sig `int main(int argc, char* argv[])`.
- `SDLK_ESCAPE`=0x1b; `SDL_WINDOW_RESIZABLE`=0x20; `void SDL_Log(const char*, ...)`;
  `const char* SDL_GetError(void)`; draw/clear/present all return bool; `SDL_DestroyRenderer/Window`
  return void.
- Design choice: classic `int main` + our own `while` loop, NOT the SDL_AppInit/SDL_AppIterate
  callback model — **the engine owns its loop** (0.1 thesis). Noted the callback alt in Further Reading.
