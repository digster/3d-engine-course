## The lesson's spine

**Nothing had to be re-derived.** Lesson 2.5's argument never counted the axes, so all of it held
and this lesson is transcription plus one genuinely new idea (rotation needs an axis) and one
deliberate cliffhanger (the 4×4's fourth column does nothing).

Structure: build `mat3`, show it still cannot translate for the same one-line reason, build `mat4`
to fix that, watch it fail, diagnose precisely *why*, and **stop**. STATE was explicit that
homogeneous coordinates must not be pre-empted, and they are not.

## Best things in the lesson

**The cycle fixes the rotation signs.** `x → y → z → x`: rotating about an axis turns the next two,
one toward the other. `R_y`'s minus sign is *below* the diagonal because the cycle wraps (z → x)
while the matrix lists x's row above z's. Derived, never recalled — this is the #1 sign error in
graphics.

**Non-commutativity with a point on an axis.** `(1,0,0)` is unmoved by `R_x`, so doing `R_x` first
wastes it entirely. x-first → `(0.6967, 0.0000, −0.7174)`; y-first → `(0.6967, 0.4050, −0.5921)`.
One of the four steps provably does nothing, which makes the asymmetry obvious rather than
arithmetic.

**The 4×4 failure, measured.** A correct translation in the correct column, displacement exactly
`(0,0,0)`. The demo measures it (mat4 result minus mat3 result) so the HUD cannot lie — it reads
`0.00 px` and keeps reading it however large the translation.

## Verified, not asserted (6 sections, all passing)

- `M*(3,2,1) = (8,6,4)`, det 24, floats in memory `2 0 0  1 3 0  0 0 4`, `sizeof` exact.
- `rotation_z`'s top-left 2×2 == `mat2`'s rotation over 121 angles, 0 mismatches; each rotation

- **Volume by lattice count** (120 steps/unit, inside iff `inverse(M)*p ∈ [0,1)³`): identity 0.00%,
  `rotation_y(0.7)` −0.01%, `scale(1.6,.8,1.3)` 0.00%, `Rx*Ry*scale` −0.00%.
- `M * inverse(M) == I` over 300 matrices — **this caught a real bug** (see below).
- The 4×4's c3 contributes exactly zero with `w = 0`; `to_mat4` is faithful and
  composition-preserving; with `w = 1` it is exactly a translation (noted for 2.7 only).
- 500 assorted `mat3`s, none moved the origin.

## Two real defects I hit

**`mat3::inverse` was wrong.** Written straight into column members, two of nine cofactors used the
wrong component — plausible-looking and wrong. Rewrote it to name elements in **written** notation
(`m00..m22`) first, which fixes the class of error. The `M*inverse(M)==I` check found it instantly.

**Figure 1's vector walk was invisible.** The first leg lay exactly along the x axis in dim dashed
grey, so the "a vector is a recipe" picture did not visibly build the vector. `check-page.js` was
green. Third lesson running that a diagram passed every automated check while failing to make its
argument — now recorded in LEARNINGS.md with the specific pattern: *anything drawn collinear with
or underneath something else is invisible even when geometrically correct.*

## Decisions

- **`identity()` → static member on every matrix type.** Forced: a zero-argument free function can
  only differ by return type. Everything else survived (overload on parameter / name / arity),
  which localises the rule nicely.
- **`mat2`'s uniform `scale(float)` removed** — unused, and a trap once a 3-D uniform scale is
  wanted.
- **No cross product, no `vec3::perpendicular`, no 4×4 det/inverse.** Each deferred to the lesson
  that needs it (3.4, 3.4, 2.9).
- **`to_vec4(v, w)` is explicit**, never an implicit conversion — so `w` is always chosen.
- **The cube is centred on the origin**, since rotation is always about the origin.
- **Five demos in one binary now, main.cpp past 1,600 lines.** The lesson points at it explicitly
  rather than fixing it: that is Module 5's argument and it is meant to be felt.
