## The lesson's spine

One question — **where do the basis vectors land?** — and everything is a consequence: the
multiplication rule, the rotation matrix, composition order, the determinant. Nothing is presented
as a rule to memorise, which is the whole reason this lesson matters for the seven that follow.

The proof that two arrows suffice is three lines from the two linearity rules, and it is worth the
space: `T(v) = T(xî + yĵ) = T(xî) + T(yĵ) = x·T(î) + y·T(ĵ)`.

## Best things in the lesson

**Rotation derived, not looked up.** `î → (cos θ, sin θ)` because that is what sine and cosine
*mean*; `ĵ` is a quarter turn ahead so it lands a quarter turn ahead. Then the bonus: `R(a)R(b)`
must equal `R(a+b)`, and multiplying it out **derives the angle-addition formulas**. Verified over
1,716 pairs, worst 2.98e-7; top-left of `R(.6)R(.9)` = 0.070737 = cos(1.5).

**The determinant is `edge_function`.** With the first point at the origin, the edge function from
2.2 *is* `c0.x*c1.y − c1.x*c0.y`. Verified identical over 28,561 integer matrices, 0 mismatches.
The engine has been computing determinants since it learned to draw a triangle.

**Translation left broken on purpose.** `T(origin) = origin` falls out of the scaling rule with
`c = 0`, so no 2×2 can translate. Stated, proved in one line, and deliberately not fixed — 2.7
earns the fourth component from exactly this gap, and Exercise 2.5.3 makes the student feel it.

## Verified, not asserted (7 sections, all passing)

- `M*(3,2) = (8,6)` for columns `(2,0)`, `(1,3)`; det 6; raw floats in memory `2, 0, 1, 3`.
- `R(a)R(b) == R(a+b)` over 1,716 pairs, worst element error 2.98e-7.
- `det == edge_function(origin, c0, c1)` over 28,561 matrices, 0 mismatches; det is multiplicative.
- **Area measured by rasterising**: 140-px square — identity/scale/shear exact, rotation −0.26%,
  `rot*scale` −0.06%. The residual scales with perimeter, so it falls as ~1/side.
- `R*S` vs `S*R` map `(1,0)` to `(0,2)` vs `(0,1)` — but **both have det 2**. Uniform scale does
  commute; composition is always associative.
- `det == 0`: `(1,0)` and `(−1,1)` both map to `(2,1)`, and all 1,681 sampled inputs land on the
  single line `x = 2y`. `inverse` returns zeros, never NaN.
- Reflection flips signed area `+60 → −60`; a det of 1.0200 takes `+60 → +61.2`, ratio exactly
  1.0200.

## Decisions

- **`mat2` = two `vec2` columns, not `float[4]`.** Column-major layout becomes a consequence of
  naming the right things rather than a convention to enforce — and the arithmetic can then be
  written as its own derivation (`c0*x + c1*y`; `{a*b.c0, a*b.c1}`), which the row-times-column
  form cannot.
- **No constructors on maths types** — default member initialisers keep them aggregates, preserving
  brace init, `constexpr`, and a layout guaranteed to match the declaration (needed in Module 4).
- **`at(row, col)` takes row first**, unchecked. It exists to read code against a written
  derivation. Different trade from `put_pixel`: check where the input can actually be wrong.
- **The y-flip lives in one function** (`to_screen`), not baked into the matrices — otherwise
  "which way does `rotation()` turn?" is permanently ambiguous.
- **The glyph is an F.** A blob cannot show a reflection, and reflection is the point of the sign.
- **Four demos in one binary now.** Well past awkward; that is the Module 5 argument, loud.
