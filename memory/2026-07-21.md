## The lesson's spine

Not "here are two algorithms". The transferable idea is the **scaling step**: the error term's
only uses are a comparison against ½ and additions, and comparisons survive multiplication by a
positive constant — so multiply by `2·dx` and every fraction disappears. Bresenham is the worked
example; the technique returns as fixed-point sub-pixel precision and as depth encoding.

Two things were derived rather than asserted, and both are short enough to be worth it:

- **Termination.** With `dx = |Δx| ≥ 0` and `dy = −|Δy| ≤ 0`, both tests failing requires
  `dx < 2·err < dy ≤ 0 ≤ dx`, i.e. `dx < dx`. One test always fires.
- **A tie theorem.** With `p = major / gcd(major, minor)`, an exact tie exists **iff `p` is
  even** — and exactly those lines are asymmetric under endpoint swap. Verified exhaustively:
  23,103 lines, 7,692 asymmetric, zero disagreements with the prediction in either direction.
  This is what makes 2.2's fill rule necessary rather than fussy, so it doubles as the bridge.

The all-octant fusion was *not* re-derived — that is sign bookkeeping that teaches nothing. Flagged
as a simplification per §10 and backed by an exhaustive check against a reflect-in/reflect-out
reference over 1600 lines.

## The benchmark inverted the textbook, and that became the lesson's best section

Received wisdom: Bresenham is faster because integers beat floats. Measured on an M4 Pro
(clang 21, `-O2`, ns/pixel, stepping only, framebuffer write excluded):

| | all octants | shallow | steep | 45° |
|---|---|---|---|---|
| Bresenham, compact (2 branches) | 1.320 | 1.247 | 1.251 | 1.248 |
| Bresenham, major-axis (1 branch) | 0.738 | 0.781 | 0.776 | 0.788 |
| DDA (`lround`) | **0.595** | **0.648** | **0.643** | 0.660 |
| DDA (`+0.5`, truncate) | 0.645 | 0.667 | 0.667 | **0.630** |

**DDA is ~2.2× faster.** Two controls pin down why, and both mattered:

- Swapping `lround` for truncate-with-bias changes almost nothing → **the floats were never the
  cost**. A modern core issues an FP add at the same one-cycle throughput as an integer add.
- Restructuring Bresenham to one branch instead of two nearly halves it, with identical
  arithmetic → **the data-dependent branches are the cost**. A mispredict is 15–20 cycles.

Ran four line mixes so no single geometry could flatter one variant, and wrote a
*second* Bresenham in the classic major-axis form specifically so "you wrote it badly" is answered
before it is asked.

**We ship Bresenham anyway**, for exactness (integers are bit-identical across compilers and
architectures, which 1.8 established floats are not) and because its error term *is* 2.2's edge
function. Writing "we chose the measurably slower one, here is the number, disagree if you like"
is a better lesson than being right about nanoseconds — and Exercise 2.1.3 asks the student to
make the opposite case.

## New tooling: docs/_template/check-page.js

The text-vs-shape defect has now shipped twice (1.3's Euler figure, then this lesson's Figure 4,
where the threshold label sat squarely on the rising sawtooth). Neither the viewBox-spill check nor
the text-vs-text check can see it, so I wrote the third: sample points along every stroke and test
them against text bounding boxes.

Two false positives had to be designed out, and both are worth remembering:

- **`<defs>` / `<marker>`.** An arrowhead's `<path>` is never painted at its own coordinates but
  still answers `getTotalLength()` and `getScreenCTM()`.
- **`.grid`.** Graph paper is *meant* to sit under labels — and the class is on the wrapping
  `<g>`, so `el.getAttribute('class')` on the `<path>` returns null and every gridline reads as an
  unclassed stroke crossing every nearby label. Needs `closest('.grid')`.

**Running a new check over the back catalogue finds history.** It immediately turned up ten
pre-existing text-on-shape defects — 1.5 ×2, 1.6 ×3, 1.7 ×4, conventions ×1. Spot-checked 1.7's
Figure 5 by screenshot and it is real: a dashed drop-line runs straight through the `u + v` label.
All cosmetic, all in already-published lessons, so **reported rather than folded into this
lesson's scope**. Left as a follow-up.

## Decisions

- **`draw_line_naive` ships**, switchable at run time, exactly like Pong's naive collision test.
  The demo colours steep spokes coral and shallow ones green, so the failure is *diagnosable* —
  it is precisely the coral half that breaks, which names the cause without a word of prose.
- **The inspector reads back the real routine's output** rather than reimplementing it at 8×
  scale. A diagram generated by different code from the thing it documents will eventually lie.
- **Pong kept on <kbd>Tab</kbd>.** Deleting a working deliverable would be a regression; and two
  demos in one binary being awkward *is* the argument for Module 5's `demos/` split, accumulating
  where it can be felt.
- **Listings spliced from the real files** again (`@@LISTING:path@@`), verified byte-identical
  afterwards.

## Numbers to trust

Naive fan lights **1483** px where DDA and Bresenham light **2081**. A steep line spanning 101
rows and 21 columns: naive lights 21, all disconnected; Bresenham lights 101, none disconnected.
Worked example `(2,2) → (12,8)` produces
`(2,2)(3,3)(4,3)(5,4)(6,4)(7,5)(8,6)(9,6)(10,7)(11,7)(12,8)`, and every pixel is the nearest row
to the true line — asserted in the harness, not eyeballed.

## Next

**2.2 — The Triangle: Edge Functions** (`02-02-triangle-edge-functions.html`). Bresenham's error
term has been answering "which side of the line is this point on?" all along; ask it for three
edges at once and a point is inside iff all three agree. That is a filled-shape rasterizer and it
is how GPUs actually do it. It also resolves 2.1's loose end: edge functions come from the edge's
*geometry* rather than a traversal, so the tie-breaking that makes Bresenham asymmetric becomes a
**fill rule** two adjacent triangles cannot disagree about.

- **`pass: true` is not "the figure is fine."** The checker ignores overlaps under 2 units, so a
  0.1-unit near miss passes and still looks broken.

- Fixed: lessons 0.1 and 0.2 had **two** KaTeX loaders — collateral from 2.1's fix. All 20 pages

## The lesson's spine

Not "how to fill a triangle" but **inside = three independent yes/no questions**. The
independence is the payload: it is why edge functions replaced scanline filling in 1988, why the
approach parallelises, and why GPUs work this way. Stated in §1 as the reason to accept a
seemingly wasteful test-every-pixel design, then cashed in by the exercises.

Everything else follows from one number. `E(A,B,P) = dot(P−A, perpendicular(B−A))` — built
entirely from Lesson 1.7's parts — whose **sign** gives the side and whose **magnitude** gives
twice the sub-triangle's area. The area reading is what makes 2.3 inevitable rather than
arbitrary: the three values sum to the total (verified for points inside *and* outside), so
dividing by the total yields barycentric weights for free.

## Verified, not asserted

- Sign convention **measured**: `(5,0),(0,10),(10,10)` — counter-clockwise on screen — gives
  `−100`. Reconciled with `CCW = front` explicitly: that is an NDC statement, and the viewport
  y-flip reverses it. Recorded in `conventions.html`, because this is the exact class of thing
  §10 says to pin down.
- Worked example checked in code: total 60; `P(5,5)` → 18/24/18 (inside); `Q(9,9)` → 42/−24/42
  (outside); both sum to 60.
- `fill_triangle` is **pixel-identical** to direct full-buffer evaluation across 144 triangles,
  including many straddling the buffer edge.
- Exactly-once coverage: a quad and a 12-triangle fan both give **0 px twice, 0 interior gaps**.
- **75×** speedup from bounding box + incremental stepping (5763 ms → 77 ms).

## The half-open discovery

A harness assertion failed — "the fill rule never drops a pixel" — and the failure was **my
assertion, not the code**. The rule *does* drop pixels: exactly the bottom row and right column,
because coverage becomes half-open. The number was 73, and 37 + 37 − 1 = 73 for a 37×37 quad.
Rewrote the check to assert that precise property instead, which is far more informative than
what I originally wrote.

Second time this session pattern has appeared (1.8 had two such). **A failing check is a claim
about the checker until proven otherwise.**

## The demo needed redesigning after the first version under-sold the bug

The rotating quad showed only **3** double-drawn pixels with the fill rule off — technically a
real defect, visually dismissible as noise. The reason turned out to be the lesson's best
insight: a pixel is only at risk if its centre lies *exactly* on the seam, and for an arbitrary
slope almost none do. Switched the coverage panel to an **axis-aligned square split by its 45°
diagonal**, which passes through every pixel centre along it: **40 of 40**, an unmissable red line.

That is not rigging the demo. Quads-split-into-triangles, terrain grids and UI rectangles are
overwhelmingly axis-aligned, so the catastrophic case *is* the common case — the same shape as
1.8's tunnelling (unreachable at 60 Hz, certain at 10). Written up as a warning callout.

## A figure that was wrong in a way no checker could see

`check-page.js` flagged a label sitting on a line in Figure 1. Investigating it revealed something
worse: **two of the three "infinite edge lines" had the right slopes but wrong offsets**, so they
never passed through the triangle's vertices at all — which would have made the seven regions
meaningless. Recomputed the endpoints from the vertex coordinates rather than eyeballing them.

The lint found a symptom; the diagnosis found a real error. Worth remembering that a cosmetic
warning can be the visible edge of a substantive one.

## Decisions

- **`fill_triangle` does not cull by winding**, deliberately, and the header says why.
- **The `−1` fill-rule bias is folded into the loop's starting value**, so correctness costs zero
  instructions in the inner loop.
- **The inside test is written `w0 >= 0 && w1 >= 0 && w2 >= 0`**, with the equivalent one-comparison
  `(w0|w1|w2) >= 0` explained in a comment rather than used — the student should be able to read
  the line without knowing C++20 guarantees two's complement.
- **The zero-area check is load-bearing**, not housekeeping: the fill rule's proof needs a
  non-degenerate edge. Called out in Exercise 2.2.5, which asks for the proof.
- **Barycentric coordinates foreshadowed, not taught.** They are visible in §3.3 and given as a
  debug assertion; normalising them is 2.3's job.
