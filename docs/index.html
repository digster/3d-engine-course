1The two-stage spine

The central bet of this course is that you should learn the graphics pipeline twice. First on the CPU, where you own every pixel and nothing is hidden. Then on the GPU, where every concept you meet is one you have already built by hand.

By the time SDL_GPU asks you to fill in a depth-stencil state, you will have written the depth test yourself and will know exactly what you are configuring. That is the whole point: recognition, not incantation.

2Who this is for

  • You can program in some language. C++ specifics — RAII, ownership and move semantics, references vs pointers, templates, const-correctness, headers and translation units — are taught in place, the first time each appears, never as a forward reference to a chapter you have not read.
  • No assumed background in graphics, linear algebra, or calculus. All maths is built from zero, geometrically. Perspective projection arrives via similar triangles before any matrix. Quaternions arrive via complex numbers in 2D. Barycentric coordinates arrive via signed areas.
  • A desktop or laptop — Windows, Linux, or macOS — and roughly 300–500 hours.

Exit profile. By the final lesson you can implement rendering techniques directly from papers and blog posts, debug GPU work in RenderDoc, reason about frame budgets and cache behaviour, design and defend engine architecture, and read real engine codebases without drowning. In short: hireable as a junior-to-mid engine or graphics programmer.

3How the course works

Every lesson follows the same shape, and ends with something visible on screen or measurably changed:

  1. The Problem — what is broken, missing, or impossible without today's idea.
  2. Building Intuition — pictures and analogies, before any formalism.
  3. The Theory — derivations, each followed by a worked example with real numbers.
  4. Implementation — short steps, narrated, every decision justified as it is made.
  5. Complete listings — every file touched, in full, with zero placeholders.
  6. Build & run, pitfalls, exercises, recap, further reading.

Three living pages

Conventions Handedness, matrices, NDC and depth, winding, units, colour. Read before Module 2. Convention mismatches are the number-one source of silent rendering bugs.
Math Toolbox A cumulative appendix of every mathematical tool, added the moment the engine needs it.
C++ Style The rules the codebase obeys — naming, headers, ownership, error handling.
No build step — for the lessons

Every lesson is a single self-contained HTML file. No server, no npm, no tooling: open it and read, even offline. The C++ obviously builds with CMake — that is the whole point — but the tutorial itself never asks you to build anything to read it.

What we build by hand, and what we do not

By hand — because building it is the pointThird-party — with a stated reason each time
Maths library (vec2/3/4, mat3/4, quaternions — no GLM), software rasterizer, OBJ parser, ECS, renderer and frame architecture, scene and asset systems, arena/pool allocators, collision and rigid-body basics. stb_image (image decoding), stb_truetype (font rasterization), Dear ImGui (debug UI and editor tooling only — never gameplay UI), cgltf (glTF, and only after the hand-rolled OBJ loader), SDL_shadercross.

Explicitly forbidden: GLM, assimp, bgfx/sokol, Bullet/PhysX/Jolt, EnTT/flecs — or any framework that would do the course's job for it.

5What "done" means

The course is not finished until all of the following exist and work. This list is a contract, not an aspiration:

  • A documented public C++ API.
  • An SDL_GPU forward PBR renderer with shadow-mapped lights, HDR, tonemapping, and a post-processing stack.
  • Skybox and image-based lighting.
  • An asset pipeline covering images, OBJ, and glTF.
  • A handle-based resource system.
  • A from-scratch ECS runtime with transform hierarchy.
  • Skeletal animation.
  • Rigid-body collision and response.
  • 3D audio, and input mapping.
  • ImGui tooling and an editor — hierarchy, inspector, gizmos.
  • Profiling hooks, serialization and a scene format, hot reload, and a job system.
  • Cross-platform CMake builds, sample demos, and an HTML documentation site.
  • A capstone game built solely on the public API.
