Module 2 — Software Rasterizer I · Lesson 2.7

Homogeneous Coordinates and What w Really Means

Two lessons ago we proved no matrix can translate. One lesson ago we built a 4×4, put a translation in the obvious place, and watched it move a cube by exactly zero pixels. Today the fourth component gets a job — and the striking thing is that not one line of arithmetic changes. operator* is byte for byte the function Lesson 2.6 wrote. What changes is one number in the input, and that number turns out to encode something we have needed a name for all along: whether a thing is a place or a direction.

1The Problem

Lesson 2.6 ended on a diagnosis rather than a fix, and it is worth restating exactly, because the fix is hiding inside it.

Where we stopped

A matrix multiplication can only ever scale each column by a component of the input and add the results up. That is all it does. To add a constant — the same offset regardless of the input — you would need a column whose multiplier is the same number every time.

Every component of our input varies with the point being transformed. Except one. The fourth one has no job, no meaning and no reason to be anything in particular.

Read that last sentence again with fresh eyes. It is not a complaint. It is a specification.

We need a component of the input that is always the same number. We have a component of the input that is currently always zero for no reason at all. The gap between those two sentences is one decision wide.

So: let the fourth component be 1.

Then c₃ × w becomes c₃ × 1, which is c₃, which is the fourth column added in full — the same offset, every time, regardless of the point. That is a translation, and it is now inside the matrix.

If that feels like it was too easy, good — you should be suspicious, and the rest of this lesson is four questions that turn a lucky guess into an argument. Exercise 2.6.4 asked you to make the demo work by passing 1, and then to list everything wrong with what you had done. This is that list:

  1. Why 1? What justifies it beyond "it makes the arithmetic come out"?
  2. Should everything carry 1? A surface normal is a 3-vector too. Does it want to be translated?
  3. Does it survive? After two of these matrices are composed and applied, is the fourth component still 1 — and is that guaranteed or lucky?
  4. Can it be something else? Is there any circumstance where w comes out as neither 0 nor 1, and if so, what could that possibly mean?

The first three make translation trustworthy. The fourth is a door, and behind it is perspective — which is why this is a whole lesson rather than a footnote.

2Building Intuition

Two kinds of thing we have been conflating

Before any algebra, a question that has nothing to do with matrices.

Here are two statements about a room. "The lamp is at three metres east, four metres north." And: "The draught is coming from three metres east, four metres north." Both are the pair (3, 4). They are not the same kind of statement.

Now pick the whole room up and move it two metres east. The lamp is now at (5, 4) — it went with the room, because it is somewhere. The draught is still coming from (3, 4), because that was never a place; it was a direction, and moving a room does not change which way is north-east.

The distinction, stated once

A position says where. Translation moves it.
A direction says which way and how far. Translation must leave it completely alone.

Both are three numbers. Rotation and scale treat them identically. Only translation can tell them apart — which is exactly why the distinction never came up until this lesson.

This is not a nicety. Every surface normal in Module 3's lighting is a direction. Every velocity in Module 7's physics is a direction. The vector from a surface to a light is a direction. The positions of the vertices those normals belong to are positions. They live side by side in the same buffers, they have the same type, and the wrong one being translated produces lighting that is subtly wrong in a way that gets worse the further your object sits from the world origin. §3.2 measures exactly how much worse.

One number can carry the difference

Now put the two ideas next to each other.

We need a fourth component that is a constant, so that a column can be added as a constant. And we have just found a distinction between two kinds of vector, where one wants the translation applied and the other wants it skipped.

Those are the same problem. The fourth component is the multiplier on the translation column, so it decides whether the translation applies. Set it to 1 and the translation is added in full. Set it to 0 and the translation is skipped entirely — which is not a broken position, it is a correct direction.

\[ M\mathbf{v} = \mathbf{c_0}x + \mathbf{c_1}y + \mathbf{c_2}z + \underbrace{\mathbf{c_3}w}_{\text{translation} \times w} \] M*v = c0*x + c1*y + c2*z + c3*w, where c3 is the translation and w decides whether it applies

So Lesson 2.6's w = 0 was never a placeholder. It was the right answer to a question nobody had asked yet — should translation move this? — for the only kind of thing we had been transforming, back when translation did not exist. Now that it does, we simply have to say which kind of thing each vector is, and one number does it.

3The Theory

Why 1?

The honest answer is not "because it works". It is that 1 is the multiplier that applies the translation exactly once, and every other value applies some other amount of it.

The fourth column holds the offset t. The arithmetic contributes t × w. So the choice of w is not a formality that enables translation — it is the translation's scale factor:

Worked example — what each value of w actually does

Take translation(1.5, −2, 0.75) and the point (0.40, −0.30, 0.90), sent through with different fourth components. The offset actually applied:

woffset appliedmeaning
0.0(0.00, 0.00, 0.00)no translation — a direction
0.5(0.75, −1.00, 0.38)half the translation
1.0(1.50, −2.00, 0.75)the translation, once
2.0(3.00, −4.00, 1.50)twice the translation

The result at w = 1 is (1.9000, −2.3000, 1.6500) — the original point plus the offset, exactly — and it comes back out with w = 1 still. All asserted in the harness.

So 1 is forced, not chosen. If you want "translate by t", the multiplier on t has to be 1. There is no convention here to disagree with, which is worth knowing because a great many other things in graphics genuinely are conventions.

That also answers a question you may not have thought to ask: why does a position keep w = 1 after the transformation rather than accumulating? Because the matrix's bottom row is (0, 0, 0, 1), so the output's fourth component is 0x + 0y + 0z + 1w = w. It comes back unchanged. §3.3 makes that precise, because it is the part that makes chains safe.

Why directions carry 0 — and why that is the point

A direction sent with w = 0 is rotated and scaled normally and never translated. That is not a degraded position. It is the correct treatment of a different kind of object, and Figure 2 shows the two side by side going through the same matrix.

Here is the part that makes this worth taking seriously rather than filing away. The error does not stay small.

Because the error is the translation, it scales exactly with it. An object at the world origin shows no symptom at all; the same object a kilometre away is catastrophically wrong. That is a miserable bug to hunt: it does not reproduce in your test scene, it appears only in the level, and it gets worse as the level gets bigger.

Lesson 3.6 lights surfaces using normals. If a normal has been translated, it is no longer perpendicular to anything and the shading is wrong everywhere except near the origin. Send directions with w = 0.

This is also why vec4.hpp now offers point() and direction() as named constructors instead of leaving you to write to_vec4(v, 1.0f). A call site that says direction(n) states an intention; one that says to_vec4(n, 0.0f) invites the next person to change the 0 to a 1 to make something compile.

Does w survive? The bottom row

Third question, and the one that decides whether any of this can be trusted in a chain. After a transformation, is the fourth component still what it was?

Work out the last component of Mv. It is the bottom row of the matrix dotted with the vector — and for every matrix we have built, that bottom row is (0, 0, 0, 1):

\[ w_{\text{out}} = 0\cdot x + 0\cdot y + 0\cdot z + 1\cdot w = w \] w_out = 0*x + 0*y + 0*z + 1*w = w

So a position stays a position and a direction stays a direction, exactly, with no drift — not "to within a rounding error", but bit-for-bit, because the arithmetic is a multiplication by one and three multiplications by zero.

But a single transform is not the interesting case. A scene graph composes ten of them. Does the bottom row survive that?

A matrix with that bottom row is called affine: a linear transformation followed by a translation. The set of affine transforms is closed under composition, which is a formal way of saying the thing you actually care about — chaining them can never produce something that is not one of them.

That closure is what makes the whole scheme safe rather than merely clever. If composing could corrupt the bottom row, then w would drift, positions would slowly stop being positions, and every scene graph would need a normalisation step somewhere. Instead it is guaranteed by the structure, and it is guaranteed exactly.

The payoff: one matrix, any chain

Look at that product's translation column in Figure 3: A·t_b + tₐ.

You have seen it before. Exercise 2.5.3 asked you to rotate an object about its own centre, found that M*(v − c) + c works, and then asked what was unsatisfying about it. The answer was that the transformation had stopped being a matrix — it was a matrix and a vector, and composing two of them meant deriving and carrying this rule yourself:

\[ (M_2, t_2) \circ (M_1, t_1) = (M_2M_1,\ M_2t_1 + t_2) \] (M2, t2) after (M1, t1) = (M2*M1, M2*t1 + t2)

That rule has not gone away. It has been absorbed. Ordinary 4×4 matrix multiplication produces exactly it, and the harness checks that claim directly: affine(A, tₐ) · affine(B, t_b) equals affine(A·B, A·t_b + tₐ). You never write the rule, never store a vector alongside a matrix, and never get the order wrong, because there is only one object and one operator.

Worked example — rotate about a point, as a single matrix

Rotating about the point c is three steps: bring c to the origin, rotate, put it back. Written as matrices, and remembering that the rightmost factor applies first:

\[ M = T(\mathbf{c}) \cdot R \cdot T(-\mathbf{c}) \] M = translation(c) * rotation * translation(-c)

That is one mat4. It composes with anything else, it goes in a scene graph, it uploads to a GPU. With c = (4, 1, −2) and a rotation about y:

  • the pivot maps to (4.0000, 1.0000, −2.0000) — it is a fixed point ✓
  • a point one unit from the pivot stays 1.000000 units from it ✓

Both verified. Exercise 2.5.3's complaint is now closed, and the closing did not require a new concept — only a fourth component with a job.

The name, and the picture behind it

What we have built is called homogeneous coordinates, and now that it is derived the name can be explained rather than announced.

So far w has been 1 or 0 and nothing else, and if that were the whole story the honest name for it would be "a flag". It is more than a flag, and here is the extra idea:

Scaling the whole 4-vector does not change the point

The 4-vector (2, 4, 6, 2) denotes the same 3-D position as (1, 2, 3, 1). To read a position out of a homogeneous 4-vector you divide by w, and (2, 4, 6) / 2 is (1, 2, 3).

So a single 3-D point corresponds to a whole family of 4-vectors, all scalar multiples of one another. "Homogeneous" is the mathematician's word for exactly that: the representation is defined only up to an overall scale.

With w = 1 the division is by one and it is invisible, which is why nothing in this lesson has needed it. That is also why xyz() in our vec4.hpp drops the fourth component rather than dividing by it, with a comment saying exactly when that stops being correct.

And directions? A direction has w = 0, so dividing by w is undefined — which is the algebra agreeing with the geometry. A direction is not at anywhere, so asking for its position has no answer. In the classical language it is a "point at infinity": the direction you approach as you slide a point further and further out.

When w is neither 0 nor 1

Fourth question, and this is the door.

Every matrix we have built has a bottom row of (0, 0, 0, 1), which is what forces w_out = w_in. Nothing in mat4.hpp requires that. Put something else in the bottom row and w comes out as a function of the input — and then the division stops being invisible.

Try a bottom row of (0, 0, −1, 0), which makes w_out = −z. In our right-handed world, −z is forward, so w_out becomes the distance in front of the camera. Feed points at increasing distance through it:

pointw outx / wy / w
(1, 1, −1)11.00001.0000
(1, 1, −2)20.50000.5000
(1, 1, −5)50.20000.2000
(1, 1, −10)100.10000.1000

Read the last two columns. The same object, twice as far away, comes out half the size. Ten times as far, a tenth the size.

That is perspective

Things further away appear smaller, and it has emerged from a matrix with an unusual bottom row plus a division by w. That division is called the perspective divide, and it is the single most important operation in the projection pipeline.

This is why w deserves a whole lesson rather than a footnote. It is not bookkeeping to make translation fit into a matrix; it is the mechanism that makes projection fit into a matrix too, so that a rotation, a camera and a perspective projection are all the same kind of object and compose with one multiply.

We are going to stop there, deliberately. Where the bottom row comes from, why it is that particular row, how the near and far planes get involved, and what happens to points behind the camera are all Lesson 2.10's, derived from similar triangles rather than presented as a matrix to copy. Lesson 2.8 uses affine transforms to build the model and world spaces first, and 2.9 builds the camera.

What you should take from this section is smaller and load-bearing: a matrix that changes w is not affine, and after one of those has been applied you must divide by w before the numbers mean anything. Everything in Module 2 up to Lesson 2.9 is affine, so the divide is a no-op and can be ignored. The moment it stops being one, it will be named.

4Implementation

Start with what did not change, because it is the most interesting line in the lesson.

Byte for byte the function Lesson 2.6 wrote and then demonstrated to be useless. It was never useless; it was waiting. Everything this lesson adds is either a named way of choosing w, or a named way of building a matrix that already worked — plus a great deal of documentation, because the arithmetic is not the hard part here and never was.

That is worth sitting with for a second as a general observation. The fix for "my 4×4 does not translate" was not in the matrix code at all. It was in what the caller said about the thing being transformed. Bugs like that are common and painful precisely because the code you stare at is correct.

4.1  Naming the two kinds of vector

Two functions, four lines, and they are the most valuable thing in this lesson's code.

They could both be spelled to_vec4(v, 1.0f) and to_vec4(v, 0.0f) — the general function is still there for the rare case that needs it. But a literal 1.0f at a call site is a magic number, and magic numbers get changed by whoever is trying to make something compile. direction(n) states an intention that survives being read by somebody in a hurry, and if they change it to point(n) they have written a sentence that is visibly wrong rather than a digit that looks arbitrary.

Why not two separate types?

The fully type-safe design gives positions and directions distinct types, so the compiler rejects translating a normal. Real engines do sometimes do this, and it does eliminate the bug entirely.

We are not doing it, for reasons worth stating rather than hiding. It doubles the surface of the maths library — every operation needs to say which combinations it accepts, and some of the answers are genuinely fiddly (position minus position is a direction; position plus position is meaningless). It also fights the GPU, where a shader receives four floats and no types at all, so the distinction has to collapse at the boundary regardless. Named constructors buy most of the safety for a twentieth of the machinery.

It is a real trade rather than an obvious call, and if you build your own engine after this course it is worth revisiting with your own judgement.

4.2  Building affine transforms

Note the fourth components. The first three columns get 0 — they are the images of the basis directions, and a rotation must not disturb w as a side effect. The fourth column gets 1, which is the bottom-right element of the matrix and the thing that makes w_out = w_in.

Put together, those choices produce the bottom row (0, 0, 0, 1) that §3.3 showed is load-bearing. It is not decoration and it is not padding: it is the invariant that keeps positions positions through any depth of composition.

affine(linear, t) gives the same matrix as translation(t) * to_mat4(linear). Both are provided because the second says how and the first says what, and because writing the product out is one more chance to get the order backwards.

4.3  The demo, which is Lesson 2.6's demo

The scene is two cubes, and they are built from the same two ingredients in opposite orders:

Rightmost first, as always. The first rotates the cube about the origin — which is its own centre, since the model is centred there — and then moves it into position, so it spins where it stands. The second moves it out first and then rotates, so the rotation swings the already-displaced cube around the world origin.

That is Lesson 2.5 §3.6's "order matters" with a translation finally in play, and it is also Exercise 2.5.3 answered: put the translation on the left and you get rotation about the object's own centre. Both are single matrices; either could be composed into a longer chain without any special handling.

The corners go through as positions, and that one argument is the entire difference from Lesson 2.6:

point_w is a variable rather than a literal 1.0f so that W can set it back to zero and reproduce Lesson 2.6 exactly, on demand. That is the same bargain the course has made four times now — draw_line_naive, Pong's naive collision, blend_space::encoded, and now this. A failure you can summon with one keystroke teaches more than a paragraph describing it.

The three axis arrows on each cube are directions, and the demo lets you get that wrong too:

Notice the structure, because it is how every renderer handles this. The object's position comes from the matrix. The arrow's direction comes from the matrix. They are combined afterwards, by us, to decide where to draw the arrow. The matrix is never asked to place a direction, because that question has no answer.

The world centres in the readout are read back out of the transformed result rather than assumed, so the numbers on screen are what was drawn — the same discipline as Lesson 2.4's centre-pixel readout and Lesson 2.6's displacement.

6Build & Run

What you should see

Two cubes. That is the milestone, and it is worth pausing on before touching anything: last lesson we could not put a second object anywhere, because every object was pinned to the origin and no matrix could move it. Two cubes in two places is the whole of this lesson, made visible.

A faint cross marks the world origin. The left cube sits away from it and spins in place. The right cube swings around it. Both are built from the same rotation and the same offset, composed in opposite orders — translation(−p) * R against R * translation(p).

Checkpoint — four things to do
  1. Watch the two cubes. Stop the spin with Space and step the angle with , and .. One stays put and turns; the other travels around the cross. The readout prints both world centres — the first stays at −1.70, the second sweeps.
  2. Press W. The corners are now sent with w = 0 and both cubes collapse onto the origin, one exactly on top of the other. Both translations are ignored. That is Lesson 2.6, reproduced with one keystroke — and it is also a very good picture of what a scene looks like when somebody forgets which kind of vector they are transforming.
  3. Press N. The axis arrows are now sent with w = 1, so they are translated as though they were places. They stop pointing along the cubes' own axes and skew off toward wherever the translation put them. The cubes themselves are unaffected — which is the point, and the reason this bug survives code review.
  4. Press Z. The rotation modes still work, including the two composition orders from Lesson 2.6, now on top of translation.
Look carefully at what N does not break

When the axis arrows are wrong, the cubes are still perfectly correct — right shape, right place, right rotation. Only the arrows are wrong, and only their direction, and only by an amount proportional to how far the object is from the origin.

Now imagine those arrows are surface normals and you cannot see them, only the lighting they produce. That is Lesson 3.6's version of this bug: everything looks structurally fine and the shading is subtly wrong, more so the further you get from the world origin.

7Common Pitfalls & Debugging

Every object piles up at the origin — positions sent with w = 0
Symptom
Objects render with correct shape and rotation but all in the same place. Rotation and scale work perfectly; only position is ignored. Nothing errors.
Cause
Vertices built with w = 0, so the translation column is multiplied by zero (§1). Often from a vertex struct that was zero-initialised, or a vec4 built component-wise where the fourth was forgotten.
Fix
Build positions with point(). To catch it in a test rather than by eye: transform the model origin and assert it lands at the transform's translation — xyz(M * point(vec3{})) == translation_of(M).
Lighting is wrong, and worse far from the origin — directions sent with w = 1
Symptom
Geometry is perfect; shading is not. Objects near the world origin look fine, objects far from it look badly lit, and moving an object changes its shading when nothing about it or the light has changed.
Cause
A normal, velocity or direction-to-light transformed as a position, so the translation was added to it. The error equals the translation, which is why it scales with distance (§3.2 — 10.77, then 107.70, then 1077.03).
Fix
direction(), not point(). The cheapest test is a length check: a unit direction through a rotation must come back unit length. If it comes back with the magnitude of your translation, this is the bug.
An object orbits when it should spin — the composition is the wrong way round
Symptom
An object rotates about the world origin instead of its own centre, swinging through a wide arc. The further it is from the origin, the more dramatic the swing.
Cause
R * translation(p) instead of translation(p) * R. The rightmost factor applies first, so the first version moves the object out and then rotates the displaced object.
Fix
Put the translation on the left. If the pivot is not the model's own origin, sandwich it: translation(c) * R * translation(−c). Both cubes in the demo are on screen at once precisely so this comparison is available whenever you need reminding.
Positions slowly stop being positions — a broken bottom row
Symptom
Transforms behave until they are composed several deep, and then objects drift, scale oddly, or vanish. Reading w back shows it is no longer 1.
Cause
A matrix in the chain whose bottom row is not (0, 0, 0, 1) — usually one assembled by hand, or read from a file in the wrong layout, or transposed. The affine guarantee of §3.3 only holds while every matrix in the chain is affine.
Fix
Build matrices with affine() and translation() rather than by filling in sixteen floats. As an assertion in debug builds, check the bottom row of any matrix entering the chain. And note that a projection matrix breaks this deliberately (§3.6) — the rule is that everything before the projection is affine, not that everything is.
Everything looks right until a projection is added — the missing divide
Symptom
Not yet — this is Lesson 2.10's, and it is here so you recognise it. Objects appear at wildly wrong sizes and positions, or the whole scene collapses, the moment a perspective matrix joins the chain.
Cause
Reading x, y, z straight out of the result and ignoring w. That is correct for affine transforms, where w is still 1, and wrong the instant something makes it otherwise.
Fix
Divide by w — the perspective divide. Our xyz() deliberately does not do it and says so in its doc comment, so that the divide will appear in Lesson 2.10 under its own name rather than having always been hidden inside an accessor.

8Exercises

2.7.1Predict the four resultswarm-up

Let M = affine(rotation_y(π/2), (10, 0, 0)). Without running anything, predict what M does to each of these, including the resulting w:

  1. point(0, 0, 0)
  2. direction(0, 0, 0)
  3. point(1, 0, 0)
  4. direction(1, 0, 0)
Solution
  1. (10, 0, 0), w = 1. The origin goes to the translation — which is why translation_of() exists: the fourth column is where the origin lands.
  2. (0, 0, 0), w = 0. A zero direction stays zero; there is nothing to rotate and no translation to apply. Note this is the one vector where the two kinds agree on the input but not the output.
  3. A quarter turn about y sends (1,0,0) to (0, 0, −1), then the translation gives (10, 0, −1), w = 1.
  4. (0, 0, −1), w = 0. Rotated, not translated — and still unit length, which is the check worth having.

Items 3 and 4 are the same three numbers going in and different answers coming out. If that still feels odd, re-read §2.1: one is a place in the room, the other is a way of pointing.

2.7.2Rotate about an arbitrary pivotwarm-up

Add a third cube to the demo that rotates about a pivot which is neither the world origin nor its own centre — say a corner of the cube. Build it as one matrix.

Hint

translation(c) * R * translation(−c), where c is the pivot. Read it right to left as a sentence: bring the pivot to the origin, rotate, put it back.

Solution

With c = (0.5, 0.5, 0.5) — a corner of the unit cube — that corner stays exactly still while the rest of the cube swings around it. Verified in the harness for a pivot at (4, 1, −2): the pivot maps to itself exactly, and a point one unit away stays 1.000000 units away.

The thing to notice is that the result is a single mat4. You can hand it to anything that takes a transform, compose it into a longer chain, or upload it to a GPU. That is what Exercise 2.5.3 could not do, and the difference is one component.

2.7.3Assert the invariantmedium

Write a check that a mat4 is affine, and use it to catch a deliberately broken matrix. Then decide where in a real engine such a check should live — and whether it should be a hard error.

Hint

The bottom row is at(3,0) through at(3,3). What must it be, and should you test for exact equality or a tolerance?

Solution sketch

at(3,0)==0 && at(3,1)==0 && at(3,2)==0 && at(3,3)==1, and exact equality is right here: every affine matrix we build produces those values as exact zeros and ones, and composition preserves them exactly (the harness confirms it after forty compositions). A tolerance would only hide a matrix that has genuinely drifted.

On where it lives: a debug-only assertion at the point a transform enters the renderer is the useful spot — cheap, and it fires close to the code that built the bad matrix rather than in the frame where the symptom appears. It must not be a hard error in release, and it must not run on the projection matrix, which breaks the invariant on purpose. That distinction — "affine up to the projection, not affine after" — is worth writing into the assertion's name.

2.7.4Make w mean something elsemedium

Build a mat4 whose bottom row is (0, 0, −1, 0). Push a few points at different depths through it and divide by the resulting w. Describe what you have just built, and then find its two failure cases.

Hint

With that bottom row, w_out = −z. In a right-handed, −z-forward world that is the distance in front of the camera. Now what is x / w?

Solution

You have built a perspective projection — the crudest possible one, but a real one. Points twice as far away come out half the size: (1, 1, −1) gives x/w = 1.0, (1, 1, −2) gives 0.5, and (1, 1, −10) gives 0.1.

The two failure cases are exactly the ones Lesson 2.10 and Lesson 3.3 have to deal with. At z = 0 — a point level with the camera — w is zero and the divide is undefined. Behind the camera, z > 0 makes w negative, and the point comes out mirrored through the origin and apparently in front of you: geometry behind the camera turning up on screen, inverted.

Neither is fixable by being careful with the arithmetic. Both are why clipping exists — removing geometry on the wrong side of the near plane before the divide ever runs — and it is Lesson 3.3's subject. Worth knowing now that clipping is not an optimisation.

2.7.5Argue the other designopen

§4.1 chose named constructors over separate position and direction types. Sketch the type-safe version — which operations it allows, which it forbids — and make the strongest case you can for it. Then decide what you would actually do.

Hint

Work out what each combination should mean. Position minus position? Position plus direction? Position plus position? Direction times scalar? Some have obvious answers, one is meaningless, and the pattern is informative.

Solution sketch

The rules fall out cleanly: position − position = direction (the vector from one to the other); position + direction = position (move a place); direction ± direction = direction; direction × scalar = direction. And position + position is meaningless — there is no sensible answer, which is a strong hint that the distinction is real rather than pedantic. (It is called an affine space, and the fact that it has a name is another hint.)

The case for it: this eliminates an entire class of bug at compile time, and it is the bug §3.2 showed is invisible near the origin and ruinous far from it — the worst possible detection profile. Some engines do exactly this.

The case against, which is what we chose: it roughly doubles the maths library's surface, every function has to state which combinations it accepts, and the distinction must collapse anyway at the GPU boundary where a shader sees four floats and no types. Named constructors get most of the benefit for a fraction of the machinery.

The honest conclusion is that this is a real trade rather than a settled question, and it depends on how large your codebase is and how many people are in it. What is not a trade is being explicit at the call site — whichever design you pick, the reader should be able to see which kind of thing is being transformed without going and looking.

9Recap & Next

What the engine can do now that it could not before

  • It can put things in places. Two lessons of "no matrix can translate" are over: translation lives in the fourth column, composes like everything else, and needs no special handling anywhere.
  • Rotation about any pivot is one matrix — T(c) · R · T(−c) — closing Exercise 2.5.3, and it is a single object that can be composed, stored and uploaded.
  • The engine distinguishes places from directions, using one number rather than two types. point() and direction() make the choice visible at every call site.
  • The affine guarantee is exact. A bottom row of (0,0,0,1) gives w_out = w_in, and the product of two such matrices has it too — verified exactly after forty compositions.
  • Not one line of arithmetic changed. operator* is byte for byte Lesson 2.6's. The fix was in what the caller said about the data.
  • And w can be something else — which makes distant things smaller, which is perspective, which is why this was a lesson rather than a footnote.

The habit worth taking: when a piece of machinery seems to be missing something, check whether it is missing an input rather than missing a feature. The 4×4 was complete in Lesson 2.6. What it lacked was a reason for one of its inputs to be anything in particular, and supplying that reason cost no code at all.

Next: 2.8 — The Space Chain: Model to World. We have the machinery; now we start using it deliberately. A vertex begins life in the coordinates its artist modelled it in and has to reach the screen, and it does so by passing through a sequence of named spaces — model, world, view, clip, NDC, screen — each one a matrix away from the last.

2.8 builds the first two links and, more importantly, establishes the discipline that keeps the rest of the module honest: always know which space a coordinate is in. Nearly every transform bug in graphics is a coordinate being used in the wrong space, and it is a class of bug that naming things carefully makes almost impossible.
