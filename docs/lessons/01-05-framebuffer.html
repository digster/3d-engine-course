Module 1 — The Loop, the Pixel, and Time · Lesson 1.5

The Framebuffer: Your First Owned Pixels

Every pixel on screen so far has been SDL's doing. We asked for a rectangle and one appeared, by means we never examined. That has to end here, because you cannot build a rasterizer on top of a function that already draws the shape for you. Today we allocate a block of memory, decide what every byte in it means, write into it by hand, and hand the result to the GPU. From this lesson to the end of Module 3, everything you see is bytes you wrote.

1The Problem

Here is the line that has been drawing our boxes since Lesson 1.2:

It works, and it is a dead end. Ask the questions the next three modules depend on and this function has no answers: which pixels did it touch? What value did it write into each? Where is that memory? How would I draw a triangle instead — or interpolate a colour across one, or reject a pixel because something nearer already occupies it?

Every one of those questions is about the individual pixel, and SDL_RenderFillRect's entire purpose is to spare you from thinking about individual pixels. It is a fine tool for drawing a rectangle and a useless foundation for a renderer, because a renderer is precisely a machine for deciding what colour each pixel should be.

So we take over. The plan is smaller than it sounds:

  1. Allocate a block of memory big enough for one image.
  2. Agree on what each part of it means.
  3. Write into it by hand.
  4. Hand the finished block to the GPU as a texture, once per frame.

Step 4 still uses SDL, and will keep doing so until Module 4 — we are not writing a display driver. But steps 1 to 3 are entirely ours, and the whole software rasterizer of Modules 2 and 3 is built out of nothing but those three steps repeated with increasing cleverness.

Why 320×180, when the window is 1280×720

Our framebuffer will be deliberately small, and stretched over the window on presentation. Partly that is pedagogy: at exactly 4× magnification every pixel you write becomes a visible 4×4 block, so put_pixel has consequences you can see from across the room. But it is also a real technique — rendering at a fixed internal resolution and scaling to the display is how a great many games ship, from pixel-art indies to console titles using dynamic resolution to hold a frame budget. A pleasant side effect: window resizing needs no code at all, because the framebuffer's size no longer depends on the window's.

2Building Intuition

2.1 Memory is a line; an image is an agreement

A computer's memory is one-dimensional. It is an enormous row of numbered boxes, and the number of a box — its address — is a single integer. There is no such thing as memory address (3, 7). Nothing in the hardware knows what a rectangle is.

An image, meanwhile, is two-dimensional by nature. So somewhere between the two, somebody has to decide how a rectangle is going to be laid down along a line. That decision is a convention — a shared agreement between the code that writes pixels and the code that reads them — and it is not enforced by anything. Get the two ends of the agreement to disagree and the machine will not complain; it will just show you a mangled picture.

The convention nearly everything uses is the same one you use to read this page: start at the top left, go along the first row to its end, then drop to the start of the next row and repeat. Laid out along a line, an image is simply its rows, one after another, nose to tail. That is called row-major order, and it is the only structural idea in this lesson.

2.2 Folding the line into a rectangle

Take a strip of memory 40 boxes long and agree that the image is 8 wide. Then the strip folds into a grid: boxes 0–7 are the top row, 8–15 the next, and so on, five rows in all. Figure 1 is that fold, drawn.

Play with the mapping below: click any cell to see where it lands in the strip, and drag the width slider to watch the same line of memory refold into differently shaped images.

width = 8 (3, 2) → 2 × 8 + 3 = 19

The strip below the grid is always the same 24 boxes of memory. Only the agreed width changes, and with it the shape of the picture those boxes represent.

3The Theory

3.1 The index formula

Derive it rather than remember it. To reach the pixel at (x, y), count from the start of the strip:

  • Every complete row above yours contributes width pixels. There are y of those rows, so they take up y × width boxes.
  • Within your own row you then step x boxes to the right.
Worked example — pushing real numbers through

An 8 × 5 image, as in Figure 1:

PixelArithmeticIndexCheck
(0, 0)0 × 8 + 00First box. Top-left.
(7, 0)0 × 8 + 77Last box of the first row.
(0, 1)1 × 8 + 08The next box along — rows really are nose to tail.
(3, 2)2 × 8 + 319The highlighted cell.
(7, 4)4 × 8 + 739Last box. 8 × 5 − 1, as it must be.

That last row is the check worth doing whenever you write one of these: the bottom-right pixel must land on the very last element, width × height − 1. If it lands anywhere else, the formula is wrong and every image you draw will be sheared.

Two consequences follow immediately, and both matter later.

Moving one pixel right adds 1 to the index; moving one pixel down adds width. Horizontal neighbours are adjacent in memory. Vertical neighbours are a whole row apart. Those two facts are why §3.5's measurement comes out the way it does, and they shape every inner loop in the rasterizer to come.

Nothing in the formula knows the width is a boundary. It happily accepts an x of 8 in an 8-wide image and returns an index that is perfectly valid memory — just not the pixel you meant. That is the next section.

3.2 When x leaves the row

Suppose you draw a 4-pixel-wide horizontal run starting at x = 6 on row 2 of our 8-wide image, and you do it without checking bounds.

Worked example — the wrap
You asked fory × width + xIndexWhich pixel that actually is
(6, 2)2 × 8 + 622(6, 2) ✓
(7, 2)2 × 8 + 723(7, 2) ✓
(8, 2)2 × 8 + 824(0, 3) — first pixel of the next row
(9, 2)2 × 8 + 925(1, 3)

No crash. No warning. Index 24 is a real pixel inside the buffer, and writing to it is a perfectly legal thing to do — it is simply the wrong pixel. Your rectangle's right-hand edge has reappeared on the far left, one row down.

Worth noticing what a lucky version of this bug looks like. If x strays far enough — or y does — the index leaves the buffer entirely, and now you are writing over whatever happens to live after your pixels in memory. That is undefined behaviour, and its symptoms range from a corrupted variable elsewhere in the program to an immediate crash. The crash is the lucky outcome. The stripe is the unlucky one, because it can survive review.

3.3 What a pixel actually is

We have been saying "pixel" as though it were a colour. In memory it is 32 bits — four bytes — holding four 8-bit numbers: red, green, blue, and alpha (opacity). Eight bits gives each channel 256 levels, and 256⁴ ≈ 4.3 billion combinations, which is why 32-bit colour became the default and stayed there.

The interesting question is the order, and this is where a great deal of confusion lives. Packing our four numbers into one Uint32:

Each component is shifted into its own byte of the 32-bit word: alpha into the top eight bits, then red, then green, with blue left at the bottom. The result reads as 0xAARRGGBB — which is exactly how you would write the colour by hand.

Worked example — packing, verified by running it
CallResultReading the bytes
pack_argb(0x12, 0x34, 0x56, 0x78)0x78123456a=0x78, r=0x12, g=0x34, b=0x56
pack_argb(255, 0, 0)0xFFFF0000Opaque red
pack_argb(0, 0, 255)0xFF0000FFOpaque blue

Note how much less confusing 0xFFFF0000 is than "the red one" once you can read the byte positions directly. Getting fluent at that is worth ten minutes with a debugger.

SDL's format names encode which question they answer, and the header spells the rule out:

  • Names with a bit count per component — ARGB8888, XRGB1555 — describe a value packed into a native-endianness integer, most significant component first. Use these when what matters is the meaning of each bit position.
  • Names with a single total bit count — RGBA32, RGB24 — describe the order of bytes in memory, and are platform-independent. Use these when what matters is the byte layout.

The two views disagree on little-endian hardware, and SDL's own header makes that concrete in a way worth seeing:

On the machine you are almost certainly reading this on, RGBA32 is ABGR8888. That looks like a typo and is not: the two names describe the same bytes from opposite ends.

How we sidestep the whole problem

Our framebuffer stores Uint32 values and we build them with shifts, never by writing individual bytes. That means we are working purely in the packed-integer view, so we ask SDL for SDL_PIXELFORMAT_ARGB8888 — the packed-integer name — and the two match by construction on every platform, big-endian included. Endianness only becomes your problem the moment you reach for a Uint8* and start writing bytes, which is exactly what an image file loader has to do. Module 6's stb_image integration meets that properly; until then we simply never open the box.

3.4 Getting it onto the screen

Our buffer lives in ordinary CPU memory. The display is driven by the GPU. Bridging them takes three steps, once per frame.

First, create the texture once, at startup:

SDL_TEXTUREACCESS_STREAMING tells SDL this texture changes every frame and must be lockable. SDL_SCALEMODE_NEAREST makes the 4× upscale produce crisp blocks rather than a blurred average of them — for a lesson about individual pixels, linear filtering would hide the subject. (SDL 3.4 also offers SDL_SCALEMODE_PIXELART, a sharper-but-smoother variant worth knowing about for real pixel-art projects.)

Then, each frame, lock it and copy — one row at a time:

SDL_LockTexture hands back a pointer and a pitch, and the pitch is the one that matters: it is the distance in bytes from the start of one row to the start of the next, which may exceed the row's actual content. We step the destination by its pitch and copy our row's worth of bytes into each row. On most machines the two pitches happen to be equal and a single big memcpy would work — which is exactly what makes this bug so good at reaching production.

Locked texture memory is write-only, and starts as garbage

SDL's header is blunt about it: locking gives write-only access, and "the pixels made available for editing don't necessarily contain the old texture data… if you need to keep a copy of the texture data you should do that at the application level." Two rules follow. Never read from the locked pointer — the contents are whatever the driver felt like leaving there. And write every pixel you care about, every frame, because last frame's image is not necessarily still in there. Our design satisfies both without effort: the framebuffer is the application-level copy the docs are asking for, and the lock is a one-way push.

Finally, draw it:

A null source rectangle means the whole texture; a null destination means "the entire rendering target". So our 320×180 image is stretched over the window at whatever size it currently is — and that is the whole of our resize handling. The window event we used to care about in 1.3 is gone from the loop entirely.

3.5 Two ways to write a pixel, measured

We now have a safe way to set a pixel:

Every call pays for two comparisons, a multiply and an add. For scattered points that is exactly right and utterly negligible. For filling a whole 320×180 background it is 57,600 repetitions of work the loop already knows the answer to: x and y are loop counters bounded by the buffer's own dimensions, so the check cannot fail, and y × width is constant across an entire row.

So the framebuffer also exposes a row pointer, and the inner loop walks it:

Worked example — measured, not guessed

Filling all 57,600 pixels of the 320×180 buffer, median of nine runs, Apple M4 Pro:

MethodDebug (-O0)Release (-O2)
put_pixel per pixel0.331 ms0.042 ms
Row pointer0.065 ms0.003 ms
Speed-up5.1×14.8×

Two things are worth extracting. The optimiser closes some of the gap but nowhere near all of it — it cannot prove the bounds check is redundant, so it keeps emitting it. And the ratio is much larger in the optimised build, because optimisation makes the useful work cheaper while the unavoidable per-call overhead stays. That is the general shape of this kind of result, and it is why performance claims measured in a Debug build should be treated as fiction.

Both loops produce byte-identical images — that is checked, not assumed — so this is a pure addressing difference. Press G in the demo to switch between them and watch the on-screen millisecond figure move.

Now the second measurement, which is where memory layout starts to matter. §3.1 established that horizontal neighbours are adjacent in memory and vertical neighbours are a whole row apart. The CPU does not fetch bytes; it fetches cache lines, typically 64 bytes — sixteen 32-bit pixels — at a time. So walking a row uses all sixteen pixels of each line it fetches. Walking a column uses one, then moves 1,280 bytes away and fetches another.

Worked example — the same writes, two orders

Filling the entire buffer through row pointers, changing only which loop is outer, at several sizes (-O2, median of seven):

FramebufferSizeRows outerColumns outerPenalty
320 × 180225 KB0.004 ms0.045 ms10.9×
640 × 360900 KB0.011 ms0.354 ms33.3×
1280 × 7203.5 MB0.053 ms1.718 ms32.7×
1920 × 10807.9 MB0.102 ms3.793 ms37.2×
3840 × 216031.6 MB0.327 ms15.923 ms48.8×

Identical instructions, identical writes, identical output — reordered. The penalty grows with the buffer because the machine's caches (64 KB of L1 data, 4 MB of L2 on this chip) stop being able to hide the waste: at 320×180 the buffer still fits in L2, so the cost is mostly wasted L1 traffic; by 4K each column step is a trip towards main memory. Rows outer, columns inner. Always. It is the cheapest performance decision in graphics programming, and it is made by choosing which loop to write first.

The measurement that lied — and what it teaches

The first version of that experiment showed no difference at all: 0.267 ms row-major against 0.267 ms column-major, a ratio of 1.00. Which would have been a fascinating result, and was entirely an artefact.

The reason: both loops were written with put_pixel. Its per-call overhead — the bounds check and index multiply from the table above — was so much larger than the difference in memory-access cost that it swamped it completely. The instrument was louder than the signal. Rewriting both loops to use row pointers, so that the only remaining difference was the access order, revealed the 10.9× gap.

Keep that failure. When a benchmark reports that something you have good reason to expect is absent, the first hypothesis should be that your measurement is not measuring it. Module 3 returns to this with real profiling tools, and Module 8 makes a discipline of it; the habit is worth starting now.

4Implementation: the framebuffer

The engine gains a third subsystem and its first gfx/ directory — the one that grows into the software rasterizer, and eventually into the SDL_GPU renderer.

C++ in place — the buffer owns its memory, and frees it for you

This is the first time the engine allocates memory that outlives a function. The C way is malloc and a matching free; the C++-with-a-C-accent way is new[] and a matching delete[]. We do neither:

A std::vector allocates when you fill it and releases when it is destroyed — including when the destruction happens because a function returned early, or because an object containing it went out of scope. There is no matching call to forget, on any path.

That pattern is RAII: resource acquisition is initialisation. A resource's lifetime is tied to an object's lifetime, so the compiler enforces cleanup instead of your discipline doing it. You have already seen the alternative in main, where the window, renderer and now the texture are each created by hand and destroyed by hand — and where every early-return path has to repeat the cleanup, which is why those early returns are so repetitive and so easy to get wrong.

Module 5 wraps those SDL handles in RAII types and deletes all that repetition. It is left standing for now on purpose: the contrast between main's manual bookkeeping and this one-line vector is the argument, and it is much more convincing seen than described.

4.1 The interface, and what it deliberately exposes

Three drawing operations, and an escape hatch:

put_pixel is bounds-checked and row is not — that asymmetry is the design, not an oversight. An API that is only safe forces you to write slow code; an API that is only fast forces every caller to get bounds right or corrupt the image. Offering both, with the safe one as the obvious default and the fast one documented as the deliberate escape hatch, is the shape this engine's drawing API keeps.

Even the escape hatch is not fully sharp. row() validates the row index and clamps it, so a caller with bad arithmetic scribbles within one row of the image rather than somewhere loose in the process. You give up the per-pixel check because the loop already guarantees it; you do not give up the containment.

4.2 fill_rect: clip once, then stop checking

fill_rect is where the two ideas of this lesson meet. The naive implementation calls put_pixel in a nested loop and pays the bounds check on every pixel. Ours clips the rectangle against the buffer once, up front, and then fills without any checks at all:

Four comparisons replace two-per-pixel — for a full-screen rectangle, four instead of 115,200. And std::fill_n on a contiguous run is the shape the standard library and the hardware both want: it compiles down to a wide vectorised store, which is why the measured fill_rect of the whole screen (0.0027 ms) beats even the hand-written row-pointer gradient loop.

The early return for a fully off-screen rectangle is not just an optimisation. Without it, x1 - x0 could be negative, and std::fill_n with a negative count is undefined behaviour — one of those cases where the "obvious" guard and the "correct" guard turn out to be the same line for different reasons.

4.3 The demo: everything you see, you drew

The 1.4 demo carries over unchanged in behaviour and completely changed in mechanism. The three timing boxes and the ball are now fill_rect calls into our buffer; the background is a gradient computed per pixel; and the ball leaves a trail of individual pixels — one per simulation step, so the spacing between dots is the fixed timestep of Lesson 1.4 made visible. Change the simulation rate with 1–4 and the trail's spacing changes with it.

One line in the render section is worth pointing at, because it is the line that is not there:

Skipping a redundant clear is a real optimisation that real engines make. It is also a trap: the moment a background stops covering the whole frame, the previous frame shows through as smearing, and the fix is a clear() that was removed for good reasons a year earlier. Module 2's rasterizer will need it back the moment we draw a scene rather than a full-screen gradient.

Finally, note that the text readout is still drawn by SDL_RenderDebugText, over our framebuffer rather than into it. It is now the only thing on screen we did not draw ourselves — we have no font, and will not until Module 6. A slightly uncomfortable seam, and an honest one.

6Build & Run

What you should see

A window filled with a soft two-axis gradient — dark at the top left, brightening right and down — with three chunky boxes tracking across it, a ball bouncing between two guide lines, and two columns of single-pixel dots marking the ball's path. Every one of those pixels came out of our buffer.

  1. Look closely at the gradient. It is visibly blocky: each 4×4 block on screen is one pixel in our 320×180 image. You are looking at individual writes.
  2. Press G. The image does not change — but the millisecond figure in the readout does, by roughly 5× in a Debug build. Same picture, same pixels, different addressing.
  3. Press 1 for a 10 Hz simulation. The trail dots spread out dramatically: each dot is one simulation step, so at 10 Hz there are six times fewer of them per second. Press 4 and they crowd together. The fixed timestep, drawn.
  4. Resize the window. Nothing in our code responds, and everything scales correctly — the framebuffer's size is now independent of the window's.
  5. Hold T. The 1.4 machinery is all still running underneath: the sage box stays smooth, the amber one judders, and the ball's bounce heights do not change.
Checkpoint — verified before publishing

Built warning-clean at -Wall -Wextra with Apple Clang against SDL 3.4.12; runs and shuts down cleanly. The framebuffer was driven through a 20-assertion harness confirming, among others:

  • pack_argb(0x12, 0x34, 0x56, 0x78) produces exactly 0x78123456.
  • A pixel written at (3, 2) in an 8-wide buffer lands at raw index 19, and its neighbours are untouched — §3.1's formula checked against memory rather than reasoning.
  • All four out-of-range writes — past each edge — are dropped, and the buffer stays entirely clean.
  • The naive index for (8, 2) is 24, which is (0, 3): the wrap of §3.2, computed rather than asserted.
  • A rectangle straddling the top-left corner paints exactly 9 pixels of a 3×3 region and nothing outside it; fully off-screen and zero/negative-sized rectangles paint nothing.
  • row(y) addresses the same memory as put_pixel, consecutive rows are exactly width apart, and a wild row index is clamped into the buffer.
  • The two gradient paths produce byte-identical 320×180 images (memcmp over all 230,400 bytes).

The performance figures in §3.5 are medians of repeated runs on an Apple M4 Pro at -O2 unless labelled otherwise. Your ratios will differ; the ordering should not.

7Common Pitfalls & Debugging

The image is sheared diagonally — everything slants, progressively worse down the screen
Symptom
The picture leans, as if each row were shifted a little further than the last. Shapes are recognisable but skewed.
Cause
A pitch mismatch. Either the upload assumed the texture's pitch equals width × 4 when the driver padded its rows, or the index formula used the wrong width — a buffer resized in one place but not another.
Fix
Copy row by row using the pitch SDL_LockTexture actually returned (§3.4). If you are debugging it, print dst_pitch and fb.pitch() side by side — a shear is nearly always those two numbers disagreeing.
Shapes wrap around to the opposite edge — candy-striping down the picture
Symptom
A rectangle drawn near the right edge leaves fragments down the left edge, one row lower each time.
Cause
An x-coordinate past width, written without a bounds check. y × width + x happily rolls over into the next row (§3.2).
Fix
Use put_pixel, or clip before writing as fill_rect does. When you find one of these, check the y direction too — a stray y writes outside the buffer entirely, which is undefined behaviour rather than a stripe.
The colours are wrong — red and blue swapped
Symptom
Everything is recognisable but the colours are off in a very specific way: what should be red renders blue and vice versa, while greys and greens look fine.
Cause
A format mismatch between what you pack and what the texture expects — almost always an ARGB/ABGR confusion, which swaps exactly the red and blue channels and leaves green where it was. Often introduced by copying code that used an SDL_PIXELFORMAT_RGBA32-style byte-order name for a packed-integer buffer (§3.3).
Fix
Pick one view and stay in it. We build pixels as integers with shifts, so we use the packed-integer name SDL_PIXELFORMAT_ARGB8888. "Green looks right, red and blue are swapped" is the fingerprint — it tells you the problem is channel order and not, say, gamma.
The previous frame smears — motion leaves trails you did not ask for
Symptom
Moving objects leave streaks. Or, more confusingly, the screen fills with garbage that changes between runs.
Cause
Not every pixel is being written each frame. Either a clear() was removed once a full-screen background made it redundant and the background later stopped covering everything, or you are relying on the locked texture retaining last frame's contents — which SDL explicitly does not promise (§3.4).
Fix
Either clear, or guarantee full coverage. Choose deliberately and leave a comment saying which, because the next person to change the background needs to know the invariant exists.
Everything is blurry after upscaling — soft pixel-art
Symptom
The 320×180 buffer scaled to 1280×720 looks smeared rather than crisp and blocky.
Cause
The texture is using linear filtering, which averages neighbouring pixels when magnifying.
Fix
SDL_SetTextureScaleMode(texture, SDL_SCALEMODE_NEAREST). For real pixel-art projects also look at SDL_SCALEMODE_PIXELART (SDL 3.4+), which handles non-integer scale factors more gracefully than either.
A "no difference" benchmark result — the optimisation you expected does nothing
Symptom
You compare two approaches you have good reason to think differ, and measure them as identical.
Cause
Something common to both paths dominates the measurement — as happened while writing this lesson, where put_pixel's overhead hid a genuine 10.9× cache effect (§3.5). Or: you measured a Debug build, or the compiler deleted the loop because its result was unused.
Fix
Make the two paths differ in only the thing under test, measure at -O2, ensure the results are consumed so nothing is optimised away, and take a median of several runs. A surprising null result deserves suspicion before it deserves belief.

8Exercises

1.5.1Cause the stripewarm-up

Add a deliberately unchecked pixel setter to framebuffer — one that skips in_bounds — and use it to draw a horizontal run that starts near the right edge and overshoots by ten pixels. Predict what you will see before you run it, then run it.

Solution

The run's overshoot appears at the far left of the row below, exactly as §3.2 predicts. Do it on several rows and you get the candy-stripe of Figure 2. It is worth seeing once, in a program small enough that you know it is not a rendering bug, so that you recognise the pattern instantly when it shows up in something complicated. Remove the unchecked setter afterwards — or keep it private and clearly named, as real rasterizers do.

1.5.2Change the resolutionwarm-up

Set k_fb_width and k_fb_height to 160×90, then to 1280×720. What changes on screen, what changes in the gradient timing, and what does not change?

Solution

At 160×90 the pixels are 8× blocks and the gradient fill takes about a quarter as long (a quarter the pixels). At 1280×720 the image is pixel-perfect on the window, the blockiness vanishes, and the fill costs roughly sixteen times the 320×180 figure. What does not change is anything else: the simulation, the timing, the layout arithmetic. The framebuffer's size is genuinely a parameter, which is what makes dynamic resolution possible in shipping games — drop the render target when the frame budget is tight, keep the game identical.

1.5.3Reproduce the cache measurementguided

Write a small benchmark that fills the whole buffer twice — once with the row loop outer, once with the column loop outer — using row pointers in both. Time each at -O2, take a median, and compare your ratio with the table in §3.5. Then repeat at 1920×1080.

Hint

Both loops must write through fb.row(y)[x], not put_pixel — that is the entire point of the exercise. Make sure the compiler cannot delete the work: fills that nothing ever reads are legal to remove entirely.

Solution

Expect a ratio somewhere between 5× and 50×, growing with the buffer size. The exact numbers depend on your cache hierarchy — check yours with sysctl hw.l2cachesize on macOS, lscpu on Linux, or Task Manager's performance tab on Windows — but the direction is universal.

Then do the thing that makes this exercise worth its time: rewrite both loops with put_pixel and watch the difference collapse to nothing. You will have reproduced the false result described in §3.5, which is a more useful thing to have experienced than the true one.

1.5.4Draw a lineguided

Add draw_line(int x0, int y0, int x1, int y1, Uint32 colour) using the simplest thing that works: step whichever axis changes more, and compute the other with floating-point interpolation. Draw a few lines from the centre of the screen at various angles. Then look carefully at a nearly-horizontal one and a nearly-vertical one.

Hint

dx = x1 - x0, dy = y1 - y0. Take steps = max(|dx|, |dy|) and advance by dx/steps, dy/steps each iteration, rounding to the nearest pixel.

Solution sketch

Stepping the longer axis is what guarantees no gaps: step the shorter one and the line comes out dotted. Lines will look jagged — that is aliasing, and it is a genuine artefact of representing a continuous line on a discrete grid rather than a bug.

This is deliberately the naive version. Lesson 2.1 opens by deriving exactly this algorithm (DDA), then shows why Bresenham replaced it — the floating-point accumulation you just wrote is doing work the integers can do exactly. Writing the naive one now means that lesson starts from something you already own.

1.5.5Make the trail fadeopen

The ball's trail is currently a uniform colour. Make older dots dimmer, so the trail fades out behind the ball. Then consider: what would it take to make the trail blend with the gradient behind it rather than replacing those pixels?

Solution sketch

Fading is straightforward — the ring buffer knows each dot's age, so scale the colour components by 1 - age/length and pack the result. Watch out for the ring buffer's wraparound when computing age; off-by-one there produces a trail with a bright dot in the middle of the faded part.

Blending is the more interesting half, and it is a real preview. To blend you must read the existing pixel, combine it with the new one weighted by alpha, and write the result back: out = src × a + dst × (1 − a). That is alpha compositing, it is why pixel_at exists on a class otherwise concerned with writing, and it is the same equation that will appear in Module 6 for transparent surfaces.

One trap worth discovering now: doing that arithmetic directly on the stored 0–255 values is subtly wrong, and blended edges will look too dark. The reason is the subject of Lesson 1.6, which is next.

9Recap & Next

What the engine can do now that it could not before

  • It owns an image. A block of memory whose every byte has a meaning we defined, and which nothing else may interpret for us.
  • It can set any pixel to any colour — safely by coordinate, or fast through a row pointer, with the tradeoff between them measured rather than assumed.
  • It presents that image properly: a streaming texture, locked, copied row by row using the driver's pitch, drawn scaled with nearest-neighbour filtering.
  • It has a resolution independent of the window, so resizing costs no code and render scale is a parameter.
  • You can read a pixel format name and know whether it is describing bits in an integer or bytes in memory.

The idea to carry forward: two-dimensionality is a convention, not a fact. index = y × width + x is the whole agreement, and every artefact in this lesson — the stripe, the shear, the swapped channels — is what it looks like when two pieces of code disagree about it.

Next: 1.6 — Colour, and an Honest Teaser of sRGB. We have been treating those 32 bits as a bag of numbers, and they are not. The value 128 is not half as bright as 255, blending two colours by averaging their stored values gives the wrong answer, and the reason is a transformation applied to every image you have ever seen. We take the first honest look at it, and pay it off completely in Module 6.
