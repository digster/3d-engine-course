Events, Properly

In 0.5 we drained the event queue with two if statements and a shrug. Now we do it properly: what an SDL_Event actually is, how SDL organises the hundred-odd kinds of them, and why the difference between while and if is the difference between a responsive game and a frozen one.

1The Problem

Our event loop from 0.5 handles exactly two things and silently discards everything else:

It works, and it taught the shape of a loop. But it leaves four real questions unanswered, and each of them will bite something we build later:

  • What is actually in an SDL_Event? We read event.key.key on faith. Why is that safe on a key event and not on a mouse event?
  • What else can arrive? Mouse, gamepad, text, display changes, drag-and-drop — dozens of kinds. How are they organised?
  • Why while and not if? Both compile. Both appear to work.
  • Is SDL_EVENT_QUIT really the right thing to check? There is a separate window-close event. Which is correct?

The third question has the most spectacular answer, so let us look at it first — because it is a bug you can create by changing one word:

The failure mode: the queue that outruns you

Change the while to an if. The program still compiles, still runs, still closes on Esc. Then you move the mouse — and the game starts responding to your inputs seconds late, drifting further behind the longer you play, until it feels like a frozen application that is somehow still drawing.

Nothing crashed. What happened is that events arrive faster than one per frame, and an if removes exactly one per frame. The queue grows without bound, and every event you eventually process is a stale report of something the player did ages ago. §3.3 works the arithmetic — and the numbers are far worse than intuition suggests.

2Building Intuition

2.1 A mailbox, not an interrupt

There are two ways a system can tell your program that something happened, and knowing which one SDL uses explains the entire shape of the loop.

The first is an interrupt (or a callback): the moment the user presses a key, the system calls your function, wherever you happen to be. This is how hardware interrupts work, and how many GUI toolkits present themselves. It sounds convenient and it is a nightmare for a game: your handler could fire in the middle of physics, halfway through updating a position, on some other thread — and you would have to make every piece of state safe against being interrupted at any instant.

The second is a queue: the system quietly writes the event down and adds it to a list. Nothing in your program is interrupted. Later, at a moment you choose, you read the list. This is SDL's model, and it is the mailbox from 0.5 §2.1, now stated precisely.

The queue model is what makes a game loop tractable. Input arrives at unpredictable times, but you consume it at exactly one predictable point in the frame — the top, before simulation. Everything downstream can then assume input is fixed for the duration of the frame. That property is worth a great deal: it is what will let Module 1.4's fixed-timestep simulation be deterministic, and it is why we never let event handling scatter itself across the frame.

2.2 One envelope, many letters

Now the shape of an event. SDL must deliver wildly different things through one function: a key press carries a keycode and modifiers; a mouse move carries coordinates; a window resize carries a new width and height; a dropped file carries a path. Yet SDL_PollEvent takes one SDL_Event*.

The trick is an envelope that is always the same size and always has the same label on the front, but whose contents differ. The label — event.type — tells you which kind of letter is inside, and therefore which fields are meaningful. Read the label first, then open the matching compartment. Read the wrong compartment and you get whatever bytes happened to be there: not a crash, just nonsense, which is worse.

In C this pattern is a tagged union, and SDL implements it with a guarantee we can measure directly. That measurement is the next section, and it is more concrete than most explanations of unions you will read.

3The Theory

3.1 The tagged union, measured

SDL_Event is a union: about forty differently-shaped structs occupying the same memory. Only one is valid at a time, and type says which.

C++ in place — union

A struct lays its members out one after another, so its size is roughly the sum of them. A union overlays its members at the same address, so its size is the largest of them, and writing one member overwrites the others. That makes a union a way of saying "this memory holds exactly one of these possibilities."

The catch is that the union itself does not remember which possibility is live — that is the programmer's job. Pairing a union with a field that records the active member is the tagged union, and reading a member other than the one the tag names is undefined behaviour in C++ (in practice: garbage). C++17's std::variant is the type-safe modern version; SDL is a C library, so it uses the classic form, and the discipline falls to us — which is exactly why our dispatch is a switch on the tag with each case touching only its own member.

For that to work at all, event.type must be readable no matter which member is live. SDL guarantees this by starting every single variant with the same header. We do not have to take that on faith — we can measure it with offsetof, and here are the real numbers from the SDL we pinned:

Look at the first column: type sits at offset 0 in every variant, and timestamp at offset 8 in every variant. That is the whole guarantee. The first sixteen bytes of an SDL_Event mean the same thing regardless of what kind of event it is, so reading event.type is always valid — and after that, the tails diverge. Figure 1 draws it.

Why 128 bytes, when the largest variant here is 48?

The union is padded to a fixed size on purpose. SDL's header explains: MSVC and GCC compute union sizes differently, so SDL adds explicit padding "to force the size to be the same for both" — an ABI concern, so that a library and an application built by different compilers agree on how big an event is. It is a small, real example of the kind of constraint that shapes public C APIs, and worth noticing now: our own engine will face the same question when it grows a public API in Module 5.

3.2 The event taxonomy

SDL defines well over a hundred event types. You will never memorise them, and you do not need to, because they are numbered by category — each family gets its own block of values:

BlockCategoryExamples (verified in SDL_events.h)
0x100ApplicationSDL_EVENT_QUIT = 0x100
0x15xDisplaymonitor added/removed, orientation
0x2xxWindowSDL_EVENT_WINDOW_SHOWN = 0x202, …RESIZED, …CLOSE_REQUESTED
0x3xxKeyboardKEY_DOWN = 0x300, KEY_UP = 0x301, TEXT_EDITING = 0x302, TEXT_INPUT = 0x303
0x4xxMouseMOUSE_MOTION = 0x400, MOUSE_BUTTON_DOWN = 0x401, MOUSE_BUTTON_UP = 0x402, MOUSE_WHEEL = 0x403
0x6xxJoystickJOYSTICK_AXIS_MOTION = 0x600
0x65xGamepadGAMEPAD_AXIS_MOTION = 0x650
0x8000User-definedSDL_EVENT_USER = 0x8000 — yours to push

SDL also defines range markers — SDL_EVENT_WINDOW_FIRST and SDL_EVENT_WINDOW_LAST — so you can test "is this any window event?" with a range comparison instead of a fifteen-case switch.

Worked example — classifying an event you have never seen

You are debugging, you print event.type, and it reads 770. What is it?

  • Convert to hex: 770 = 0x302. (Because 770 = 3·256 + 0·16 + 2.)
  • It falls in [0x300, 0x400) — the keyboard block.
  • Counting from KEY_DOWN = 0x300: 0x301 is KEY_UP, so 0x302 is SDL_EVENT_TEXT_EDITING.

You classified it without looking anything up, and confirmed it by counting two entries down the header's list. That is the practical value of a numbered taxonomy: an unknown event is never completely unknown, because its number tells you which family it belongs to and therefore which union member would be the one to read.

3.3 Why you must drain to empty

Now the arithmetic promised in §1. The question: what does changing while to if actually cost?

Events do not arrive one per frame. A modern gaming mouse reports at 1000 Hz — a thousand motion events every second — while your game runs at 60 frames per second. So each frame, roughly seventeen mouse events are waiting for you before you count anything else.

Worked example — the compounding backlog

A 1000 Hz mouse, a 60 fps game, and an if instead of a while:

  • Arriving per frame: 1000 / 60 = 16.67 events.
  • Drained per frame: 1 (that is what if means).
  • Net growth: 16.67 − 1 = 15.67 events added to the queue, every frame.
  • After one second (60 frames): 60 × 15.67 ≈ 940 events backed up.
  • Time to clear that backlog, draining 60 per second: 940 / 60 ≈ 15.7 seconds.

So one second of mouse movement puts you roughly sixteen seconds behind, and every further second of movement adds another sixteen. The lag does not merely exist — it compounds. Within a minute the program is processing input from a minute ago; it is an application that has, for all practical purposes, stopped responding, even though it is still cheerfully drawing frames.

With while, the same 16.67 events per frame are all drained in the same frame. The queue returns to empty every frame, latency stays at a fraction of a frame, and the number of events simply does not matter. One keyword; the entire difference.

Note the corollary, which surprises people: draining is mandatory, but reacting is optional. An event you do not care about still has to come off the queue. Our default: case does nothing at all — and that is not laziness, it is the entire job. The queue is emptied whether or not anything is done with the contents.

3.4 Quit vs window-close

Finally, the two events that both mean "we're done", and are not the same thing:

EventScopeMeans
SDL_EVENT_QUIT Application "This program should terminate." Not tied to any window.
SDL_EVENT_WINDOW_CLOSE_REQUESTED One window The header's words: "the window manager requests that the window be closed." It names which window in event.window.windowID.

With a single window the distinction is invisible — either one means quit, which is why 0.5 got away with checking only SDL_EVENT_QUIT. The moment you have two windows it becomes essential: closing a tool palette must close that window, not the whole program. Handling both now, distinctly, costs three lines and builds the right habit.

A quit you did not expect: Ctrl-C

SDL installs signal handlers, so pressing Ctrl-C in the terminal (SIGINT), or a kill (SIGTERM), does not tear your process down mid-frame — SDL converts it into an SDL_EVENT_QUIT that arrives through the normal queue. Your ordinary handler runs, the loop exits, and the destroy-and-quit path executes cleanly.

This is not a claim from the docs: while verifying this lesson, the engine was run under a timeout that sends SIGTERM, and it printed Quit requested and shut down through the normal path. Handling SDL_EVENT_QUIT properly gives you clean Ctrl-C termination for free.

4Implementation: a real dispatch

Replace the two ifs with a switch on the tag. The structure matters as much as the cases: tag first, then the matching member, and nothing else.

Three things in there are deliberate. The switch reads event.type before touching any variant member — Figure 1's rule made structural. Each case then touches only its own member: event.window in the window cases, event.key in the keyboard case, and never the two mixed. And the default: case is an explicit, empty statement of intent: drained, deliberately ignored.

The resize case finally pays off the exercise from 0.5.2. The header documents SDL_EVENT_WINDOW_RESIZED as "Window has been resized to data1xdata2", so event.window.data1 and data2 are the new width and height — a fact we read from the header rather than guessed, exactly as 0.6 taught.

Why not extract this into its own file yet?

It is tempting to move event handling into src/platform/events.cpp right now. We are deliberately not doing that. The engine's internal structure grows when a boundary earns its existence (0.1 §3.4: a boundary belongs where a rate or an owner changes), and right now nothing has changed hands — this is still one program, one loop. In 1.2 we build an input layer that genuinely owns state across frames, and that is a real owner boundary. Premature file-splitting is architecture theatre, and Module 5 exists precisely so the real refactor is done once, with reasons.

7Common Pitfalls & Debugging

if instead of while — input lags further behind the longer you play
Symptom
The game responds to clicks and keys seconds after they happen, and it gets worse over time. It looks like a performance problem; it is not.
Cause
The inner poll is an if, so exactly one event is removed per frame while many more arrive. The queue grows without bound (§3.3: about 940 events backed up after one second of mouse movement).
Fix
Make it while (SDL_PollEvent(&event)). If you want to see the problem first, count iterations of the inner loop and log the count — a healthy frame drains a handful; a sick one drains one while dozens pile up.
Reading the wrong union member — plausible but wrong values
Symptom
A coordinate or keycode holds a value that is not nonsense enough to look like corruption, but is definitely wrong — mouse positions that jump, keys that are not the key you pressed.
Cause
You read event.motion (or event.key, or …) on an event whose type says otherwise. All variants overlap the same bytes (Figure 1), so you get whatever the live variant happened to store there. This is undefined behaviour, and its punishment is bad data rather than a crash.
Fix
Only touch a member inside the case that its type authorises. If you find yourself reaching for a member outside its case, that is the bug.
SDL2's SDL_WINDOWEVENT does not exist — "undeclared identifier"
Symptom
Code copied from an SDL2 tutorial checks event.type == SDL_WINDOWEVENT and then switches on event.window.event. Neither compiles.
Cause
SDL2 had a single SDL_WINDOWEVENT type with a sub-type inside the struct. SDL3 flattened this: every window event is its own top-level type — SDL_EVENT_WINDOW_RESIZED, SDL_EVENT_WINDOW_CLOSE_REQUESTED, and so on — so they sit directly in your main switch.
Fix
Use the flat SDL3 types. This flattening is a genuine improvement: one switch, no nested dispatch. The migration guide lists the mapping.
Handling only SDL_EVENT_QUIT in a multi-window app — closing one window kills everything
Symptom
You add a second window (a tools palette, say) and closing it terminates the whole program.
Cause
Treating every close as application-scoped. SDL_EVENT_WINDOW_CLOSE_REQUESTED names the specific window in event.window.windowID; ignoring that ID throws the information away.
Fix
Compare event.window.windowID against the window you mean (SDL_GetWindowID gives you a window's ID) and destroy just that one. Not needed today with a single window — but the structure is already in place for when it is.
Trying to handle every event type — a switch that never ends
Symptom
The dispatch grows toward a hundred cases in pursuit of "completeness", and most do nothing.
Cause
Confusing draining with reacting. The queue does not care whether you interpret an event; it only cares that you remove it.
Fix
Handle what your game needs and let default: absorb the rest. Add a case the day you need it. An empty default is a complete and correct answer to "what about all the others?".

8Exercises

1.1.1See the floodwarm-up

In default:, log the event type in hex: SDL_Log("unhandled 0x%X", event.type);. Run it, move the mouse for two seconds, then stop and read the output. Which categories do you recognise from §3.2, and roughly how many events arrived?

Solution

You will see a torrent of 0x400 (mouse motion), plus 0x200-block window events when the cursor enters or leaves, and possibly 0x40x button events. The sheer volume from two seconds of mouse movement is the point: this is §3.3's arithmetic made visible, and it is the best argument for while that exists. Remember to remove the log afterwards — 0.6's pitfall about print-debugging that never leaves.

1.1.2Create the lag on purposeguided

Change the inner while to an if, rebuild, and use the program: move the mouse for a few seconds, then press Esc. What happens, and how long does it take? Then put it back.

Solution

Esc does not appear to work — for a long time. The key-down event is real and is sitting in the queue, but it is behind the hundreds of mouse-motion events that arrived first, and the program is retiring them at one per frame. Eventually it reaches your keypress and quits. You have just experienced latency as a queueing problem rather than a speed problem: the program was never slow, it was behind. That distinction matters again in Module 8 when we profile for real.

1.1.3Test a whole categoryguided

Add a branch that catches any window event, using the range markers SDL_EVENT_WINDOW_FIRST and SDL_EVENT_WINDOW_LAST, and logs it. Why is a range test sometimes better than fifteen case labels — and when is it worse?

Hint

A switch case cannot express a range portably, so this one wants an if before or after the switch: if (event.type >= SDL_EVENT_WINDOW_FIRST && event.type <= SDL_EVENT_WINDOW_LAST) { … }.

Solution sketch

A range test is better when you want uniform treatment of a family — logging every window event, or forwarding them all to a window manager object — because it stays correct when SDL adds a new window event type. It is worse when you actually need to distinguish members of the family, because it hides which ones you handle: a reader cannot tell from the range test whether resize is covered. Use ranges for categorical decisions and explicit cases for behavioural ones.

1.1.4Measure an event's ageopen

Every event carries a timestamp in nanoseconds (Figure 1 — it is in the common header, so it is available on any event). Compare it against SDL_GetTicksNS() when you process the event, and log the difference. What is a typical age with while? What happens to it with if?

Solution sketch

With while, ages are small — a fraction of a frame, since events are consumed very soon after they are queued. With if, the age climbs without limit, and watching that number grow is the most direct possible measurement of the backlog: it is input latency, in nanoseconds, reported by the events themselves.

This little experiment is worth more than it looks. It is the first time in this course we measure a time rather than reason about one, and it foreshadows the whole of 1.3 and 1.4 — where the difference between "how fast" and "how late" becomes the central design problem of the loop.

9Recap & Next

What the engine can do now that it could not before

  • It dispatches events properly. A switch on the tag, each case touching only its own union member, with quit, window-close, resize, and keyboard handled distinctly.
  • You understand the envelope. SDL_Event is a tagged union whose first sixteen bytes are identical across every variant — measured, not assumed — which is precisely why event.type is always safe and everything else is conditional.
  • You can navigate a hundred event types by category, and classify an unknown one from its number alone.
  • You can prove why the loop drains to empty — and roughly how catastrophic if would be.

The idea to carry forward: draining is mandatory; reacting is optional. The queue is a mailbox you must empty every frame, whatever you decide to do with the letters.

Next: 1.2 — Input: State vs Events. Events tell you what happened: "the W key went down." But gameplay usually wants to know what is: "is W held right now?" Those are different questions with different bugs, and answering them from the same event stream requires keeping state across frames — which is the first thing in this engine that genuinely owns something, and therefore the first thing that has earned its own module.
