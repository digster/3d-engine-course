Input: State vs Events

Lesson 1.1 left us with a clean event dispatch, and it is about to fail us. Not because it is wrong — it is exactly right — but because the question gameplay asks is not the question an event answers. An event says the W key went down. Movement needs to know is W down right now. Those two sentences are one word apart and produce completely different bugs. Today we answer both, from one snapshot, and build the engine's first real subsystem to hold the answer.

1The Problem

We have a working event dispatch. Let us use it for the most ordinary thing a game does: move something when a key is held. The obvious code writes itself — we already have a SDL_EVENT_KEY_DOWN case, so we put the movement in it:

Build it, hold D, and watch. The box jumps once. Then it sits there, for what feels like an embarrassingly long time. Then it starts sliding — but in lurches, far slower than it should, and it never becomes smooth. Let go and press again: one jump, long pause, lurching.

Nothing is broken. Every line does exactly what it says. What you are watching is the operating system's key repeat: the mechanism that lets you hold a key in a text editor and get aaaaaaaa. The OS sends one key-down when the key goes down, waits out a repeat delay, and then starts synthesising further key-downs at a repeat rate. SDL faithfully forwards all of them. Figure 1 shows the shape of what arrives.

Now the number, because "it feels wrong" is not an engineering statement.

Worked example — how far the box actually gets

A 60 fps game, a key held for exactly one second, with a typical repeat delay of 0.5 s and a repeat rate of 30 per second. Movement is 6 px per event.

  • The initial press: 1 event, at t = 0.
  • The delay: from t = 0 to t = 0.5 s, no events at all — that is 30 frames of a held key producing nothing.
  • The repeats: 30 per second for the remaining 0.5 s = 0.5 × 30 = 15 events, one every other frame.
  • Total: 1 + 15 = 16 movements, so 16 × 6 = 96 px.

Polling the key's state once per frame instead gives 60 movements — 360 px — and every frame contributes, so the motion is smooth. The event-driven version travels 3.75× less far, and stutters getting there.

And here is the part that should genuinely alarm you: the repeat delay and repeat rate are settings in the player's operating system. On both Windows and macOS a user can drag the repeat rate down to roughly 2 per second. That player gets 1 + 1 = 2 movements in the first second instead of 16 — eight times slower than the player sitting next to them, in the same build, on the same hardware, because of a preference they set for typing. Your game's movement speed is not allowed to live in the Keyboard control panel.

Filtering the repeats does not save it either. SDL marks synthesised repeats with event.key.repeat == true, so you could ignore them — and then a held key produces exactly one movement, ever. The box moves 6 px and stops. The problem was never which events to keep. The problem is that "held" is not an event at all. It is a condition that persists between events, and nothing in the event stream represents it.

Meanwhile, the mirror-image bug is waiting on the other side. Suppose we fix movement by polling, and then wire jumping the same way: "if the jump key is down, jump." Hold the key and the character jumps sixty times a second. A menu cursor wired this way blurs from the top of the list to the bottom on the lightest tap. So polling is not simply better than events; it is different, and using it for the wrong thing produces its own family of bugs. We need both, and we need to know which is which.

2Building Intuition

2.1 The doorbell and the light switch

Two devices in an ordinary house, and the whole lesson is in the difference between them.

A doorbell is about the moment of pressing. It rings once when pressed. Holding the button in does not produce a continuous ring that means "someone is still standing there" — and if you leave the house and come back, there is no way to look at the doorbell and learn whether it was pressed while you were out. A doorbell reports transitions, and if nobody is listening at the instant it rings, the information is gone.

A light switch is about the condition. It has no memory of being flipped and no interest in when. It simply is up or down, and you can walk in at any hour and read it. A switch reports a state, and the state is still there whenever you get around to looking.

Events are doorbells. The state array we are about to meet is a light switch. And notice the asymmetry, because it decides the design: if you watch the doorbell continuously, you can reconstruct the switch — every ring tells you it flipped, so you can maintain your own record of the current position. The reverse does not hold nearly as cleanly: from the switch alone, glanced at occasionally, you cannot know how many times it was flipped between glances.

That asymmetry says the state should be primary and the transitions derived from it — provided we glance often enough. We glance every frame, which turns out to be enough for everything except a few genuinely special cases we will name in §3.6.

2.2 Two questions, one honest answer each

Write out what gameplay actually asks, and the code sorts itself into two piles:

The actionThe question it asksKind
Walk forward while W is heldIs W down right now?Level
Aim down sights while the right button is heldIs the button down right now?Level
JumpDid Space go down this frame?Edge
Open the pause menuDid Esc go down this frame?Edge
Release a charged shotDid the fire button come up this frame?Edge
Move a menu cursor one itemDid Down go down this frame?Edge

The test is mechanical, and it is worth internalising because it never stops being useful: if holding the key longer should do more, it is a level. If holding it longer should do nothing extra, it is an edge. Walking further by holding longer: level. Jumping twice by holding longer: absurd, so edge.

Both piles are about the same physical key at the same instant. So they had better come from the same underlying data, or they will eventually disagree — and an engine in which key_down(W) and key_pressed(W) tell contradictory stories is an engine with a bug nobody will find quickly.

3The Theory

3.1 Levels and edges, defined

These words are borrowed from electronics, where a signal is a voltage over time and you can build a circuit that reacts to either the voltage itself or to the instant it changes.

  • A level is the value of the signal now: high or low, down or up. It is true for as long as the condition holds — one frame or ten thousand.
  • An edge is a change in the level. A rising edge is the instant low becomes high; a falling edge is the instant high becomes low. There is exactly one rising edge per press, no matter how long the press lasts.

3.2 Deriving edges from state

Here is the move that makes everything else simple. We do not track edges. We derive them, from two things we already have: what the key is doing now, and what it was doing last frame.

SDL keeps a live array of key states for us, one entry per physical key, and hands it over on request:

An SDL2 habit that will bite you

In SDL2 this returned const Uint8 * and you tested state[k] != 0. SDL3 returns const bool *. Any snippet you find online older than SDL3 will declare the wrong pointer type, and since Uint8 and bool are both one byte on every platform we target, it may even appear to work — until it does not. Take the type from the header, not from a search result.

Keep one copy of that array as keys_ ("now") and one as keys_prev_ ("last frame"), and every question has a one-line answer:

truetruetruefalsefalseStill held. No edge; the press already happened.

Read the columns and the three definitions fall out. In plain words: down is just the current value; pressed is "true now and false before"; released is "false now and true before". In code, exactly as literal as it sounds:

That is the entire mechanism. There is no event handler for edges, no list of "keys pressed this frame" to maintain, nothing to clear at the right moment and forget to clear at the wrong one. One source of truth, two snapshots, three answers. Because the edges are computed from the same array the level query reads, it is structurally impossible for them to disagree — the class of bug where a character keeps walking after the jump code decided the key was up simply cannot occur.

It also means the queries are pure: asking does not change anything. You may call key_pressed(SDL_SCANCODE_SPACE) in five different systems in the same frame and all five get true. Compare that with the tempting alternative of a "consume the press" API, where the first caller wins and the other four silently see nothing — a race between subsystems that depends on call order. We will meet that hazard for real in Module 5; here we have avoided it for free.

One aside for later. pressed and released are both special cases of "the value changed", which is keys_[k] != keys_prev_[k] — an exclusive-or. If you ever pack key states into bit words instead of a byte per key, the whole frame's edge detection becomes changed = cur ^ prev; pressed = changed & cur; across sixteen keys at a time. We do not need that speed now, and reaching for it before it is needed would be exactly the premature optimisation Module 8 teaches you to avoid. It is worth knowing the door is there.

3.3 Why it must be a copy

Now the trap. Read the signature again: SDL_GetKeyboardState returns const bool * — a pointer, and the header is explicit that it points into SDL's own internals and stays valid for the life of the program. So the tidy-looking version of our update is:

It compiles cleanly. It runs. Movement works perfectly. And key_pressed returns false forever — nothing you press ever registers as a press, in any menu, for any key, with nothing in the debugger looking wrong.

The reason is that SDL_GetKeyboardState returns the same address every time. It is not a fresh array per call; it is a window onto the one array SDL keeps mutating. So after those two lines, keys_ and keys_prev_ hold the identical pointer value. They are two names for one array. Evaluate keys_[k] && !keys_prev_[k] and you are evaluating x && !x, which is false for every possible x. The code is not comparing two moments in time; it is comparing the present to itself. Figure 3 puts the two versions side by side.

C++ in place — a pointer names a thing; a copy is a thing

Assigning a pointer copies the address, which takes eight bytes and no time. Assigning an array copies the contents. Both are spelled with =, and that is exactly why this bug is so easy to write: the syntax gives no hint that one of them shares and the other duplicates.

The question to train yourself to ask at every assignment is "do I now have a second thing, or a second name for the same thing?" When you want a snapshot — a value frozen at a moment — a second name is useless, because the thing it names keeps changing underneath you. This is the same reasoning that will decide, in Module 5, whether a system stores a mesh, a mesh&, or a handle to one. Get it wrong there and you get dangling references; get it wrong here and you get an engine in which nothing is ever pressed.

So we copy: std::copy_n(sdl_keys, n, keys_.begin()). Five hundred and twelve bytes per frame, which at 60 fps is 30 kB/s — utterly negligible, and it buys correctness that no amount of cleverness with pointers can recover.

3.4 Scancodes and keycodes

SDL will describe a key to you in two entirely different languages, and choosing the wrong one is the most common internationalisation bug in games. In SDL_KeyboardEvent both are right there next to each other:

  • A scancode is a physical position. SDL_SCANCODE_W means "the key that sits where W is on a US QWERTY board" — second key of the top letter row — regardless of what is printed on it or what it types.
  • A keycode is a symbol. SDLK_W means "the key that produces the letter w on this user's layout", wherever that key happens to be.

On a US keyboard these two coincide, which is why the bug survives testing. On a French AZERTY keyboard they do not, as Figure 4 shows.

Hence the rule this course follows without exception: positions are scancodes; symbols are keycodes. Movement, and anything else where the shape under the hand is what matters, uses SDL_Scancode. Text entry and shortcuts you would describe to a player ("press Ctrl+S to save" — you mean the S key, the letter) use SDL_Keycode. This is also why our whole input API is typed in terms of SDL_Scancode: the array SDL hands us is indexed by scancode, and making the parameter type match makes the wrong call fail to compile rather than fail in French.

Why the state array can only be scancodes

It could not have been otherwise. Keycodes depend on the active layout, which the user can change at any moment — mid-game, with a hotkey. A fixed-size array indexed by "whatever letter this key currently produces" would have entries that move around under you. Physical positions do not move, so they are the only sane index for a persistent array. When you meet an API whose shape seems arbitrary, asking "what would the alternative have had to cope with?" usually explains it.

3.5 The ordering rule

SDL's keyboard array does not update itself continuously in the background. It updates when SDL pumps — when it drains the operating system's messages and turns them into SDL events, updating its internal state on the way past. The header for SDL_GetKeyboardState says so directly: use SDL_PumpEvents() to update the state array.

We never call SDL_PumpEvents ourselves, because SDL_PollEvent already does it — in SDL's source, SDL_PollEvent forwards to SDL_WaitEventTimeoutNS(event, 0), which pumps before it looks at the queue. That is convenient, and it also imposes an order on the frame, shown in Figure 5.

A frame of lag at 60 fps is 16.7 ms, which sounds ignorable and is not: it is added to every other source of latency in the chain, and it is the kind of "the controls feel mushy" complaint that is nearly impossible to diagnose from the outside. Getting it free by writing two lines in the right order is the best trade in this lesson.

The ordering has a second payoff that is easy to miss. Consider the classic bug: you hold W to run forward, then alt-tab to another window. Your application never sees a key-up — the key was released while another program had focus — so a naive input system believes W is still down, and the character runs into a wall until you come back. SDL fixes this for us, and the mechanism is worth knowing exactly:

Verified in SDL's source, not assumed

In src/events/SDL_keyboard.c, SDL_SetKeyboardFocus() detects the case where focus moves away from every SDL window and calls SDL_ResetKeyboard(), which walks the key state and — this is the important part — sends a key-up event for every key it finds held. The header for SDL_ResetKeyboard says the same thing: it generates key-up events for all pressed keys.

Because the fix arrives as events, it only reaches your input system if you drain the queue before sampling. Our order does. As a bonus, the released keys produce a proper falling edge, so a "on release, throw the grenade" handler still fires correctly when the player alt-tabs — which is a more sensible outcome than the alternatives.

3.6 What state cannot tell you

We have made state primary. Two honest limitations come with that, and pretending otherwise would be the hand-waving this course does not do.

First: a tap between two samples is invisible. The header is admirably blunt about it — if a key is pressed and released before you process events, the pressed state never shows up in SDL_GetKeyboardState. Press and release W within one 16 ms frame and the snapshot at each end reads "up" both times. No level, no edge, nothing happened.

In practice this is close to a non-issue for a keyboard: the fastest human key tap is on the order of 30–50 ms, comfortably more than one frame at 60 fps. It matters more than you would like on two occasions — when the frame rate collapses (a 200 ms hitch will eat taps, and it is a real complaint in stuttering games), and in Lesson 1.4, where a fixed-timestep loop can run several simulation steps between input samples. We will name it again there rather than let it ambush you. The rigorous fix, when a game truly needs it, is to feed the edges from the event stream (which never loses a transition) instead of from sampling — the doorbell reconstructing the switch, from §2.1. We are not paying that complexity now, and I would rather tell you the boundary than let you assume there isn't one.

Second: some inputs have no state to poll at all. Ask yourself what mouse_wheel_position() would even return. There is no such quantity: a wheel is not somewhere, it only ever moves. Nothing in SDL can tell you "how scrolled" the wheel is, because the wheel itself does not know. The same is true of typed text (a stream of characters, not a condition) and of relative mouse motion in a first-person camera (a delta since the last report).

So the input system needs both halves after all — but the split is principled rather than arbitrary:

QuantityHas a "current value"?How we get it
Key down / upYes — the key is up or downPoll, once per frame
Mouse button down / upYesPoll, once per frame
Cursor positionYes — the cursor is somewherePoll, once per frame
Wheel movementNo — only deltas existAccumulate from events
Typed textNo — a streamAccumulate from events

That last column is the design. Anything with a current value gets sampled once per frame; anything that only exists as change gets caught from the queue as it goes past. Both halves are then published together, so that from the outside there is one coherent snapshot of "input, this frame" — and no caller ever has to know which mechanism produced which field.

4Implementation: the input subsystem

This is the first thing we build that does not belong in main.cpp. Input is a genuine subsystem: it owns state, it has an invariant ("the snapshot is refreshed exactly once per frame"), and every future system will ask it questions. So it gets its own files — and the project acquires the shape it will keep until the big refactor in Module 5:

C++ in place — headers and translation units

A C++ compiler does not compile "a program". It compiles one translation unit at a time: a single .cpp file with every #include pasted in, textually, to produce one enormous file that is then turned into one object file. Our build now has two — main.cpp and input.cpp — and they are compiled in complete ignorance of each other. The linker joins the object files afterwards, matching each function call in one to the function body in another.

That is what the header is for. input.hpp contains declarations — the shape of the class and the signatures of its functions, enough for the compiler to check every call site in main.cpp. input.cpp contains the definitions — the bodies the linker will go looking for. A missing declaration is a compile error ("no member named key_down"); a missing definition is a link error ("undefined symbol engine::input::key_down"). Learning to read which one you got tells you immediately whether you forgot an include or forgot to add a file to the build.

#pragma once at the top of the header means "if this file has already been pasted into this translation unit, skip it." Without it, two headers that both include input.hpp would define class input twice in one translation unit, which is an error. Every header in this course starts with it.

4.1 The interface, decided before the implementation

Write the header first. It is the part other code has to live with, and deciding it first forces the question "what does a caller actually want?" ahead of "how do I do this?"

Three things about those three lines. They take SDL_Scancode, so §3.4's rule is enforced by the type system rather than by a comment nobody reads — pass a keycode and it does not compile. They are const, which says asking a question does not change the answer, and lets a system holding a const input& still use it. And they are [[nodiscard]], so in.key_pressed(SDL_SCANCODE_SPACE); written as a statement — a real typo, usually a half-finished if — is a warning instead of silence.

The state itself is two arrays and two words:

SDL_SCANCODE_COUNT is 512 — SDL's own "one past the last scancode" marker, so the array is exactly the right size by construction and stays right if SDL ever adds keys. The {} value-initialises every element to false: without it these are raw bools full of whatever was in that memory, and the engine would start life believing a random subset of the keyboard was held down. Two characters, one whole class of startup bug.

The mouse gets the same two-snapshot treatment, but its five buttons fit in one 32-bit word as a bitmask, so "copy the previous state" is a single assignment and the edge tests become bit operations. It is the identical idea at a different scale — which is the hint from §3.2 about packing keys into words, already paying off once.

4.3 The one event we cannot poll

feed_event handles the wheel, and only the wheel:

Two events in one frame add up, which is correct: a fast flick that produces three notches between frames should scroll three notches, not one.

The sign is the interesting part, and it is a real portability bug hiding in one line. When a user enables "natural scrolling" — the default on macOS, an option on Windows and most Linux desktops — SDL does not quietly negate the numbers for you. It reports the raw values and sets event.wheel.direction to SDL_MOUSEWHEEL_FLIPPED, documented in the header as meaning the X and Y values are opposite and should be multiplied by −1. Ignore the flag and your zoom control works on your machine and runs backwards on your colleague's, with both of you certain the other has it wrong.

4.5 The build change

A new .cpp is a new translation unit, and the build has to be told. Two changes:

Sources are listed by hand. CMake will happily glob a directory for you with file(GLOB ...), and you should not let it: a glob is evaluated when CMake configures, not when it builds, so a file added afterwards is silently absent from the build until someone reconfigures — and the symptom is a link error naming a function you are looking straight at. One line per file is a small price for a build that cannot lie to you.

target_include_directories(engine PRIVATE src) makes src/ a root for our own includes, which is why main.cpp says #include "core/input.hpp" rather than a relative path. Every file spells the include the same way regardless of where it sits, so moving a file does not break the files that include it. PRIVATE means the setting applies to building this target and is not handed on to anything that links it — correct here, since nothing links an executable. When engine becomes a library in Module 5, that keyword becomes a real design decision and gets the discussion it deserves.

If your own build behaves differently in any of these, trust the difference and go looking — that is what a checkpoint is for.

7Common Pitfalls & Debugging

Movement stutters, then lurches — one step, a long pause, then jerky motion
Symptom
Holding a direction key moves the object once, pauses for roughly half a second, then moves it in a rhythmic lurch. The rate differs between machines and between two players on identical hardware.
Cause
Movement is driven by SDL_EVENT_KEY_DOWN events. You are receiving the OS's key-repeat stream, whose delay and rate are user preferences meant for typing.
Fix
Poll the level: if (in.key_down(SDL_SCANCODE_D)), evaluated every frame. Filtering repeats with event.key.repeat is not the fix — that leaves you with exactly one movement per press.
A jump, shot, or menu item fires continuously — one tap scrolls the whole menu
Symptom
The action repeats every frame while the key is held: the character bunny-hops, the menu cursor teleports to the bottom of the list, the weapon empties its magazine on a tap.
Cause
A discrete action is reading a level. At 60 fps a 200 ms tap is twelve frames, so it fires twelve times.
Fix
key_pressed, not key_down. Apply the test from §2.2: if holding longer should not do more, it is an edge. (Genuine auto-fire is then an explicit choice with its own timer, not an accident of the query you happened to call.)
No key is ever pressed — levels work perfectly, edges never fire
Symptom
Movement is flawless, but key_pressed and key_released return false for every key, always. No crash, no warning, and both arrays look correct in the debugger.
Cause
You stored SDL's pointer instead of copying its contents (§3.3). keys_ and keys_prev_ hold the same address, so the edge test evaluates x && !x.
Fix
Copy: std::copy_n(sdl_keys, n, keys_.begin()), into a std::array you own. To confirm the diagnosis in a debugger, print the two addresses rather than the values — if they match, that is your bug in one line.
WASD does nothing on a French or German keyboard — works for you, broken for a whole country
Symptom
A player reports that movement is unusable. On AZERTY the movement keys are scattered; on QWERTZ, W and Z are swapped relative to your expectation.
Cause
Movement is keyed off SDLK_* keycodes, which follow the layout's letters rather than the keys' positions (§3.4).
Fix
SDL_SCANCODE_* everywhere position is what matters. Our API takes SDL_Scancode precisely so this mistake fails to compile. To reproduce the bug deliberately, add an AZERTY layout in your OS settings and switch to it — it takes a minute and is worth doing once.
Input feels one frame late, or a key sticks after alt-tab — mushy controls; the character runs on forever
Symptom
Everything works but feels slightly behind. Or: hold W, alt-tab away, come back, and the character is still walking.
Cause
update() is called before the event drain, so it samples state the pump has not refreshed yet (§3.5). SDL's own fix for the stuck key arrives as key-up events, so sampling before draining misses it too.
Fix
Drain, then update(), then simulate. If you are unsure which order your loop actually has, log a counter in both — the number tells you immediately.
The wheel does nothing, or scrolls backwards for some users — works on your desk, inverted on someone else's
Symptom
Either the wheel is ignored entirely, or zoom works for you and runs the wrong way for a colleague — usually one on macOS.
Cause
Ignored: you tried to poll it, or forgot to pass events to feed_event. There is no wheel state to poll (§3.6). Inverted: you ignored event.wheel.direction, which is SDL_MOUSEWHEEL_FLIPPED under natural scrolling.
Fix
Accumulate from events, and negate when the direction is flipped. Test it by toggling natural scrolling in your OS settings — a ten-second check that catches a bug you would otherwise ship.
Link error: undefined symbol: engine::input::update() — it compiles, it will not link
Symptom
Every file compiles, then the link step fails naming functions that plainly exist in input.cpp.
Cause
src/core/input.cpp was not added to add_executable, so its translation unit was never compiled and the linker has no definitions to find. The header supplied the declarations, which is why the compile step was perfectly happy.
Fix
Add the file to the source list and reconfigure. This is the compile-error-versus-link-error distinction from §4 doing useful work: a missing declaration fails at compile time, a missing definition fails at link time, and knowing which you got points straight at the cause.

8Exercises

1.2.1Feel the differencewarm-up

Change the colour cycle from in.key_pressed(SDL_SCANCODE_SPACE) to in.key_down(SDL_SCANCODE_SPACE). Rebuild, hold Space for one second, and count the colour changes. Then predict the number before you look, using your frame rate and the palette size. Put it back afterwards.

Solution

At 60 fps, holding for one second gives 60 changes through a 4-colour palette — 15 complete cycles. The box does not appear to change colour so much as vibrate, and which colour it lands on when you release is pure chance, decided by the exact frame your finger left the key. That randomness is the real tell: an action whose outcome depends on when a frame boundary happened to fall is asking the wrong question of the input system.

1.2.2Drag instead of teleportwarm-up

Currently a click teleports the box once. Make the box follow the cursor while the left button is held, then stop following when you let go. How many characters does the change take?

Solution

Change in.mouse_pressed(SDL_BUTTON_LEFT) to in.mouse_down(SDL_BUTTON_LEFT) — seven characters. "Teleport once" and "drag" are the same code differing only in which question it asks, which is a fair summary of the whole lesson. For a more polished drag, record the offset between the cursor and the box corner on the mouse_pressed frame and preserve it while dragging, so the box does not snap its centre to the cursor at the start.

1.2.3Diagonal movement is too fastguided

Hold W and D together and watch the box carefully. It moves diagonally — and faster than it moves along either axis alone. Measure it: time a traversal of the window horizontally, then time the same horizontal distance while moving diagonally. By what factor is it faster, and why? What would you do about it?

Hint

Write down the per-frame displacement in each case. Straight right is (6, 0). Up-and-right is (6, −6). How long is each of those arrows?

Solution

The diagonal step is (6, −6), whose length is √(6² + 6²) = √72 ≈ 8.49 — a factor of √2 ≈ 1.414, so diagonal movement is about 41% faster. This is the "strafe-running" bug that shipped in several famous games, and speedrunners still exploit the ones where it survives.

The fix is to make the direction a unit-length arrow before scaling it by speed: divide (dx, dy) by its own length whenever it is non-zero, so every direction contributes exactly one step's worth of movement. That operation is called normalisation, it is the first genuinely vector-shaped thing the engine needs, and it is the opening act of Lesson 1.7 — where we build vec2 properly rather than bolting a square root onto main.cpp. Leaving the bug in place for five lessons is deliberate: it is much easier to care about vectors once you have watched something misbehave for want of one.

1.2.4How long has it been held?open

Add key_held_frames(SDL_Scancode), returning the number of consecutive frames a key has been down (0 if it is up). Then use it: make the box's speed ramp up over the first half second of holding a direction, so it eases into motion instead of starting at full speed.

Hint

You need one more piece of per-key state. update() already knows, for every key, whether it is down and whether it just went down — which is exactly enough to decide between incrementing a counter and resetting it.

Solution sketch

Add std::array<Uint32, SDL_SCANCODE_COUNT> held_frames_{}; and, at the end of update(), loop over the array: held_frames_[i] = keys_[i] ? held_frames_[i] + 1 : 0;. The query returns the entry. A 512-iteration loop per frame is nothing, but it is worth noticing that it is now work proportional to the whole keyboard rather than to the keys you asked about — the first time in this engine that a convenience has a per-frame cost attached, and exactly the kind of trade Module 8 teaches you to measure rather than guess about.

For the ramp, map frames to a factor — for instance min(1.0f, held / 30.0f) at 60 fps — and multiply the speed by it. Be aware you have just written another piece of logic in units of frames rather than seconds, which will be wrong on any other refresh rate. Lesson 1.3 converts both.

1.2.5Actions, not keysopen

Right now main.cpp spells out SDL_SCANCODE_A || SDL_SCANCODE_LEFT at every movement site. Design (do not necessarily build) a layer where gameplay asks action_down(action::move_left) and a separate binding table decides which scancodes that means. What does the table look like? Where does it come from at startup? What breaks if a player binds one key to two actions?

Solution sketch

The core is an enum class action plus a map from action to a small set of bindings, with the query becoming "is any bound key down for this action?" — and the edge queries needing care, since "pressed" for an action with two bound keys should mean "no bound key was down last frame and at least one is now", not "any bound key was pressed" (otherwise pressing the second key while holding the first fires a spurious edge).

The interesting design questions are the ones the exercise asks. Bindings want to be data — loaded from a config file so players can rebind without a recompile, which drags in the serialisation work of Module 8. Analogue sources (a gamepad stick that is 0.6 to the left) need a threshold to answer a boolean question, so the layer eventually wants a value query alongside the boolean ones. And a key bound to two actions is not necessarily an error: it is fine across contexts (Esc closes a menu in one, opens it in another), which is the argument for a stack of input contexts rather than a single flat table.

We build this properly in Module 5's input-mapping lesson, once the engine has an asset system to load the bindings from. Sketching it now is worthwhile because the design will feel inevitable rather than imposed when we get there — and because you will notice main.cpp asking for it every time you type a scancode twice.

9Recap & Next

What the engine can do now that it could not before

  • It has an input subsystem. The first real one — its own header and implementation file, its own state, its own invariant, and a public interface the rest of the engine will use for the next eighty lessons.
  • It answers both input questions, from one source of truth. Levels for continuous actions, edges for discrete ones, derived from two snapshots one frame apart — so they cannot contradict each other, and asking does not consume anything.
  • It is layout-independent. Movement is addressed by physical position, so WASD is ZQSD on AZERTY with no localisation code, and the wrong choice does not compile.
  • The frame has its permanent shape: drain, sample, simulate, render — with a reason for the order, not a habit.
  • It builds from multiple translation units, and you can tell a missing declaration from a missing definition by which stage of the build complains.

The idea to carry forward: events tell you what happened; state tells you what is. Most of gameplay wants the second, most of tooling wants the first, and the honest way to serve both is to keep one snapshot and derive the rest — while knowing exactly which inputs cannot be snapshotted at all.

Next: 1.3 — Frames, Delta Time, and Why Naive Loops Lie. We now have a box that moves six pixels per frame, which means its speed is whatever the monitor feels like — a bug we deliberately left standing and can now see clearly. Fixing it means measuring time honestly, which turns out to be harder and far more interesting than it sounds: the obvious fix has its own failure mode, the fix for that has one too, and following the chain to the end is how we arrive at the loop the engine will keep for good.

10Further Reading

  • SDL3 wiki SDL_GetKeyboardState — short, and every sentence matters: the pointer is SDL's, pumping is what refreshes it, and a press-and-release between polls is invisible. The three facts §3 is built on.
  • SDL3 wiki SDL_Scancode and SDL_Keycode — the two languages side by side. Read both pages together; the distinction is much clearer as a pair than either is alone.
  • SDL3 wiki Migrating to SDL 3.0 — the keyboard-state type change from Uint8 to bool, and the flattening of the key event's keysym nesting. Worth a skim before you copy any SDL code you find in the wild.
  • Source SDL_keyboard.c — find SDL_SetKeyboardFocus and SDL_ResetKeyboard and read the twenty lines that stop keys sticking when you alt-tab. Reading the implementation of a library you depend on is a habit worth building early, and this is an unusually approachable place to start.
  • Book Jason Gregory, Game Engine Architecture — the human-interface-device chapter, especially its treatment of button state, chords, and gesture detection. It is the professional version of what we built today, and it is where Exercise 1.2.5 leads.
  • Reference Unity's GetKey / GetKeyDown / GetKeyUp — confirmation that this is not a course-specific invention. Every mainstream engine exposes exactly these three queries, under different names, for exactly the reasons in §3.1.
