Module 2 — Software Rasterizer I · Lesson 2.11

The Viewport Transform

One link remains before the milestone: turning normalized device coordinates into actual framebuffer pixels. It is the simplest transform in the whole pipeline — three independent scale-and-offset maps — with one twist that has been quietly haunting the course since Lesson 2.5: the y-flip. NDC counts +y upward; the framebuffer counts rows downward. That single sign has lived, undeciphered, inside a dozen different functions. Today it gets derived, named, and given exactly one home.

1The Problem

Lesson 2.10 got us to normalized device coordinates: after the perspective divide, every visible point has x, y ∈ [−1, 1] and z ∈ [0, 1]. That is a lovely, tidy space — and it is not pixels. It says nothing about how big the window is, where on the screen we are drawing, or which corner is the origin. NDC is deliberately device-independent: the same projection matrix works whether you render to a phone, a monitor, or a 320×180 software buffer. Something else has to nail that abstract [−1, 1] square onto a concrete grid of pixels.

We have, in fact, been doing exactly that since Lesson 2.9 — but with a shrug. Here is the line the demo has been using, buried inside project:

Three hand-picked constants and a minus sign, with a comment that gestures at what the minus is for without ever explaining it. Two things are wrong with leaving it there. First, it is underived: those constants are an affine map and that minus is a coordinate-system flip, and a course that derives the perspective divide from similar triangles should not hand-wave its very last step. Second, it is homeless: that same minus sign has appeared, in slightly different disguises, in Lesson 2.5's to_screen, in 2.6's cube, in every demo since. A fact that important, scattered across a dozen functions, is a bug waiting to happen.

The failure mode — the world upside down

Drop that minus sign — write + ndc.y * k_vp_half_h instead of − — and the entire scene renders upside down. Not tilted, not mirrored left-to-right: flipped top for bottom, the floor above your head and the sky below your feet. It is one of the most common bugs a graphics beginner hits, and the reason is exactly the thing we are about to pin down: NDC's +y points up, but a framebuffer — like every image format, like the memory it lives in — counts rows downward from the top. If nobody reconciles those two, up and down trade places.

So this lesson does something small and important: it takes that last hop, derives it properly, and moves it into one named thing — an engine::viewport — that mirrors the GPU's own viewport so Module 4 will reuse the identical numbers. The picture on screen will not change by a single pixel. What changes is that the pipeline finally has a complete, explicable last step, and the y-flip has exactly one place to live.

2Building Intuition

NDC is a unit screen with no size

Think of NDC as a standard-sized screen: always two units wide, two units tall, origin dead centre, +x right, +y up. Every renderer in the world agrees on this frame. It is like a film negative in a standard 35 mm frame: the image is composed and complete, but it has no physical size until you decide how big to print it and where to hang it.

The viewport is that decision. It is a rectangle of real pixels — a position and a size on the actual framebuffer — onto which the NDC square gets stretched. Want the render to fill a 1920×1080 window? Viewport (0, 0, 1920, 1080). Want it in the top-left quarter for a split-screen? Viewport (0, 0, 960, 540). Want a little minimap in the corner? A small viewport over there. The projection and everything before it do not change at all — only this final rectangle does. That is the whole point of NDC being device-independent: the viewport is the one place the actual pixel dimensions enter.

The negative and the enlarger

Keep the negative analogy going, because it makes the flip inevitable rather than surprising. To print the negative you put it in an enlarger, which projects it onto photographic paper. Two things happen: the image is scaled to the paper size, and — because of how the paper sits — the projected image can land flipped relative to the negative.

Our flip is exactly this, and it is forced by a disagreement about which way is up. NDC, like graph paper and like the clip space SDL_GPU hands us, has +y pointing up. A framebuffer is an array of rows in memory, and row 0 is the top of the image — so as the pixel y coordinate increases, you move down the screen. Two frames, one with +y up and one with +y down. To go between them you must reverse y. Figure 1 shows the correspondence that the viewport has to honour.

3The Theory

Three affine maps

The viewport transform is three independent maps, one per axis, and each is the simplest kind of map there is: an affine one — a scale followed by an offset, out = scale · in + offset. No division, no trig, nothing coupled between axes. Let the viewport be a rectangle with top-left corner (vx, vy), width w and height h.

Start with x, which has no flip and is the clean case. NDC x runs [−1, 1]; we want screen x to run [vx, vx + w], left edge to right edge. The standard move is to first remap [−1, 1] to [0, 1] — call that fraction t — then scale by the width and add the offset:

\[ t_x = \frac{x_{ndc} + 1}{2}, \qquad x_{screen} = v_x + t_x \cdot w \] t_x = (ndc.x + 1) / 2; screen.x = vx + t_x * w

Check the ends: x_ndc = −1 gives t_x = 0 and screen.x = vx (left edge); x_ndc = +1 gives t_x = 1 and screen.x = vx + w (right edge); x_ndc = 0 gives the centre vx + w/2. That is the entire x map, and y would be identical — if not for the flip.

The y-flip, derived and homed

Now y, and the one subtlety in the whole lesson. We form the same fraction t_y = (y_ndc + 1)/2, which is 0 at the NDC bottom (y = −1) and 1 at the NDC top (y = +1). But the framebuffer's y runs the other way: 0 at the top, h at the bottom. So the NDC top (t_y = 1) must map to screen y = vy (the top), and the NDC bottom (t_y = 0) to screen y = vy + h (the bottom). That is achieved by using (1 − t_y) where x used t_x:

\[ t_y = \frac{y_{ndc} + 1}{2}, \qquad y_{screen} = v_y + (1 - t_y)\cdot h \] t_y = (ndc.y + 1) / 2; screen.y = vy + (1 - t_y) * h

That (1 − t_y) is the flip — the whole of it, derived rather than decreed. Read the ends once more to be sure: y_ndc = +1 (top of NDC) gives t_y = 1, so y_screen = vy (top row); y_ndc = −1 (bottom) gives t_y = 0, so y_screen = vy + h (bottom row). Up maps to up. Forget the (1 − ⋯) and up maps to the bottom row — the upside-down render from §1.

One sign, one home — the lone minus, retired

The old demo wrote this as vy + h/2 − y_ndc · h/2, which is algebraically the same thing (expand (1 − (y+1)/2)·h = (1 − y)/2 · h = h/2 − y·h/2). Both are correct; the difference is that the (1 − t_y) form says what it is doing — reversing a fraction — while the bare −y_ndc just looks like a mysterious negative. This is the same minus sign that has appeared in to_screen since Lesson 2.5's basis demo, in every 3-D demo since, and in project last lesson. From now on it lives in exactly one function, viewport::to_screen, and nowhere else. A convention that has one home cannot be gotten subtly wrong in the eleventh place that needs it.

Depth through the viewport

The viewport carries a third axis, z, even though our software rasterizer does not use it yet (the z-buffer is Lesson 3.1). NDC depth is already [0, 1] from the projection matrix (Lesson 2.10 §3.4), and the viewport maps it onto a [min_depth, max_depth] range — almost always [0, 1], i.e. an identity, but occasionally narrowed:

\[ z_{screen} = \text{min\_depth} + z_{ndc}\,(\text{max\_depth} - \text{min\_depth}) \] screen.z = min_depth + ndc.z * (max_depth - min_depth)

Why expose min_depth and max_depth at all, if they are usually 0 and 1? Because the GPU does — SDL_GPUViewport has these exact fields — and because narrowing the range is a real trick: render a HUD or a first-person weapon with a viewport depth range of [0, 0.1] and it is guaranteed to sit in front of a world rendered at [0.1, 1], without a separate pass or a depth clear. We mirror the fields now so the value is ready when Lesson 3.1 gives depth somewhere to be stored.

The viewport, assembled and walked

Collect the three maps into a struct that mirrors SDL_GPUViewport field-for-field, and the whole transform is one short method. Let us walk a couple of points through a concrete viewport by hand.

⚠ Verify

SDL_GPUViewport's fields are x, y, w, h, min_depth, max_depth, with x, y the top-left offset — verified against SDL3/SDL_gpu.h (the struct's own doc comments say "left offset" and "top offset"). Our engine::viewport uses the same names and meanings, so Module 4 fills an SDL_GPUViewport from ours by copying fields.

Worked example — a full-window viewport

Take a viewport filling an 800×600 window: x = 0, y = 0, w = 800, h = 600, depth [0, 1]. Map three NDC points.

The centre, ndc = (0, 0, 0.5):

  • t_x = (0+1)/2 = 0.5, so x = 0 + 0.5·800 = 400
  • t_y = (0+1)/2 = 0.5, so y = 0 + (1−0.5)·600 = 300
  • z = 0 + 0.5·1 = 0.5

→ (400, 300, 0.5): dead centre of the window, as it must be.

The top-left of NDC, ndc = (−1, +1, 0):

  • t_x = 0 → x = 0;   t_y = 1 → y = (1−1)·600 = 0

→ (0, 0, 0): the top-left pixel. NDC's up-and-left corner is the framebuffer's first pixel — the flip working exactly as intended.

An off-centre point, ndc = (0.5, 0.25, 0.9):

  • t_x = 0.75 → x = 600
  • t_y = 0.625 → y = (1 − 0.625)·600 = 0.375·600 = 225
  • z = 0.9

→ (600, 225, 0.9): right of centre (as x > 0), and above centre (y = 225 < 300, because a positive NDC y is up, and up is a smaller screen row). The harness confirms all three.

Drag the point below through an NDC square and watch where it lands in the framebuffer. The two panels use opposite y directions on purpose — raise the point in NDC and its image climbs toward the top of the framebuffer, which is the flip made visible.

4Implementation

The engine gains one small header. It mirrors SDL_GPUViewport so Module 4 reuses the exact numbers, and its one method is the three affine maps from §3:

It is constexpr and takes ndc by value — twelve bytes, cheaper copied than referenced (Lesson 1.7) — and it returns a vec3: the pixel x and y, and the device depth. That the flip is the single token (1.0f - ty), right there with a comment naming it, is the entire architectural win of the lesson.

C++ in place — a struct that mirrors a C API

engine::viewport is a plain aggregate with the same fields, in the same order, as the C struct SDL_GPUViewport. That is deliberate: in Module 4, converting ours to SDL's is a field-by-field copy (or, since the layouts match, conceptually a reinterpret — though we will copy explicitly rather than rely on layout identity, because relying on it is a fragile bet no comment can defend). Designing our types to shadow the API we will target keeps the eventual port mechanical. It is the same instinct behind our mat4 already being column-major like HLSL constant buffers (Lesson 2.6): match the destination early, port cheaply later.

In the demo, the three loose constants become one viewport, and project ends by calling it:

Those numbers are chosen so the viewport's centre is (92, 90) and its extent is the old 172 × 96.75 — a 16:9 rectangle, matching the aspect the projection matrix bakes in (Lesson 2.10 §3.3). The result is byte-for-byte the same pixels as last lesson: the harness sweeps an NDC grid through both the old constants and the new viewport and the worst difference is 0.000e+00. This lesson is a refactor with a derivation attached, not a change to the image — which is the honest thing to say about it.

The one visible addition is in the HUD. The probe vertex, which last lesson stopped at NDC, now goes one hop further — through the viewport to screen pixels — so the on-screen readout finally shows the complete chain, and the screen line's y makes the flip concrete: a positive NDC y produces a screen y above centre.

7Common Pitfalls & Debugging

The whole scene is upside down — floor on top, sky on the bottom
Symptom
Everything renders flipped top-to-bottom. Not mirrored left-right — vertically inverted.
Cause
The y-flip is missing or doubled. Either y_screen = vy + t_y·h (no flip, so NDC's up maps to the bottom), or the flip was applied twice (once in the viewport and once elsewhere — the very scattering this lesson fixes).
Fix
Use (1 − t_y) in exactly one place — viewport::to_screen — and remove any other stray −y. Confirm with a known point: NDC (0, +1) (top-centre) must map to y = vy (top row), not vy + h.
The render is squashed or stretched — circles become ovals
Symptom
The scene is compressed horizontally or vertically; a world sphere is an ellipse.
Cause
The viewport's w : h ratio does not match the aspect the projection matrix was built with. The projection pre-corrects x for a specific aspect; if the pixel rectangle has a different shape, the correction is wrong.
Fix
Keep them in lockstep: viewport.w / viewport.h == aspect. The demo's viewport is deliberately 172 : 96.75 = 16:9, matching the projection's aspect. If you resize the viewport, rebuild the projection with the new aspect (or vice versa).
A thin seam or gap at the viewport edge — one row/column off
Symptom
The rendered rectangle is a pixel short of, or a pixel over, where you expected its edges; a split-screen has a one-pixel seam or overlap down the middle.
Cause
Pixel-centre versus pixel-edge conventions, and integer truncation. NDC ±1 maps to the viewport's edges (vx and vx + w), which are the boundaries between pixels, not pixel centres. Rounding a float screen coordinate to an int without deciding a convention gives off-by-one edges.
Fix
Be deliberate about the float-to-int step. Our rasterizer (Lesson 2.2) samples pixel centres at (x + 0.5, y + 0.5), so screen coordinates stay in floats until the edge functions consume them, and the fill rule (2.2's top-left rule) settles shared edges. For a split-screen, give each viewport a width that leaves no gap: w1 + w2 == framebuffer_width, exactly.
Depth comes out reversed or clipped — near/far confusion at the viewport
Symptom
Once a z-buffer exists (Lesson 3.1), depth testing keeps the wrong surfaces, or a HUD you tried to pin in front ends up behind.
Cause
min_depth and max_depth set backwards (e.g. 1 and 0), or a viewport depth range that overlaps another it was meant to sit in front of.
Fix
For normal rendering leave them [0, 1]. For the pin-in-front trick, give the front layer a strictly smaller range ([0, 0.1]) than the world ([0.1, 1]) and keep the depth test direction consistent with both. Reversing min and max flips the depth sense — deliberate for reversed-Z (Lesson 2.10, Exercise 2.10.5), a bug otherwise.
The GPU viewport doesn't match the software one — Module-4 image differs
Symptom
Looking ahead: when the same scene is later rendered on the GPU (Module 4), it lands in a different place or orientation than the software render.
Cause
A mismatch between our viewport and the SDL_GPUViewport handed to the GPU — usually the y convention (whether the API expects top-left or bottom-left origin) or a field copied to the wrong slot.
Fix
Fill SDL_GPUViewport from our fields by name, and pin the GPU's NDC/viewport conventions on the Conventions page before relying on them (§4–5 there, and the SDL wiki). We designed engine::viewport to mirror the C struct precisely so this stays a copy, not a translation — but "mirrors" is a claim to verify against the header, which is why it carries a ⚠ VERIFY above.

8Exercises

2.11.1Map a point by handwarm-up

A viewport occupies the bottom-right quarter of a 1280×720 window: x = 640, y = 360, w = 640, h = 360, depth [0, 1]. Where does ndc = (−0.5, 0.5, 0.2) land in framebuffer pixels?

Hint

Form t_x = (ndc.x + 1)/2 and t_y = (ndc.y + 1)/2, then screen.x = x + t_x·w and screen.y = y + (1 − t_y)·h. Do not forget the offset — this viewport does not start at the origin.

Solution

t_x = 0.25 → screen.x = 640 + 0.25·640 = 800. t_y = 0.75 → screen.y = 360 + (1 − 0.75)·360 = 360 + 90 = 450. screen.z = 0.2. So (800, 450, 0.2) — inside the bottom-right quarter (x ∈ [640, 1280], y ∈ [360, 720]), left-of-and-above that sub-rectangle's centre, as the negative ndc.x and positive ndc.y demand.

2.11.2Split-screenguided

Render the scene twice, side by side, from two different cameras — the classic two-player split-screen. With a viewport type this is nearly free: two viewports, two view matrices, one draw loop. Give the left player (0, 0, 160, 180) and the right (160, 0, 160, 180) in the 320×180 framebuffer.

Hint

Make project take the viewport as a parameter (as it already takes the projection), then call the draw code twice with the two viewports and two look_ats. Watch the aspect: a 160×180 viewport is taller than wide, so each half needs a projection built with aspect = 160/180, not 16:9, or the split halves will stretch.

Solution sketch

The core is a loop over {left, right} pairs of (viewport, view), drawing the whole scene each time — nothing about the model, world, or projection stages changes. The lesson is that the viewport is the only thing that localises a render to a region, so multi-view rendering (split-screen, mirrors, minimaps, shadow-map atlases) is all the same trick: same scene, different viewport. The aspect gotcha is the one real subtlety, and it is Pitfall 2 arriving on cue.

2.11.3A corner minimapmedium

Add a small top-down minimap in a corner: a second viewport (say (250, 6, 64, 64)) showing the scene from a camera high above, looking straight down. Draw the main view first, then the minimap over it.

Hint

The minimap is just another (viewport, view, projection) triple drawn on top. A top-down camera looks straight down, which is the look-at singularity from Lesson 2.9 — so tilt it a hair off vertical, or supply an up hint of (0, 0, −1) instead of (0, 1, 0) so it is not parallel to the view direction. An orthographic projection often reads better than perspective for a map.

Solution sketch

Two things make this instructive beyond "another viewport". First, it forces you to confront the 2.9 singularity in a place you actually want to look straight down — and the fix (a non-vertical up hint) is the real technique. Second, drawing the minimap after the main view, with its own small viewport, is exactly how HUD and picture-in-picture rendering work: later draws with tighter viewports layer on top. Without a depth buffer yet, draw order is all you have, which previews why Lesson 3.1's z-buffer matters for the world but not for flat overlays.

2.11.4Pin a gizmo in front with depth rangemedium

Once Lesson 3.1 gives you a z-buffer, a gizmo (say the world-axis triad) can be made to draw in front of everything by rendering it with a viewport whose depth range is [0, 0.1] while the world uses [0.1, 1]. Predict, using §3.4's formula, what device depth a far-plane point (ndc.z = 1) gets under each range, and explain why the gizmo always wins the depth test.

Hint

Plug ndc.z = 1 into min_depth + ndc.z·(max_depth − min_depth) for each range. Then plug ndc.z = 0 (near plane) too. Compare the two intervals the two renders occupy on the [0, 1] depth axis.

Solution sketch

The gizmo's depths span [0, 0.1]; the world's span [0.1, 1]. Even the gizmo's farthest point (device depth 0.1) is at least as near as the world's nearest (0.1), so with a "keep the smaller depth" test the gizmo never loses. This is a genuine production trick (weapon viewmodels, editor gizmos) that costs nothing but two viewport floats — and it only works because the viewport's depth range is a real, separate knob from the projection's near/far, which is why SDL_GPUViewport exposes it and we mirrored it.

2.11.5Where exactly is a pixel?open

The viewport maps NDC ±1 to the viewport's edges. But a pixel is a little square with a centre. Work out the relationship: for a viewport of width w pixels, what NDC x lands exactly on the centre of pixel column 0? Of the last column? Then explain how this interacts with the rasterizer's pixel-centre sampling (Lesson 2.2) and its top-left fill rule — and why getting it subtly wrong causes half-pixel shimmer along edges.

Solution sketch

Pixel column i's centre is at screen x = vx + i + 0.5, so it corresponds to t_x = (i + 0.5)/w and thus ndc.x = 2t_x − 1 = (2i + 1)/w − 1. Column 0's centre is at ndc.x = 1/w − 1 (a touch inside the left edge, not on it), and the last column's centre at 1 − 1/w. The half-pixel offset is the whole story: the rasterizer samples centres (x + 0.5, y + 0.5), so as long as the viewport maps edges to edges and the sampler adds the half, everything lines up. Drop the half somewhere — sample corners instead of centres, or map NDC to centres instead of edges — and geometry shifts by half a pixel, which reads as shimmer on thin edges and seams between adjacent viewports. This is the software-rasterizer face of the same "pixel centre" convention every GPU pins down, and it is why Lesson 2.2 was so careful about the + 0.5.

9Recap & Next

What the engine can do now that it could not before

  • Turn NDC into pixels, properly. There is an engine::viewport — a rectangle of the framebuffer and a depth range — whose to_screen is the pipeline's final transform. The chain is complete: model → world → view → clip → NDC → screen.
  • Explain the y-flip. NDC's +y is up, the framebuffer's is down, and the reversal is the single (1 − t_y) in one function — the lone minus sign that has drifted through the course since Lesson 2.5, finally derived and homed.
  • Localise a render to a region. Split-screen, minimaps, mirrors, HUD layers — all the same trick: same scene, different viewport. And the depth range is a real knob, ready for the z-buffer.
  • Port cheaply later. The viewport mirrors SDL_GPUViewport field-for-field, so Module 4 configures the GPU with the same numbers.

Next: 2.12 — Milestone: A Spinning Wireframe Mesh. Every transform in the pipeline now exists and is understood. Lesson 2.12 spends none of its length on new theory — it spends all of it lighting the whole chain up end to end: a real mesh, more than a cube, spinning in perspective through model → world → view → clip → NDC → screen, with a camera you fly and not one line you cannot explain. It is the payoff of the entire software-rasterizer-geometry arc, and the last thing before Module 3 fills the triangles in — with a z-buffer, clipping, culling, textures, and light.
