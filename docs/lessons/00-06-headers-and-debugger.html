Reading Headers & the Debugger

Two skills you will use in nearly every lesson from here on: reading a library's headers as the one true answer to "what does this really do?", and living inside a debugger so that a misbehaving program becomes a thing you inspect, not a thing you guess at.

1The Problem

Over the last two lessons, two words kept recurring: verify and bug. We said "verify this signature against the header" more than once, and 0.5 ended with a pitfall about a window that flashes and closes. Both point at skills we have used without naming, and this lesson — the last of Module 0 — names them, because you will lean on them constantly.

The first is knowing what an API actually is. You have already seen that a plausible snippet from the internet is often SDL2 and will not compile, and that a wiki page can lag the real code. There is exactly one authority on what a function does: its declaration in the header. Reading that fluently is a skill, not a chore.

The second is knowing what your running program actually does — which is rarely what you think it does when there is a bug. Here is the failure mode this lesson exists to kill:

The failure mode: debugging by voodoo

The window flashes and closes. You do not know why, so you start changing things: move a line, add a SDL_Delay, reorder the loop, comment something out, rebuild, try again. Eventually it works — or seems to — and you move on without ever knowing what was wrong.

This is debugging by voodoo, and it is slow, it teaches you nothing, and worst of all it often "fixes" the symptom by coincidence while the real bug hides, waiting to reappear. The alternative is not cleverness — it is observation. A debugger lets you stop the program mid-flight and simply look at what is true: which line is running, what each variable holds, how you got here. Bugs do not survive being looked at directly. The whole skill is learning to look.

2Building Intuition

2.1 Two questions, two sources of truth

Almost every moment of confusion in systems programming is one of two questions, and each has a single authoritative answer:

  • "What is this API supposed to do?" — answered by the header. The header is the contract: the exact types, the exact parameters, the exact return. Not the tutorial, not your memory, not the wiki (which is prose about the contract, and can drift). When they disagree, the header wins, because the header is what the compiler reads.
  • "What is my program actually doing?" — answered by the debugger. The debugger is an X-ray: it shows the live state of the running program, which is the ground truth that your mental model is only a guess at. When your model and the debugger disagree, your model is wrong — that gap is the bug.

Beginners answer both questions by guessing and rebuilding. Professionals answer them by reading — the header for the contract, the debugger for the behaviour. The rebuild-and-pray loop is not a faster version of this; it is a different, worse activity that happens to involve the same keyboard.

2.2 Three tools of visibility

"Look at what the program is doing" is not one tool but three, and they trade power against convenience. You will use all three, often on the same bug, and knowing which to reach for is half the skill. Figure 1 is the ladder.

3The Theory

3.1 Reading a header

There are three good ways to find a symbol's real declaration, in rough order of speed:

  1. Grep the fetched source. FetchContent already put SDL's headers under build/_deps/sdl3-src/include/, so they are on your disk. Grepping them is instant and always matches the exact version you build against.
  2. Grep the source on GitHub at your pinned tag, with curl. Useful before you have a build, or to check a different version.
  3. Read the wiki. The SDL3 wiki is the prose companion — good for understanding why and seeing examples, but it is generated from the headers and can lag them, so the header is the tiebreaker.

Either way, you get back a line that looks noisier than it is. Here is SDL_Init's declaration, exactly as it appears in the header — and how to read past the macros to the signature underneath:

Worked example — decoding a C declaration

The header says:

extern SDL_DECLSPEC bool SDLCALL SDL_Init(SDL_InitFlags flags);

Strip the two macros that are pure platform plumbing and carry no meaning for you:

  • extern — "defined elsewhere" (in the SDL library). Every function declaration in a header is extern; ignore it.
  • SDL_DECLSPEC — an export/visibility macro (it becomes __declspec(dllexport) on Windows, nothing elsewhere). It controls how the symbol crosses a DLL boundary. Ignore it.
  • SDLCALL — the calling convention (which registers arguments go in). It matters to the linker, never to you. Ignore it.

What is left is the signature you actually care about:

bool SDL_Init(SDL_InitFlags flags);

Returns bool — so if (!SDL_Init(...)) is the right test — and takes one SDL_InitFlags. That is the whole contract. Once you learn to mentally delete SDL_DECLSPEC and SDLCALL on sight, every SDL declaration reads as cleanly as a textbook signature. (Other libraries have their own versions of these two macros — GLAPI, VKAPI_CALL, and so on — and the same "delete the plumbing" move works on all of them.)

3.2 Instrumenting with logs

The middle rung of Figure 1 is logging, and its virtue is reach: a log line runs on every machine your program runs on, including the ones you will never touch — a player's PC, a CI runner, a server. The debugger cannot go there; a log line already is there.

We use SDL_Log rather than printf for one concrete reason: on some platforms (Windows GUI apps, Android) plain stdout goes nowhere visible, and SDL_Log routes to wherever that platform actually shows logs. It is printf-style — a format string and arguments — so it costs nothing to learn.

The single most valuable log line in any program is the one that prints, at startup, what you are running: versions, build configuration, key paths. When a bug report says "it doesn't work", the first thing you want is to know which build and which library versions the reporter had — and a startup line hands it to you for free. That is exactly the line we add to the engine this lesson.

Logging is not shameful, and it is not a substitute

There is a snobbery that says "real programmers use the debugger, only beginners printf". Ignore it. Logging and debugging answer different questions: logging is for always-on, everywhere facts and for bugs you cannot reproduce locally; the debugger is for interactive, deep investigation of a bug you can reproduce. The best engineers use both fluently and never sneer at either. What is a mistake is leaving noisy debug prints scattered through the code after the bug is fixed — a startup line earns its place; printf("here 5") does not.

3.3 The debugger's vocabulary

A debugger runs your program under its control, able to pause it (a breakpoint), resume it (continue), advance it one line at a time (step), and read its state (inspect) at any pause. The three big debuggers — lldb (macOS/Clang), gdb (Linux/GCC), and Visual Studio's (Windows/MSVC) — differ only in spelling. Learn the concepts once; the table translates them.

The distinction between step over and step into is the one beginners miss: over runs a called function to completion and stops on the next line here; into descends inside that function to debug it line by line. Step over SDL_CreateWindow — you do not want to debug SDL's insides; step into your own helper when the bug might live there. And the backtrace — the chain of who-called-whom that led to the current line — is how you answer "how did I even get here?", the question behind most crashes.

Do not forget to revert

That bool running = false; was a bug we introduced to hunt. Put it back to true — the committed engine has the correct line plus the new startup log, and nothing else. (This is also a nudge toward version control: git diff would show you exactly the one-character bug, and git stash or git checkout would undo an experiment instantly. 7Common Pitfalls & Debugging

Breakpoints do nothing / "no debug symbols" — the debugger can't stop
Symptom
Breakpoints are greyed out, ignored, or the debugger says it has no line information.
Cause
You built without debug symbols. A default or Release build strips the mapping from machine code back to your source lines, so the debugger has nothing to attach a breakpoint to.
Fix
Configure a Debug build: cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug (single-config generators) or build with cmake --build build --config Debug (Visual Studio). Debug adds -g and turns optimization off, which is what makes stepping sane.
A variable shows <optimized out> — can't inspect a value
Symptom
Inspecting a variable in the debugger reports <optimized out> or a nonsense value, and stepping jumps around unpredictably.
Cause
You are debugging an optimized build. The optimizer reorders and eliminates code, so "this line" and "this variable" may not exist as written. This is the "works in Debug, weird in Release" family.
Fix
Debug the Debug build. If a bug only appears in Release, it is usually undefined behaviour that the optimizer exposed — a topic we take seriously with sanitizers later. For now, keep the debugger on the un-optimized build.
The header you read doesn't match your build — the signature is subtly different
Symptom
A function's signature in the docs or a header you found online does not match what compiles.
Cause
You read a different SDL version's header than you build against — the wiki's latest, or main instead of your pinned release-3.4.12. APIs evolve between versions.
Fix
Read the header at your pinned tag — grep build/_deps/sdl3-src/include/, which is exactly what you build against, or use the release-3.4.12 URL. The version on disk is the one that counts.
Drowning in SDL_DECLSPEC and SDLCALL — the declaration looks alien
Symptom
Header declarations look impenetrable, full of upper-case words you do not recognise.
Cause
Export and calling-convention macros (SDL_DECLSPEC, SDLCALL, and their equivalents in every C library) sit between the return type and the name, and they carry no meaning for a caller.
Fix
Mentally delete them, as in §3.1. What remains — return type, name, parameters — is the whole contract. This is a reading habit that becomes automatic within a week.
Print-debugging that never leaves — the code is littered with logs
Symptom
Weeks later the source is full of SDL_Log("here"), SDL_Log("5"), and commented-out prints.
Cause
Print-debugging left in place. Logging is a fine tool, but temporary probes are temporary; leaving them turns real logs into noise and the code into a mess.
Fix
Delete probe prints once the bug is found — git diff makes them easy to spot. Keep only logs that earn their place, like the startup version line. When you find yourself adding and removing prints repeatedly, that is the signal to switch to the debugger instead.

8Exercises

0.6.1Decode a declarationwarm-up

Grep the SDL headers for SDL_CreateRenderer and read its full declaration. Strip the macros and write out the plain signature: what does it return, and what are its parameters? Then do the same for SDL_GetError.

Solution

SDL_Renderer *SDL_CreateRenderer(SDL_Window *window, const char *name) — returns a pointer to a renderer (null on failure), taking the window to attach to and a backend name (nullptr for the default). const char *SDL_GetError(void) — returns a read-only C string, takes nothing. The const on the return is the library telling you: do not modify or free this string, it is SDL's. Reading that const correctly is exactly the fluency this lesson builds.

0.6.2A conditional breakpointguided

With the engine running under the debugger, set a breakpoint on the key-handling line that fires only when a key-down event arrives — not on every event. Then press a few keys and confirm it stops only for key presses. Why is a conditional breakpoint better here than a plain one?

Hint

In lldb: breakpoint set --file main.cpp --line <line> --condition 'event.type == SDL_EVENT_KEY_DOWN'. In gdb: break main.cpp:<line> if event.type == SDL_EVENT_KEY_DOWN.

Solution sketch

A plain breakpoint inside the event loop would stop on every event — mouse motion fires dozens per second — and you would spend all your time hitting "continue". The condition makes the debugger do the filtering, stopping only when the event is the one you care about. Conditional breakpoints are how you find a needle in a busy loop without drowning, and the loop of an engine is the busiest place there is.

0.6.3Read the call stackguided

Set a breakpoint inside the event loop, run, and when it stops, print the backtrace (bt). What does it show, and what would it show if the breakpoint were instead inside a helper function called from the loop?

Solution sketch

Here the backtrace is short — essentially main at the current line, sitting on top of SDL's platform entry-point frames from SDL_main.h (a nice glimpse of the machinery from 0.5). If you were stopped inside a helper, the backtrace would show the whole chain: helper ← its caller ← … ← main. That chain is how you answer "how did I get here?" — indispensable when a function is called from many places and only one of them misbehaves. It is the first tool you reach for on a crash.

0.6.4Find it in the header, then confirm it liveopen

Exercise 0.5.2 asked you to log a window's new size on resize, using event.window.data1/data2. Now do it properly: first grep the header to confirm those field names and types for your pinned SDL, then add the log, then set a breakpoint on it and inspect event.window in the debugger as you drag the window. Three tools, one small feature.

Solution sketch

Grepping build/_deps/sdl3-src/include/SDL3/SDL_events.h for SDL_WindowEvent shows the struct, with Sint32 data1 and data2 holding the new width and height for a resize. The log confirms it at runtime; the debugger lets you pause on a single resize and inspect the whole event.window struct — windowID, type, and the data fields — in one place. This is the loop you will run thousands of times over the course: header for the contract, log for the always-on signal, debugger for the deep look. You now have all three.

9Recap & Next

What you can do now — and Module 0 is complete

  • You read headers as the source of truth. You can find any SDL declaration (grep the fetched source, or curl the pinned tag) and strip the macro plumbing to the real signature.
  • You have three tools of visibility — compiler, logging, debugger — and know which to reach for.
  • You hunt bugs by observation. Symptom → hypothesis → test with a breakpoint → inspect → root cause. No more voodoo.
  • The engine has a voice — a startup log line — and you know why that one line matters.

The idea to carry forward: when you are unsure, look — don't guess. The header tells you what the API is; the debugger tells you what your program does. Between them, almost nothing stays mysterious for long.

That completes Module 0. You have a verified cross-platform toolchain, a portable CMake build that fetches SDL3, a window with a clean event loop, and the two investigative skills to debug all of it. The scaffolding is done — everything from here builds the engine proper.

10Further Reading

  • Reference The LLDB Tutorial and the GDB manual — the authoritative command references. You need only a dozen commands; these have the rest for when you do.
  • Reference Visual Studio debugger docs — if you are on Windows, its graphical debugger is genuinely excellent; this is the guided tour.
  • Tool Compiler Explorer (godbolt) — paste a snippet and see the assembly and the errors from many compilers at once. The fastest way to answer "does this even compile, and what does it become?"
  • SDL3 wiki SDL_Log and the logging category — log priorities and categories, for when one startup line grows into a real logging strategy (which we build in Module 5).
  • Essay “Rubber duck debugging” and the observation-first mindset — the discipline of explaining the bug aloud, step by step, until the wrong assumption reveals itself. The debugger is the tool; this is the habit.
