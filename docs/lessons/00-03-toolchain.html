1The Problem

You cannot build an engine without a compiler, and installing one is not the hard part — installing one correctly is. A misconfigured toolchain does not announce itself. It compiles simple programs fine, lulls you into trust, and then fails three weeks from now on a C++20 feature, with an error message that points at your code rather than at the real culprit, which is a flag you never set.

Here is the failure mode, concretely. This is our hello.cpp — you will write it in a few pages — compiled with the wrong language standard:

Two things are worth noticing about that error, because they are the whole reason this lesson exists.

First, our static_assert fired on purpose — "This engine requires C++20" — and it fired at compile time, the moment the mistake was made, not at runtime after you had built a launcher around a broken binary. That is a design pattern you will use everywhere: make misconfiguration fail loudly and early, close to its cause. We build that check into the very first file, deliberately.

Second, the other error blames your code — "unknown type name consteval". The code is fine; the compiler is simply not being told to understand C++20. A beginner reads that message and starts editing hello.cpp; an engineer reads it and checks the flags. This lesson is how you become the second kind. (The exact wording and ordering of these two errors vary by compiler — GCC and MSVC phrase them differently — but the two root-cause signals are the same everywhere.)

The promise from 0.2, coming due

Lessons 0.1 and 0.2 shipped no code, and I said 0.3 would. This is the lesson where you get a compiler and produce a running program — proof that the toolchain works before we ask it to build anything that matters. From here on, every lesson ships files.

2Building Intuition

2.1 A "compiler" is four programs

When you say "compile", you picture one program turning source into an executable. That picture is wrong in a way that matters, because almost every confusing build error you will ever hit lives in the gap between two of the stages the single word hides.

A build is a small assembly line. Source text goes in one end; a runnable program comes out the other; and in between, four distinct tools each do one job and hand the result to the next. Figure 1 is the line.

Why does the seam matter? Because the two stages either side of it fail in completely different languages:

  • A compile error ("unknown type name consteval") comes from the compiler reading one file. It points at a line and a column. It is about syntax and types.
  • A link error ("undefined reference to foo()") comes from the linker trying to connect all the pieces. It points at no line at all — just a name it could not find. It is about definitions across files.

Beginners treat these as the same kind of problem and flail. Knowing which stage spoke to you tells you where to look before you have read a word of the message. That single distinction saves more debugging time than any tool.

2.2 The translation unit

The preprocessor's output has a name worth knowing, because the whole rest of C++'s build model hangs off it: the translation unit.

The preprocessor is not clever. It does textual substitution: every #include is replaced by the entire contents of the named file, every #define is expanded, every #if is resolved. The result — your file with all its includes pasted bodily in, recursively — is one translation unit, and it is what the compiler actually sees. Your 30-line hello.cpp, after #include <cstdio> and #include <version> expand, becomes thousands of lines. The compiler compiles that.

Right now we have one source file, so one translation unit, so one object file. That feels trivial. But the engine will grow to hundreds of files, and the moment there are two, the linker's job — stitching separately-compiled units into one program — becomes the source of a whole category of errors. Figure 2 shows where we are heading, so the single-file case reads as the simplest instance of a general picture rather than as the whole story.

C++ in place — separate compilation

This "compile each file alone, link at the end" model is separate compilation, and it is why C++ has headers at all. A file being compiled in isolation cannot see the others, so it needs declarations — promises that a function or type exists elsewhere — to compile against. Headers carry those promises; the .cpp files carry the definitions; the linker matches them up. When it cannot find a definition, that is the "undefined reference" link error from §2.1. The style guide's include rules exist to keep this machinery honest, and we lean on it hard from Module 2 on.

3The Theory

3.1 The four stages, in detail

You can watch each stage happen. Most of the time you run all four at once, but the compiler will stop after any stage you name, and seeing the intermediate output makes the abstraction concrete. Here is hello.cpp taken apart one stage at a time (the exact flags are Clang/GCC; the idea is identical on MSVC):

StageConsumesProducesStop after it withFails when…
Preprocess hello.cpp + all #included files one translation unit (pure C++, no directives left) -E a header is missing — fatal error: 'foo.h' file not found
Compile the translation unit assembly text (.s) -S a syntax or type error — points at a line and column
Assemble assembly text an object file (.o / .obj) — machine code with holes -c almost never; you rarely see this stage fail
Link all object files + libraries the executable (the default — run all four) a name has no definition — undefined reference to …, no line number

"Machine code with holes" is the key phrase for the object file. When hello.o is assembled, it contains the machine code for main, but every call to printf is a hole — a note that says "a function named printf goes here, fill it in later". The linker fills the holes by finding printf in the C runtime library. That is, precisely, what linking is: hole-filling across files. An "undefined reference" is a hole the linker could not fill.

3.2 Predefined macros: how a program knows who built it

Every compiler silently #defines a set of macros before it reads your first line, describing itself and its configuration. Our hello.cpp reads a few of them to report its own build. This is not a party trick — it is how real cross-platform code adapts to its compiler, and the engine will use exactly this mechanism to paper over platform differences.

MacroDefined byTells you
__clang__Clang (incl. Apple Clang)the compiler is Clang
__GNUC__GCC — and Clang, for compatibility"GCC-compatible". Not "is GCC".
_MSC_VERMSVC onlythe compiler is Microsoft's, and its version
__cplusplusevery conforming compilerthe C++ standard in use — except on MSVC, see §3.3
_MSVC_LANGMSVC onlythe C++ standard MSVC is really using
Clang also defines __GNUC__ — order matters

Because Clang defines __GNUC__ for compatibility with GCC-targeting code, a naïve #if defined(__GNUC__) matches Clang too. That is why hello.cpp checks __clang__ first: the most specific test wins. Get the order backwards and every Clang build proudly reports itself as GCC. A small bug, but a representative one — predefined macros are full of these overlaps.

Worked example — decoding _MSC_VER

MSVC's version macro is an integer like 1939. It reads as an implied major.minor of the compiler toolset: split off the last two digits.

  • 1939 → major 19, minor 39 → compiler version 19.39.
  • The bands map to Visual Studio releases: 1900 = VS 2015, 1910–1916 = VS 2017, 1920–1929 = VS 2019, 1930–1939… = VS 2022.
  • So 1939 is a Visual Studio 2022 toolset.

⚠ VERIFY: the exact minor-to-VS-update mapping (e.g. which update ships 1939 precisely) against Microsoft's predefined-macros documentation if you need it exactly — the bands above are stable and sufficient for our purposes, but the precise patch mapping changes with each update.

3.3 What -std=c++20 actually selects, and the MSVC trap

The __cplusplus macro is an integer naming the standard, encoded as the year and month the standard was finalised. It is worth decoding once, because then the number is never mysterious again:

Worked example — decoding __cplusplus
  • 201103L → 2011-03 → C++11 (March 2011).
  • 201402L → 2014-02 → C++14.
  • 201703L → 2017-03 → C++17.
  • 202002L → 2020-02 → C++20. This is the value we require.

The trailing L just marks it a long literal. Our static_assert(ENG_CPP_STANDARD >= 202002L) reads directly as "the standard is at least February 2020" — which is exactly "C++20 or newer".

Now the trap, and it is a genuinely nasty one that costs people hours. MSVC, by default, reports __cplusplus == 199711L — the value for C++98 — no matter what you pass to /std. It does this for backward compatibility with old code that sniffed the macro. So a correctly-configured C++20 MSVC build will, by default, lie about its standard through the one macro everyone checks.

There are two independent fixes, and you want both:

  • Pass /Zc:__cplusplus, which tells MSVC to report the honest value. Do this because libraries you do not control — including SDL — check __cplusplus, and they will misbehave if it lies.
  • In your own code, read _MSVC_LANG instead, which always reflects the real /std value. That is exactly what hello.cpp's ENG_CPP_STANDARD macro does, so our static_assert is honest even if someone forgets the flag.

We do both, and hello.cpp prints both numbers so you can see the discrepancy with your own eyes. On MSVC without /Zc:__cplusplus, you would see standard: 202002 (from _MSVC_LANG, honest) next to __cplusplus: 199711 (the lie). Seeing that gap once fixes the concept permanently.

4Implementation: install, then compile by hand

Find your platform below, install, and verify from a terminal before writing any code. We deliberately use no build system yet — just the raw compiler — because 0.4 introduces CMake, and CMake is far easier to understand once you have felt what it is automating. You are about to do by hand the thing CMake will do for you.

You need one compiler, not all three

Pick the one native to your OS: MSVC on Windows, GCC or Clang on Linux, Clang on macOS. The course supports all three because the engine must build everywhere, but you only need your own. Later lessons give commands for each; today, just get one working.

4.1 Windows — MSVC

Install Visual Studio 2022 Community (free). In the installer, check the "Desktop development with C++" workload — this single checkbox pulls in the MSVC compiler (cl.exe), the Windows SDK, a debugger, and even a copy of CMake. If you want no IDE, "Build Tools for Visual Studio 2022" installs the same compiler alone.

The catch that stops most beginners: cl.exe is not on your PATH in a normal terminal, and it needs a batch of environment variables set to find its headers and libraries. Microsoft solves this with a preconfigured prompt. From the Start menu, open:

4.3 macOS — Clang

macOS uses Apple's build of Clang, and you do not need the full Xcode app — just the command line tools:

Apple Clang's version number is not upstream Clang's

Apple ships its own version numbering. "Apple clang version 15" is not LLVM Clang 15 — it is based on some nearby upstream release, but the numbers do not line up, and __clang_major__ reports Apple's number. In practice a current Xcode's Clang fully supports the C++20 we use, so this rarely bites — but do not compare an Apple Clang version against a "requires Clang ≥ N" note as though they were the same scale.

6Build & Run

One command per platform. Note the flags — each earns its place, listed below.

FlagDoes what
-std=c++20 / /std:c++20Selects the C++20 standard. Without it, most compilers default to an older one and consteval fails.

7Common Pitfalls & Debugging

MSVC reports C++98 with /std:c++20 set — __cplusplus says 199711
Symptom
You passed /std:c++20, but code that checks __cplusplus >= 202002L — yours or a library's — behaves as though it is C++98.
Cause
MSVC freezes __cplusplus at 199711L by default, for backward compatibility with old macro-sniffing code. The /std flag genuinely works; only the reporting macro is stuck.
Fix
Add /Zc:__cplusplus to the compile command. In your own code, prefer _MSVC_LANG, which never lies — which is exactly why hello.cpp uses it. You need both: the flag for libraries, the macro for yourself.
cl or g++ is "not recognised" — command not found
Symptom
The compiler is installed, but the terminal says the command does not exist.
Cause
The compiler's folder is not on your PATH. On Windows this is the norm for cl.exe — it is deliberately only available inside the Native Tools prompt, which sets up the environment. On Linux/macOS it usually means the install did not finish, or a new shell has not picked up the change.
Fix
Windows: open the x64 Native Tools Command Prompt for VS 2022 from the Start menu rather than a plain terminal — do not try to fix PATH by hand. Linux/macOS: re-run the install, then open a fresh terminal.
consteval is an "unknown type name" — error at the consteval line
Symptom
The exact error from §1: the compiler does not recognise consteval, and the static_assert fires.
Cause
You are not compiling as C++20. Either the -std//std flag is missing, or your compiler is too old to support it (GCC < 10, Clang < 10 roughly — well below our GCC 12 / Clang 15 floor).
Fix
Add the standard flag. If it is already there, check your compiler version against §4 — an old distro can ship a compiler years out of date. Install a newer one.
Every Clang build says "GCC" — the compiler line is wrong
Symptom
You are on Clang (or Apple Clang), but hello prints compiler : GCC ….
Cause
Clang defines __GNUC__ for GCC compatibility, so a #if that tests __GNUC__ before __clang__ matches Clang and mislabels it.
Fix
Check __clang__ first, as the listing does. The general rule: order predefined-macro checks most-specific first. (If you hit this, you edited the order — put it back.)
Windows: 32-bit build, or a linker complaining about architecture — weird link errors
Symptom
Things build but link strangely, or later SDL binaries refuse to link with "module machine type conflicts".
Cause
You opened the x86 or plain "Developer" command prompt, which targets 32-bit. This course is 64-bit only, and mixing 32- and 64-bit objects/libraries fails at link.
Fix
Always use the x64 Native Tools prompt. If unsure, run cl with no arguments — the banner ends in for x64 when you are right.

8Exercises

0.3.1Break it on purposewarm-up

Compile hello.cpp with an old standard on purpose — -std=c++14 (or /std:c++14) — and read the error. Then fix it. The goal is to recognise this failure instantly when it happens for real, three weeks from now, on a machine you did not configure.

What you should see, and why

Two errors: consteval is unknown (it is a C++20 keyword the older standard does not know), and the static_assert fires with our message. Both point at the same root cause — the wrong standard — from two angles. Restoring -std=c++20 clears both at once, which is the tell that they were one problem, not two.

0.3.2Watch the stages happenguided

On GCC or Clang, run the build stopping after each stage, and look at what each produces:

  1. clang++ -std=c++20 -E hello.cpp | wc -l — how many lines is the translation unit?
  2. clang++ -std=c++20 -S hello.cpp — open hello.s. That is the assembly.
  3. clang++ -std=c++20 -c hello.cpp — produces hello.o, the object file with holes.

Why is the preprocessed output so much larger than your 30-line file?

Solution

The translation unit is typically a few hundred to a couple of thousand lines, even though you wrote about thirty. The reason is §2.2: #include <cstdio> and #include <version> are replaced by the entire text of those headers — and their includes, recursively. The compiler never sees your 30 lines; it sees the whole expanded unit. This is also why compile times balloon as a project grows: every .cpp re-expands and re-compiles all the headers it pulls in, which is a performance problem we take seriously much later.

0.3.3Provoke a link errorguided

Add this line above main, then build normally: 

int missing_function();  // declared, never defined

…and call it from main with missing_function();. Predict which stage fails before you build. Were you right?

Solution

It compiles cleanly — the declaration is a promise the compiler happily accepts — and then fails at link with "undefined reference to missing_function()" (Clang/GCC) or "unresolved external symbol" (MSVC), with no line number. That is the signature of a link error from §3.1: the compiler believed the promise, the linker went looking for the definition, and there was a hole it could not fill. Recognising "no line number, talks about a symbol" as link, not compile, is a genuinely useful reflex — it tells you to look at what is defined where, not at your syntax.

0.3.4Interrogate your compileropen

Compilers can dump every macro they predefine. Run one of these and skim the output — there are hundreds: 

clang++ -std=c++20 -dM -E -x c++ /dev/null   # Clang / GCC
cl /std:c++20 /Bv /EP nul                      # MSVC is fiddlier; see docs

Find three macros that describe your platform (OS or CPU) rather than your compiler. What might the engine use them for?

Solution sketch

You will find things like __APPLE__, __linux__, _WIN32 (OS), and __x86_64__, __aarch64__, _M_X64 (CPU). The engine uses exactly these to compile the right platform code — a Windows-specific window creation path behind #if defined(_WIN32), say, or a NEON code path behind __aarch64__. This is the mechanism behind the platform layer: the OS-specific code is quarantined behind these macros so the rest of the engine never has to think about which system it is on. You have just met, in miniature, how one codebase targets three operating systems.

9Recap & Next

What the engine can do now that it could not before

  • It builds. You have a working C++20 toolchain on your OS, verified from a terminal, and a program that proves it.
  • You can read a build. The four stages — preprocess, compile, assemble, link — are distinct, and you can tell a compile error from a link error before reading the message, because they speak different dialects.
  • You have the first file. hello.cpp exists, compiled by hand, with a compile-time guard that will never let a misconfigured standard slip through silently.

The idea to carry forward: a build is an assembly line, not a black box. When something goes wrong later — and it will — the first question is always "which stage spoke to me?"

10Further Reading

  • Reference MSVC predefined macros — the authoritative list, including the _MSC_VER table and the /Zc:__cplusplus story. Worth a bookmark.
  • Reference GCC common predefined macros and the predef project — a cross-compiler catalogue of every identity macro, for when you write platform code in Module 5.
  • Reference cppreference: feature-test macros — what <version> gives you: a standard way to ask "does this compiler have that specific C++ feature?"
  • Talk Matt Godbolt, "The Bits Between the Bits" (and Compiler Explorer) — watch source become assembly, live, in your browser. The single best way to make the "compile" stage of Figure 1 real.
  • Book John Lakos, Large-Scale C++ — the deep treatment of translation units, physical design, and linking. Far beyond today's needs, but it is where the separate-compilation model of §2.2 leads, and it underpins the Module 5 refactor.
