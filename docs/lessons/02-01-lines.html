Module 2 — Software Rasterizer I · Lesson 2.1

Lines: DDA, then Bresenham

A framebuffer can set one pixel. A line is a continuous, infinitely thin thing that happens to pass through some of them. Deciding which is the first rasterization question you will ever answer, and the shape of the answer — walk one axis, keep an error term, decide — is the shape of every one that follows. We will derive two algorithms, then measure them, and the measurement will contradict the famous reason for preferring the second one.

1The Problem

Module 1 ended with a complete game, and there is a small debt buried in it. Since Lesson 1.7 the demos have drawn arrows using a routine that lived in main.cpp under a comment apologising for itself. Today that debt comes due, because Module 2 is going to draw a great many lines — every wireframe edge of every mesh — and the routine that draws them is the first piece of the software rasterizer.

So: you have a framebuffer that can set one pixel. Draw a line from (x₀, y₀) to (x₁, y₁).

Everyone writes the same thing first, and it is a perfectly reasonable thing to write. A line is y = mx + b. So walk x from one end to the other, work out y at each step, and set that pixel:

Try it on a gently sloping line and it is perfect. Try it on a fan of lines radiating in every direction — which is the demo you will build today — and half of them come apart.

The failure mode — a dotted line

Every line steeper than 45° is drawn as a row of disconnected dots, and the steeper it gets the further apart they are. Not roughness, not aliasing: actual holes, with background visible through them.

It is quantifiable, which is better than describable. A line from (64, 10) to (84, 110) spans 101 rows and 21 columns. The loop above lights 21 pixels — one per column — and every single one of them is disconnected from the others. A correct routine lights 101. In the fan you will build, the naive version lights 1483 pixels where the correct ones light 2081.

And there is a second failure, which is worth more than the first because it is not a matter of degree. Ask that loop for a vertical line — from (5, 0) to (5, 100) — and x1 - x0 is zero. The slope is not large. It does not exist. There is no value of m for which y = mx + b describes the set of points where x = 5.

That is not a rounding problem or a special case to patch around. It is the equation being the wrong description of the thing. y = mx + b describes a function from x to y, and a line is not a function from x to y — it is a set of points, and some lines have several points per column and some have none. Any formulation that needs an if to survive one of its own inputs is telling you something about the formulation.

Both failures have the same root, and once you see it the fix writes itself.

2Building Intuition

Which pixels are the line?

Start by being honest about what is being asked. A mathematical line has no width. A pixel is a little square with a definite position. The line does not "contain" pixels and pixels are not "on" the line — the line passes over some of them, clips the corners of others, and misses the rest. Drawing it means choosing a set of pixels that stands in for something that was never made of pixels in the first place (Figure 1).

Figure 1 also contains the answer to the naive routine's first failure, if you look at what the rule says: in each column. That is fine for a shallow line, where each column really does contain one interesting pixel. For a steep line a single column contains many rows of the line, and "one pixel per column" throws all but one of them away.

One step, one pixel

Say it as a picture. Walk along the line and ask how far it moves per step (Figure 2). Step x by one on a line with slope 5, and y leaps five rows. Two pixels five rows apart are not neighbours; they are two dots with four empty rows between them.

Notice what the major-axis rule does to the vertical line. There is no slope to compute and no division to guard, because we never form dy/dx at all; we form dy/max(|dx|,|dy|), whose denominator is zero only for a line of zero length. The special case did not get handled. It stopped existing.

The staircase is not a bug

One more thing to settle before the algebra, because it will save you from trying to fix something that is not broken. The output is a staircase. It will always be a staircase. There is no cleverer choice of pixels that makes a diagonal line look smooth, because the pixels are where they are and a diagonal line is not made of horizontal runs.

You have met this before. Lesson 1.8 spent its length on a simulation that advanced in steps and therefore could not see what happened between them; the ball tunnelled through a paddle because collision was sampled at instants rather than over an interval. This is the same thing in space rather than time. We are sampling a continuous line at the discrete positions of pixel centres, and everything the sampling misses has to go somewhere. Here it goes into the staircase.

This is aliasing, and it has a real fix — later

The jagged edge has a name, aliasing, and a family of fixes (anti-aliasing) that work by admitting that a pixel is an area rather than a point: a pixel the line half-covers should be drawn half-bright. Xiaolin Wu's line algorithm does exactly that and is a small modification of what we are about to write — Exercise 2.1.5.

We are not doing it now, deliberately. This is the ninety-percent picture; the missing ten percent is coverage-based sampling, and it arrives properly in Module 6 alongside the rest of the sampling story, where it can be paid off once for lines, texture minification, and shadow edges together. For now: aliasing is a consequence of point-sampling, it is expected, and it is not a defect in your arithmetic.

3The Theory

DDA, derived

Take the major-axis rule and write it down. Let dx = x₁ − x₀ and dy = y₁ − y₀, and let n be the number of steps:

\[ n = \max\!\left(|dx|,\ |dy|\right) \] n = max(|dx|, |dy|)

In prose: the number of steps is the larger of the horizontal and vertical extents. Divide the journey into that many equal parts, and each part advances the position by:

\[ \Delta x = \frac{dx}{n}, \qquad \Delta y = \frac{dy}{n} \] x_inc = dx / n, y_inc = dy / n

The point of choosing n that way is what it guarantees about these two numbers. Whichever of |dx| and |dy| was larger is n, so its increment is exactly ±1; the other is smaller, so its increment is at most 1 in magnitude. One coordinate advances by a whole pixel every step and the other never advances by more than one. A gap requires a jump of two, so there can be no gaps.

That is the Digital Differential Analyser, and the name is a fossil worth knowing: a differential analyser was an analogue machine that solved differential equations by mechanical integration, and this is that idea digitised — take a small constant step, add the derivative, repeat.

Worked example — DDA on a steep line

From (64, 10) to (84, 110): dx = 20, dy = 100.

  • n = max(20, 100) = 100 — the major axis is y.
  • x_inc = 20/100 = 0.2, y_inc = 100/100 = 1.0
  • Step 0 plots (64, 10). Step 1 is at (64.2, 11) → (64, 11). Step 5 is at (65.0, 15) → (65, 15).
  • 101 steps, 101 pixels, each exactly one row below the last. Solid.

Compare the naive loop on the same line: 21 pixels, all disconnected. The difference is entirely the choice of which axis to walk.

DDA is correct, it is about as short as the naive version, and it handles every octant. You could stop here — and it would be reasonable. But look at what happens inside the loop, because that is where the next algorithm comes from:

Two floating-point additions and two roundings, per pixel. In 1962 that was ruinous — the machines Jack Bresenham was driving plotters with had no floating-point hardware at all, and a float add was a subroutine call costing tens of operations. We will see in §3.7 that this argument has not merely weakened since; it has reversed. But there is a second problem with the loop above that has not aged at all, and it is the one that matters:

The defect that is not about speed

x and y are running sums. The value at step i is the result of i additions, each of which rounded to the nearest representable float. The error does not cancel; it accumulates, and it grows with the length of the line.

So the pixels a DDA chooses depend on the order the additions happened in, on the precision of the accumulator, and on the compiler's liberty to contract a*b+c into a fused multiply-add. Those are exactly the things Lesson 1.8 identified as the reason floating-point determinism does not survive a change of machine. A line is a small thing to get subtly wrong, but in Module 2.12 these lines become the edges of a mesh, and two adjacent triangles that disagree about where their shared edge lies produce visible cracks.

So the question is not really "can we avoid the floats to go faster?" It is: can we make the decision exactly? And the answer turns out to be yes, using nothing but integers, which is a much stronger property than being fast.

The midpoint decision

Work in the first octant — where the line goes right and down-or-level, and is no steeper than 45°, so 0 ≤ dy ≤ dx. Everything else follows by reflection (§3.5), and doing the reflections later keeps this derivation clean.

In this octant x is the major axis, so we plot exactly one pixel per column and the only question at each step is whether y stays or advances by one. Just two candidates (Figure 3). Call them E (east, same row) and NE (north-east, next row) — the traditional names, from a convention where y increases upward. In our framebuffer +y is down, so "NE" is down-and-right on screen; the names are historical and the geometry is unchanged.

Now make it arithmetic. Let m = dy/dx be the slope, and define the error e as how far the true line has drifted from the row we plotted:

\[ e \;=\; y_{\text{true}} - y_{\text{plotted}} \] e = (the line's y here) - (the y we plotted here)

At the start we are exactly on the line, so e = 0. Each step right advances the true line by m while the plotted row stays put, so e grows by m. And the midpoint test is now trivially expressible: the line has passed M exactly when e ≥ ½. When it does, we plot the next row instead — and since the plotted row just went up by one, e goes down by one.

That is the entire algorithm, in three rules:

\[ e \mathrel{+}= m \quad\text{each step};\qquad \text{if } e \ge \tfrac{1}{2}:\; y \mathrel{+}= 1,\; e \mathrel{-}= 1 \] each step: e += m; then if e >= 0.5: y += 1 and e -= 1

In prose: accumulate the slope into an error term; whenever it reaches half a pixel, step the minor axis and take a whole pixel back out. Note there is no longer any evaluation of the line — no multiply, no y = mx + b. We are tracking a difference, and differences are cheap.

Making it integral

The rules above still contain m = dy/dx and ½, so e is fractional. Here is the step that makes Bresenham Bresenham, and it is one line of reasoning.

Look at what e is used for. It is compared against ½, and it has m added and 1 subtracted. Nothing else. And comparisons survive multiplication by a positive constant: e ≥ ½ says the same thing as ke ≥ k/2 for any k > 0.

So choose the k that clears every denominator. The denominators are dx (from m) and 2 (from ½), so take k = 2dx and define E = 2·dx·e. Substituting:

Fractional ruleMultiply by 2dxInteger rule
e starts at 0—E starts at 0
e += dy/dx2dx · dy/dx = 2dyE += 2dy
e ≥ ½2dx · ½ = dxE ≥ dx
e −= 12dx · 1 = 2dxE −= 2dx

Every quantity on the right is an integer, because dx and dy are. No division survived. No rounding is performed anywhere, so none can accumulate: for a given four integers this produces one exact answer, on every machine, forever.

What just happened is a technique, not a trick

"The value is only ever compared and incremented, so scale it until the fractions vanish" is a move you will make again. It is how edge functions stay exact in Lesson 2.2, how sub-pixel precision is handled in real rasterizers (fixed-point, with a chosen number of fractional bits), and how depth values are stored in Module 3.

The general shape: if a quantity's only uses are order-preserving, you are free to scale it into a representation you like better.

All eight octants

Everything above assumed the first octant. Real lines point anywhere, and there are eight cases — chosen by three independent bits: is x increasing, is y increasing, and is the line steep or shallow (Figure 5).

Writing eight loops would be miserable, and reflecting in and out costs bookkeeping on every pixel. The form everyone actually ships fuses the reflections into two sign variables and two independent tests:

Two things about it are worth understanding rather than accepting. First, the two ifs are independent — neither is an else of the other. They ask separate questions: should x step? and should y step? On a shallow line the first fires almost always and the second rarely; on a steep line it is the reverse; on an exact 45° line both fire every iteration and the walk goes diagonally. One loop covers every octant because the octant only affects how often each test passes.

Second — and this is the part that makes the loop safe rather than merely plausible — at least one of the two tests always fires, so the position always advances and the loop always terminates. That is provable in one line:

Why the loop cannot hang

Suppose both tests failed. Test one failing means 2·err < dy. Test two failing means 2·err > dx. Together they require:

\[ dx \;<\; 2\,err \;<\; dy \;\le\; 0 \;\le\; dx \] dx < 2*err < dy <= 0 <= dx

— which says dx < dx. Impossible, because dx is an absolute value and therefore ≥ 0, while dy is a negated absolute value and therefore ≤ 0. So at least one test passes on every iteration, every step moves at least one pixel closer to the endpoint, and the endpoint is reached in a finite number of steps.

Where this lesson simplifies, and how it is covered

Deriving the compact form directly — showing algebraically that err = dx + dy with those two tests reproduces the midpoint decision in all eight octants — is a page of sign bookkeeping that teaches very little. This is the ninety-percent picture, and rather than wave at the missing ten percent, the course checks it: the verification harness implements the reflect-in / reflect-out reference from Figure 5 and compares it against the compact form for 1600 lines radiating in every direction, including all the axis-aligned and exactly-diagonal boundary cases.

They agree on every pixel of every line. If you want the algebra, Bresenham's original 1965 paper is three pages and does it properly — it is in Further Reading.

Ties, and a small theorem

One question was skated over in §3.2, and following it leads somewhere genuinely interesting. What if the line passes exactly through the midpoint M? Then e is exactly ½, both candidate pixels are exactly equally close, and there is no right answer.

We have to pick. The code writes E ≥ dx rather than E > dx, which breaks ties toward NE — a choice, not a discovery. And that choice has a visible consequence:

Bresenham is not symmetric

Drawing from A to B can light a different set of pixels than drawing from B to A. Reversing the line reverses which of the two tied candidates is "NE", so every tie flips to the other side.

Measured over 23,103 lines from a common centre: 7,692 of them — one in three — change pixels when the endpoints are swapped. For example (2,2) → (12,7) lights row 5 at x = 7 and x = 8, while (12,7) → (2,2) lights row 4 at those columns.

Which lines? Not the ones you would guess by eye. The answer is exact, and it is a fact about arithmetic rather than about geometry.

A tie exists if and only if major / gcd is even

Write M for the major-axis extent and n for the minor, g = gcd(M, n), and reduce the slope to lowest terms as n/M = q/p where p = M/g and q = n/g are coprime.

A tie at step k means the accumulated error is exactly a half-integer:

\[ k\,\frac{n}{M} = j + \tfrac12 \quad\Longleftrightarrow\quad 2kq = (2j+1)\,p \] k * (n/M) = j + 1/2 <=> 2*k*q = (2*j + 1) * p
  • If p is odd: the left side is even and the right side is odd × odd = odd. No solution. No tie is possible.
  • If p is even: take k = p/2, which is a whole number. Then 2kq = pq = (2j+1)p gives q = 2j+1 — and q is odd, because it is coprime to the even p. A tie occurs, at step p/2 and every p steps after.

So slope 1/2 (p = 2, even) ties constantly; slope 3/5 (p = 5, odd) never ties; and a perfect 45° diagonal (p = 1, odd) never ties either — which is why diagonals are the one case everybody's line routine agrees on.

Verified exhaustively. Over all 23,103 lines from a common centre in a 160×160 grid, the set of lines that are asymmetric under endpoint swap is exactly the set this predicts — 7,692 lines, with zero disagreements in either direction.

This is not trivia. Module 2.2 draws triangles, and adjacent triangles share edges. If the shared edge is rasterised differently depending on which triangle asks for it, you get cracks of background between them, or a doubly-drawn seam that shows through transparency. Real rasterizers solve it with a fill rule — a tie-breaking convention that depends only on the edge's geometry and not on the direction it is traversed. That is a Lesson 2.2 topic, and you now know precisely why it needs to exist.

Is it actually faster? (measured)

Here is the claim you will find in every textbook and most of the internet: Bresenham is faster than DDA because integer arithmetic is faster than floating-point arithmetic.

That claim was decisively true in 1962. Let us check whether it is true now, because this course measures rather than repeats. The benchmark below times the stepping of each algorithm with the framebuffer write removed, so what is compared is the arithmetic and nothing else; it runs several line mixes so no single geometry can flatter one variant; and it includes a second Bresenham written in the classic major-axis form with one branch in the loop rather than two, so that "Bresenham lost because you wrote it badly" is answered up front.

The folklore is inverted on this hardware

DDA is about 2.2× faster than the compact Bresenham, and still about 1.2× faster than the best Bresenham. The integer algorithm is the slow one.

Why? Two reasons, and the benchmark isolates both.

The floats were never the problem. A modern out-of-order core issues a floating-point add with the same one-cycle throughput as an integer add, and has more ports to do it on. Replacing std::lround with the old +0.5-and-truncate trick changes almost nothing (0.595 → 0.645 ns), which tells you the rounding is not the cost either. The DDA inner loop is straight-line code — two adds, two converts, no decisions — and a superscalar pipeline eats it.

The branches are the problem. Bresenham's inner loop asks a data-dependent question every pixel, and on a line of general slope the answer alternates in a pattern the branch predictor cannot learn. Each mispredict throws away the pipeline: on the order of 15–20 cycles, which at these scales is enormous. The evidence is in the second row of the table — restructuring Bresenham so the loop contains one branch instead of two nearly halves its cost (1.320 → 0.738 ns), without changing a single arithmetic operation.

So why does draw_line use Bresenham?

Because speed was never the good argument, and this course would rather teach you the good one.

  • Exactness. The chosen pixels are a deterministic function of four integers. No accumulation, no rounding mode, no fused-multiply-add contraction, no -ffast-math surprise, identical on every compiler and every architecture. Lesson 1.8 showed that floating-point determinism does not survive a change of machine; integer determinism does. When these lines become mesh edges in Lesson 2.12, that is worth more than a nanosecond.
  • It is the next lesson. Bresenham's error term is a running answer to "which side of the line is this point on?" — and that question, asked for three edges at once, is the edge function that Lesson 2.2 uses to fill triangles, and that GPUs implement in hardware. Learning it here means arriving at 2.2 already understanding the mechanism.
  • Lines are not the hot path. A wireframe cube is a few hundred pixels; a filled scene is millions. Optimising the line routine would be optimising the wrong thing — precisely the trap Lesson 1.5 fell into deliberately when its first benchmark measured both loop orders through put_pixel and found a ratio of 1.00×.

Choosing the measurably slower algorithm, on purpose, for stated reasons, with the number written down so you can disagree — that is what engineering judgement actually looks like. Exercise 2.1.3 invites you to make the opposite call.

4Implementation

Module 2 gets a new file, and the name matters: src/gfx/raster.hpp. A framebuffer knows how to set one pixel; a rasterizer knows which pixels a shape is made of. Every file added to gfx/ for the rest of Module 2 and 3 answers that same question about a more interesting shape — triangles, then triangles with depth, then triangles with texture.

The header starts with a line borrowed from Lesson 1.8:

Every function here takes a framebuffer& and none of them needs to know its layout, so the compiler needs only the name. raster.cpp includes the real header because it actually writes pixels. Include what you use; forward declare what you merely mention — the habit that keeps a large C++ build from taking twenty minutes.

All three routines share one signature, and that is deliberate rather than incidental:

A function pointer, and the demo swaps it. This is the same discipline Lesson 1.6 used to compare the two mixing rules: if the two paths differ in anything except the one thing under test, you are comparing code rather than ideas. Here the fan, the colours, the counting and the timing are byte-for-byte identical across all three algorithms, so every difference you see on screen is attributable.

C++ in place — function pointers

void (*)(framebuffer&, int, int, int, int, Uint32) reads outwards from the inner parenthesis: pointer to a function taking those six parameters and returning void. The using alias exists because the raw spelling is unreadable at the point of use, not because it does anything extra.

A function pointer is the simplest thing in C++ that lets a decision be made at run time and then acted on many times without re-deciding. It costs an indirect call — which the branch predictor handles very well when the target is stable, as it is here for a whole frame. When we need to bind state alongside the function (a lambda's captures), we will reach for std::function or a template parameter instead; those have their own costs, and Module 5 has that conversation properly when the renderer needs callbacks.

The demo draws a fan of spokes rather than a few illustrative lines, and that is the whole test strategy in one picture. A fan crosses every octant, so a routine that is wrong in any of them is wrong on screen rather than wrong in theory:

Steep spokes are drawn coral and shallow ones green, so the naive routine's failure is not just visible but diagnosable: it is exactly the coral half that comes apart, which tells you immediately that the bug is about the major axis and not about, say, negative coordinates.

Finally, the pixel inspector — the part of the demo that turns the theory into something you can look at. It draws the line into a tiny 15×13 framebuffer, then reads that back and paints each lit cell as an 8×8 square:

Note what this does not do: it does not reimplement the algorithm at 8× scale to produce a picture of it. It runs the same function the engine runs and reads out the answer. A diagram that is generated by different code from the thing it documents will eventually lie about it; this one cannot.

6Build & Run

What you should see

A window titled "Lines — Module 2", showing a slowly rotating fan of 32 spokes on the left and a magnified 15×13 pixel grid on the right (Figure 6). Steep spokes are coral, shallow ones green. In the inspector, yellow squares are the pixels the algorithm chose and the thin blue line is the ideal line they are standing in for.

Checkpoint

With Bresenham selected the readout should say 2081 px lit for the fan, and every spoke should be continuous — put your face close to the screen and follow a coral one out from the centre. Press 2 for DDA: the count stays 2081, and the picture is to all appearances identical. That is the point of the next step.

Now break it

  1. Press 1 for the naive routine. The coral spokes — every one steeper than 45° — turn into dotted lines, while the green ones are untouched. The count drops from 2081 to 1483.
  2. Watch the fan rotate (or press Space to stop it and start it). A spoke is solid, then falls apart, then re-forms, and the transition happens exactly as it crosses 45° — the moment its major axis changes from x to y.
  3. Look at the inspector with the naive routine selected and a steep line showing: the chosen squares are separated by empty rows, and the ideal blue line runs through the gaps between them. That is the failure at its most literal.
  4. Press 3 and watch it heal without anything else changing.

Then compare the us/fan figure between 2 and 3. On the machine this lesson was written on, DDA is the faster of the two — and now you know why, and why we shipped the other one anyway.

7Common Pitfalls & Debugging

Steep lines are dotted — gaps that widen with slope
Symptom
Shallow lines are perfect; lines steeper than 45° are rows of separated dots.
Cause
One pixel per column, on a line that spans more rows than columns. The loop is walking the minor axis.
Fix
Walk max(|dx|, |dy|) steps instead. The diagnostic that confirms it before you change anything: count the lit pixels. If the total equals |dx| + 1 when the line spans more rows than columns, this is certainly the bug.
Vertical lines crash or vanish — division by zero
Symptom
A perfectly vertical line draws nothing, draws one pixel, or fills the screen. With floats there is no crash — you get inf or NaN and it propagates silently, exactly as in Lesson 1.3 §3.5.
Cause
dy / dx with dx == 0. The slope of a vertical line does not exist, so any formulation that computes one has to special-case it.
Fix
Do not compute a slope. Dividing by max(|dx|, |dy|) is zero only for a zero-length line, and Bresenham divides by nothing at all. The special case stops existing rather than getting handled.
The last pixel is missing — lines one short at one end
Symptom
Lines look right, but a rectangle drawn as four lines has gaps at its corners, or a closed polygon does not quite close.
Cause
An exclusive endpoint: a loop written for (i = 0; i < steps; ++i) or a Bresenham that tests for the end before plotting rather than after.
Fix
Decide the convention and write it down. Ours is inclusive at both ends — hence i <= steps, and the break placed after put_pixel. Half-open ranges are the better choice for filling (Lesson 2.2 uses them for exactly the shared-edge reason), but for a visible outline you want both endpoints.
The line is a staircase — jagged edges
Symptom
Diagonal lines have visible steps rather than looking smooth.
Cause
Nothing. This is correct behaviour. The pixels are where they are, and a diagonal line is not made of horizontal runs. It is aliasing, the consequence of sampling a continuous shape at discrete points.
Fix
None needed here, and resist the urge to invent one — time spent trying to choose better pixels is wasted, because there is no better choice within one bit per pixel. The real fix is more bits: draw partly-covered pixels partly bright (Xiaolin Wu, Exercise 2.1.5; the general treatment in Module 6).
Drawing off-screen is slow — frame rate collapses when geometry leaves the view
Symptom
Everything is fine until an object moves off the edge of the screen, at which point the frame rate falls off a cliff — while the picture stays correct.
Cause
put_pixel is bounds-checked, so out-of-range pixels are discarded rather than written. Correct, and completely wasted: a line from (-100000, 0) to (100000, 0) costs 200,001 iterations to draw at most 320 pixels.
Fix
Clip the line to the viewport before walking it — Cohen–Sutherland, Exercise 2.1.4. Note the shape of this bug: correctness masked a performance problem, so nothing looked wrong until the profile said so. Module 3 makes clipping non-optional for a reason of correctness rather than speed, when geometry behind the camera turns inside out.
Two triangles sharing an edge have a crack — a hairline of background
Symptom
A seam of background pixels shows between two shapes that share an edge, or the shared edge looks doubly dark through transparency.
Cause
§3.6: Bresenham is asymmetric on any line whose major / gcd(major, minor) is even — a third of all lines. The shared edge is traversed in opposite directions by the two shapes, so every tie falls the other way and the two rasterisations disagree.
Fix
Do not rely on line-drawing for shared boundaries. Filled shapes need a fill rule that depends only on the edge's geometry, never on traversal direction — Lesson 2.2. If you must draw the outline, canonicalise the endpoint order first (always draw from the lexicographically smaller endpoint) so both shapes ask the identical question.

8Exercises

2.1.1Trace it by hand, then check the machinewarm-up

Work out the pixels for (0, 0) → (8, 3) using the integer rules from §3.3 — constants 2dy = 6, threshold dx = 8, payback 2dx = 16 — and write the table as in §3.4, including the "true y" and "nearest row" columns. Then print what draw_line actually produces and compare.

Hint

Nine columns, so nine pixels. The slope is 3/8 = 0.375, and 8 / gcd(8,3) = 8 is even — so by §3.6 this line does contain a tie. Find which column it lands in and note which way your trace breaks it.

Solution

(0,0) (1,0) (2,1) (3,1) (4,2) (5,2) (6,2) (7,3) (8,3).

The tie is at k = p/2 = 4: the true y at x = 4 is exactly 1.5, dead between rows 1 and 2. The E ≥ dx comparison takes it, giving row 2. Had the code been written E > dx you would get row 1 there and a different — equally correct — line. Draw (8,3) → (0,0) and you will see the other choice appear on its own.

2.1.2Prove the equivalence on your own machinewarm-up

Write the reflect-in / reflect-out reference from Figure 5: implement the midpoint algorithm for the first octant only, then handle the other seven by negating x, negating y, and swapping the axes before the loop, undoing each transformation on every pixel that comes out. Compare it against draw_line for every endpoint on a lattice. They must agree on every pixel.

Hint

Reduce, then run, then un-reduce — and un-reduce in the exact reverse order you reduced. If you swap the axes first and negate second on the way in, you must negate first and swap second on the way out. Getting this backwards produces lines that are correct in four octants and mirrored in the other four, which is a satisfying bug to look at.

Solution sketch

The course's harness does exactly this and finds zero mismatches over 1600 lines. The valuable part is not the result — you already know they agree — but that you now have a reference implementation: something slow and obviously correct to check a fast and clever thing against. That pattern is worth more than this exercise. Every optimisation in Module 3 and beyond should be born with one.

2.1.3Disagree with us, with evidencemedium

§3.7 chose the measurably slower algorithm for reasons of exactness. Make the opposite case. Reproduce the benchmark on your machine, then find a workload where the difference actually matters — and one where it provably does not.

Hint

Time the whole frame, not just the line routine. Then ask what fraction of it the lines were. A wireframe cube is a few hundred pixels out of 57,600 in our framebuffer; you will need to work quite hard to construct a scene where line stepping is more than a rounding error in the frame time. That difficulty is the answer to half the question.

Solution sketch

A defensible case for DDA: a debug-draw system emitting tens of thousands of lines per frame (Module 5 builds one), where nothing is ever reused between frames and exactness buys nothing. A defensible case for Bresenham: anything whose output is compared between runs, machines, or adjacent primitives.

The judgement worth taking away is that "which is faster" was the least interesting question available, and answering it took the most work. Ask what the code is for first.

2.1.4Clip before you walkmedium

Implement Cohen–Sutherland clipping and apply it before the loop. Assign each endpoint a 4-bit outcode (above / below / left / right of the viewport); if both are zero the line is entirely inside, if their bitwise AND is non-zero it is entirely outside and can be rejected immediately, and otherwise clip against one violated edge and repeat.

Hint

The trivial-reject test is the elegant part: if both endpoints are above the top edge, both outcodes have the "above" bit set, so their AND is non-zero. One AND rejects the majority of off-screen geometry with no arithmetic at all.

Two traps. Clipping moves an endpoint, which changes which pixels the line lights — the clipped line is a slightly different line. And a clipped endpoint lands on a fractional coordinate; rounding it re-introduces exactly the drift Bresenham exists to avoid. Both are real, and both are why professional rasterizers clip in fixed-point.

Solution sketch

Verify with the property that matters: for every line, the set of pixels drawn inside the viewport must be identical with and without clipping. That is a stronger test than "it looks the same" and it will catch the rounding issue above immediately. Expect to discover that exact agreement requires care, and that the honest fix is to clip the parameter range of the walk rather than the endpoints themselves.

2.1.5Anti-alias it (Xiaolin Wu)open

Admit that a pixel is an area. Implement Wu's algorithm: at each step along the major axis, light two pixels — the ones either side of the true line — with brightnesses proportional to how close the line is to each. The error term you already have is exactly the coverage fraction you need.

Hint

Where Bresenham asks "is e ≥ ½?" and throws the rest away, Wu uses e: the far pixel gets e and the near one 1 − e. You will need to blend against the existing pixel rather than overwrite it, which means reading the framebuffer back — and blending brightness correctly means doing it in linear light, not on stored sRGB values. That is Lesson 1.6's mix_linear, and this is its first real use.

Solution sketch

Do it wrong first, on purpose: blend with mix_encoded and compare. The anti-aliased line will look thinner and darker than the aliased one, because averaging sRGB values under-represents the light. Switch to mix_linear and it snaps to the right weight. Lesson 1.6 argued this with ramps; here it is in a shape you care about.

The deeper thing you will notice: Wu's lines are no longer a set of pixels but a set of coverages, so "which pixels is this line?" stops having a yes/no answer. That is the conceptual door into Module 6's sampling material, and it is worth pushing on now while the shape is this small.

9Recap & Next

What the engine can do now that it could not before

  • It has a rasterizer. src/gfx/raster exists, and draw_line is correct in all eight octants, endpoint-inclusive, and free of floating point.
  • The debt from Lesson 1.7 is paid. The apologetic local DDA in main.cpp is gone, derived properly, and replaced by engine code.
  • The error-term technique. Accumulate a difference, compare it against a threshold, discharge — and scale away the fractions because the only operations are order-preserving. You will use this again within one lesson.
  • An exactness argument you can make out loud. Integer decisions are reproducible in a way floating-point ones are not, and now you have measured what that costs.
  • A small theorem about ties that predicts precisely which lines are ambiguous — and a preview of why filled shapes need a fill rule.

And a habit worth more than any of it: the received reason for a decision may have expired decades ago. "Integers are faster than floats" was true, is now false on this hardware, and is still repeated everywhere. We kept Bresenham anyway — but for the real reason, with the number written down, which is a completely different intellectual position from keeping it out of deference.

Next: 2.2 — The Triangle: Edge Functions. A line divides the plane in two, and Bresenham's error term has been quietly telling us which side of that division we are on. Ask that question for the three edges of a triangle at once and a point is inside exactly when all three answers agree — which is a rasterizer for filled shapes, and happens to be how GPUs actually do it.

It also resolves the loose end from §3.6. Edge functions are computed from the edge's geometry rather than from a traversal, so the tie-breaking that makes Bresenham asymmetric becomes a fill rule — a convention two adjacent triangles cannot disagree about.

10Further Reading

  • Bresenham (1965) Algorithm for computer control of a digital plotter, IBM Systems Journal 4(1). Three pages, and it does the sign algebra this lesson deliberately skipped. Worth reading for the context as much as the maths: the constraint was a plotter with no floating-point unit at all.
  • Foley & van Dam Computer Graphics: Principles and Practice, §3.2. The classic textbook treatment, including the midpoint formulation for circles and ellipses — the same error-term technique applied to a curve, which is the best way to confirm you have understood the technique rather than memorised the line case.
  • Xiaolin Wu (1991) An efficient antialiasing technique, SIGGRAPH '91. The anti-aliased line of Exercise 2.1.5.
  • Fabian Giesen A trip through the Graphics Pipeline and the "Optimizing Software Occlusion Culling" series. The best available writing on how real rasterizers work; read it after Lesson 2.2, when edge functions will make the whole thing click.
  • Agner Fog Optimizing software in C++, the chapters on branch prediction. If §3.7's result surprised you, this explains the machine that produced it.
