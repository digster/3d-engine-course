Checkpoint: Pong

Seven lessons have given you a clock, an input snapshot, a fixed-step accumulator, a framebuffer, a colour model and a vector type. Not one of them is a game. Today they become one — and the assembly is not glue work, because putting the parts together exposes a failure that no demo so far could possibly have shown you: a ball that passes clean through a paddle, on someone else's machine, while working perfectly on yours.

1The Problem

Take stock of what you own. A clock that measures time in integer nanoseconds and admits when it lies. An input that answers both "is this held?" and "was this just pressed?" from one coherent snapshot. A fixed_step accumulator that decouples the simulation's rate from the display's. A framebuffer whose every pixel you set yourself. A colour module that knows the difference between a stored number and the light it stands for. And a vec2 that separates direction from speed.

Six subsystems, all correct, all tested, and together they do not constitute a game. Something has to want something — and the moment something does, a category of bug appears that none of the previous seven lessons could have shown you, for a simple reason: nothing in this course has ever hit anything before.

So before we write a line, here is the bug. Not described in the abstract at the end of the lesson, but named now, because you are going to build it on purpose and watch it happen.

The failure mode — the ball goes through the paddle

You write the obvious collision test: after moving the ball, do the ball's rectangle and the paddle's rectangle overlap? It is four comparisons, it is correct, and every book shows it. You play. It works.

Then someone runs your game on a machine where the simulation ticks more slowly, or you speed the ball up for a boss fight, and the ball begins passing through the paddle as though it were not there. Not intermittently — reliably, and more often the faster the ball travels. This is called tunnelling, and the reason it is worth a whole section of a checkpoint lesson is the part that follows.

You cannot see it on your machine. We will calculate the exact speed at which the naive test starts failing: it is (paddle width + ball width) ÷ h. With our numbers at 60 Hz that is 480 pixels per second, and the ball's speed is capped at 260. The bug is not just absent from your testing — it is unreachable from your testing, sitting in the code, waiting for a step size you never tried.

There is a second, gentler failure waiting too, and it is a callback. Lesson 1.4 ended with a rule stated and then not used, because nothing in that demo teleported: never interpolate across a discontinuity. Pong teleports the ball to the centre of the court every time someone scores. Interpolate that and the renderer draws a ball streaking across the whole screen in one frame, at a speed the ball never had, on the frame you least want a visual glitch — the one where the player just lost a point.

Both of these are the same lesson wearing different clothes, and it is the lesson this checkpoint exists to teach: a simulation that advances in steps does not see the spaces between them. Everything else today — the game, the AI, the score, the bouncing — is consolidation you have earned. That one idea is new, and it will come back in Module 3 when a triangle straddles the near plane, and again in Module 7 when a bullet meets a wall.

2Building Intuition

State, rule, picture

Strip a game of its genre and three things are left. There is state: every number that says what is currently true — where the ball is, how fast, what the score is. There is a rule that turns state at one instant into state a moment later, given what the players asked for. And there is a picture drawn from that state, which changes nothing.

That division is not an architecture fashion; it is forced on us by Lesson 1.4. The fixed-step loop runs the rule zero, one, or several times per frame, and then draws once. Any code that both advanced the game and drew it would advance the game a variable number of times per second — which is precisely the disease the fixed timestep cured. Figure 1 shows the three parts with the loop wrapped around them, and it is worth comparing against the loop diagram in Lesson 1.4: this is that diagram with the boxes named.

Where the engine stops

Here is a question worth more than it first appears. Which directory does Pong live in?

Everything written so far — src/core, src/gfx, src/math — answers questions that any game would ask. What time is it. Which keys are down. How do I set a pixel. How long is this arrow. Not one line of it knows that paddles exist, and that is not an accident of what we happened to need: it is the property that makes those files an engine rather than a program.

Pong is the opposite. It knows nothing but paddles. So it gets its own directory, src/game/, and with it a rule about which way the arrows point (Figure 2): game code may depend on engine code, and engine code may never depend on game code.

It is fair to ask why we bother, given that everything still compiles into one executable and nothing stops framebuffer.cpp from including pong.hpp tomorrow. The answer is that the Module 5 refactor is coming either way, and its difficulty is decided entirely by how many wrong-way dependencies have accumulated by then. Zero of them is a rename. Forty of them is a rewrite. The cheapest moment to maintain a boundary is always before anything has crossed it.

Every step is a teleport

Now the idea the whole lesson turns on, and it is best met as a picture rather than a formula.

We say the ball "moves at 105 pixels per second". It does not. Once per simulation step, code reads the ball's position, computes a new one, and writes it back. In between those two positions the ball is nowhere — there is no instant at which it occupies the space between, because there is no instant at all. The ball's smooth path across the court is something your eye constructs from a sequence of stills, exactly as it does at the cinema.

Which means: if you ask "where is the ball, and is it touching anything?", you are asking about the stills. And a paddle can sit entirely inside the gap between two stills. Figure 3 is that situation drawn to scale with our real numbers.

Sit with the last sentence of that caption, because it is the useful one. The naive collision test is not buggy. Every individual answer it gives is correct. The defect is in the question: "are these two boxes overlapping?" is a question about a moment, and collision is a fact about an interval. Fixing it means changing the question to "did the ball's path cross the paddle at any point during this step?" — and that question, unlike the first, has an answer that does not depend on how finely we chop up time.

This shape recurs — remember it

Sampling a continuous thing at intervals and then being surprised by what fell between the samples is one of the two or three most productive bug generators in all of graphics. It is tunnelling here. It is aliasing when we sample a triangle's coverage once per pixel and get staircase edges (Module 2). It is the shimmering of a distant checkerboard when we sample a texture once per pixel and miss the detail between (Module 3). It is shadow acne when we sample a depth map at a resolution the geometry does not respect (Module 6). Same shape, four modules, four different-looking artifacts.

3The Theory

Two boxes, two axes

Start with the test we are going to outgrow, because understanding it properly is what makes its limitation obvious — and because in a slightly generalised form it is one of the workhorses of collision detection for the rest of your career.

Two axis-aligned rectangles. When do they overlap? The tempting approach is to enumerate the ways they might touch — corner to corner, edge across edge — and it goes badly, because there are a lot of cases. Turn the question over instead and ask when they are apart. Two shapes are apart if you can slide a straight line between them without touching either. And a line between them means: there is some direction along which their shadows do not overlap.

Shadows. That is the dot product's picture from Lesson 1.7, and it is doing real work here. For axis-aligned boxes there are only two directions worth testing — x and y — because those are the only directions the boxes' faces point. So:

\[ \text{overlap} \iff (\text{shadows overlap on } x) \;\wedge\; (\text{shadows overlap on } y) \] overlap == (they overlap on x) AND (they overlap on y)

In prose, so this survives a dead CDN: two boxes overlap exactly when their shadows overlap on both axes; a gap on either axis means they are apart. And "shadows overlap on x" is itself two comparisons — each box must start before the other one ends:

\[ a_{\min} < b_{\max} \;\wedge\; a_{\max} > b_{\min} \] a_min < b_max AND a_max > b_min
Worked example — the ball that got away

Take the exact situation from Figure 3, at the step where the ball has already crossed the paddle. The ball is 4×4 with its top-left at (5.5, 70); the left paddle is 4×30 with its top-left at (10, 60).

  • Ball's shadows: x spans [5.5, 9.5], y spans [70, 74].
  • Paddle's shadows: x spans [10, 14], y spans [60, 90].
  • On y: is 70 < 90? Yes. Is 74 > 60? Yes. The shadows overlap — the ball is at the right height.
  • On x: is 5.5 < 14? Yes. Is 9.5 > 10? No. A gap of half a pixel.

One axis separates them, so they are apart, so there is no collision. The arithmetic is flawless. The ball is on the wrong side of the paddle and about to score a point that should never have happened.

You have just met the Separating Axis Theorem

"Two convex shapes are apart exactly when some axis exists on which their shadows do not overlap" is the Separating Axis Theorem, and it is completely general — it works for any two convex shapes in any number of dimensions. What changes with the shapes is only which axes you must test. For axis-aligned boxes the candidate axes are the two you already know, which is why the test collapses to four comparisons and looks like nothing at all.

Rotate one box and the obvious axes stop being obvious; you must test the face normals of both shapes. That is Module 7's SAT lesson, with diagrams, and you will find you already understand the theorem and are only learning the bookkeeping.

Reflection, from the shadow

Before fixing the timing problem, we need to know what a bounce is. For a ball hitting the top wall the answer looks too easy to need a formula — flip the sign of the y velocity, done — and if we stopped there we would have learned nothing transferable and would be stuck the first time a surface was not axis-aligned.

So derive it properly, from the picture. A velocity v arrives at a surface whose unit normal is n̂ (Figure 4). Split v into two parts:

  • the part along the normal — that is v's shadow on n̂, and it is the part driving the ball into the surface;
  • everything left over, which necessarily runs along the surface.

A bounce leaves the sideways part completely alone — the ball keeps sliding along the wall at the same rate — and reverses the part heading into the wall. To reverse a part, subtract it twice: once to cancel it, and once more to send an equal amount back out.

Writing that down, with project_onto from Lesson 1.7 giving the shadow:

\[ \operatorname{reflect}(\mathbf{v}, \hat{\mathbf{n}}) = \mathbf{v} - 2(\mathbf{v}\cdot\hat{\mathbf{n}})\,\hat{\mathbf{n}} \] reflect(v, n) = v - 2 * dot(v, n) * n

In prose: the reflected vector is the original minus twice its shadow on the normal. Three multiplies and a subtract, and it works for a wall at any angle in any dimension.

The precondition that will bite you

n̂ must be unit length. Lesson 1.7's project_onto(a, b) divides by |b|²; the formula above has no such division, because with a unit normal that division is by one. Hand it a normal of length 2 and you get four times the correction — the ball leaves faster than it arrived, and gains energy on every bounce until it is a blur. Hand it a length-½ normal and the ball sinks into the wall. Nothing warns you. Normalise at the source and the problem never arises.

Worked example 1 — the easy case, to check the formula

A ball moving down and to the right, v = (3, 4), hits the floor. Remember that in framebuffer coordinates +y points down, so the floor's inward normal points up: n̂ = (0, −1), which is already unit length.

  • v · n̂ = 3·0 + 4·(−1) = −4
  • 2(v · n̂) n̂ = 2 · (−4) · (0, −1) = (0, 8)
  • reflect = (3, 4) − (0, 8) = (3, −4)

The x component is untouched and the y component is negated: the ball keeps drifting right at exactly the same rate and now travels upward. Which is what a sign flip would have given — and that agreement is the point of the check. A derivation that disagreed with the obvious case would be wrong.

Worked example 2 — the case a sign flip cannot do

Now tilt the wall to 45°, so its unit normal is n̂ = (1/√2, −1/√2) ≈ (0.70711, −0.70711). Drop a ball straight down onto it: v = (0, 5).

  • v · n̂ = 0·0.70711 + 5·(−0.70711) = −3.53553
  • 2(v · n̂) n̂ = −7.07107 · (0.70711, −0.70711) = (−5, 5)
  • reflect = (0, 5) − (−5, 5) = (5, 0)

A vertical drop leaves horizontally — which is exactly right for a 45° ramp, and is something no amount of negating a single component could ever produce. Note also that |(0,5)| = 5 and |(5,0)| = 5: the reflection preserved the speed, as a mirror must. That length check is the cheapest test you can run on any reflection code, and it catches an unnormalised normal instantly.

The swept test: time of impact

Now the fix for Figure 3. We stop asking where the ball ended up and start asking what the ball's path crossed.

Pong makes this unusually clean, and it is worth saying why rather than getting lucky. A paddle is a thin vertical bar, and the interesting thing about it is a single vertical line: the face the ball can strike. Call its x coordinate face_x. Then a bounce happens if and only if two things are true during this step:

  1. the ball's leading edge crossed face_x — it began on one side and ended on the other; and
  2. at the exact moment of crossing, the ball overlapped the paddle vertically.

The first condition is a comparison. The second needs a number: when did the crossing happen? Because the ball's height at the end of the step is not its height at the moment of contact, and using the wrong one is the same mistake we just fixed, rotated ninety degrees.

Finding that moment is straightforward, because within a single step the velocity is constant — the ball travels in a straight line at a steady rate. Let t run from 0 at the start of the step to 1 at the end. The leading edge goes from lead_from to lead_to, linearly, so:

\[ t = \frac{\textit{face}_x - \textit{lead}_{\textit{from}}}{\textit{lead}_{\textit{to}} - \textit{lead}_{\textit{from}}} \] t = (face_x - lead_from) / (lead_to - lead_from)

In prose: the fraction of the step at which contact occurs is how far the face was from the start, divided by how far the edge travelled in total. It is the same "what fraction of the way is this?" arithmetic as Lesson 1.4's alpha, and once you have t you can ask where anything was at that instant by interpolating it (Figure 5).

One detail in that condition looks redundant and is not. We require that the leading edge began on the near side and ended on the far side. Testing only "ended past the face" would also fire for a ball that got behind the paddle on some earlier step and is now on its way to conceding a point — it would be dragged back and bounced out of a position it had legitimately reached. Requiring both makes it a crossing test rather than a which-side-am-I-on test.

That pair of comparisons has a second, quieter benefit: it guarantees the denominator is not zero. If lead_from and lead_to were equal, one of the two strict comparisons would necessarily have failed and we would have returned already. Crossing implies motion. It is worth noticing when a division is safe by construction rather than by a guard, because those are the divisions that stay safe when someone edits the code around them.

Worked example — the step from Figure 3, done both ways

The ball is 4×4 at (16, 70), travelling straight left at 105 px/s. The simulation is at 10 Hz, so h = 0.1 s and the ball moves −105 × 0.1 = −10.5 pixels this step, ending at x = 5.5. The left paddle spans x ∈ [10, 14] and y ∈ [60, 90], so its face is at face_x = 14.

The naive test — do the boxes overlap at x = 5.5?

  • Ball's x span is [5.5, 9.5]; paddle's is [10, 14]. Is 9.5 > 10? No. Separated. No collision reported.

The swept test — did the path cross the face?

  • Moving left, so the leading edge is the ball's left edge: lead_from = 16, lead_to = 5.5.
  • Crossed? 16 ≥ 14 and 5.5 < 14. Yes.
  • t = (14 − 16) / (5.5 − 16) = (−2) / (−10.5) = 0.190476
  • Height at that instant: y = 70 + (70 − 70)·t = 70, so the ball spans [70, 74], comfortably inside the paddle's [60, 90]. Hit.

Same ball, same paddle, same step. One test says the ball is behind the paddle; the other says it struck it 19% of the way through the step. Only the second answer is about the ball.

And the response, with t in hand, is a small pleasure. Put the ball exactly on the face it touched, give it its new velocity, and then let it travel the remaining (1 − t) of the step in the new direction. Continuing the numbers above: the new speed is 105 + 7 = 112 px/s (the ball speeds up on every hit), the bounce direction works out to (0.978148, −0.207912) in §3.5 below, so the new velocity is (109.5525, −23.2861). The remainder is 0.1 × (1 − 0.190476) = 0.0809524 seconds, and the ball finishes the step at:

\[ x = 14 + 109.5525 \times 0.0809524 = 22.8685 \] x = 14 + 109.5525 * 0.0809524 = 22.8685

That number is not decorative — it is what the code prints when you run it, and it is one of the checks that was run before this lesson was published.

Exactly when the naive test breaks

Here is the part to take with you even if you forget Pong entirely.

The naive overlap test misses when the ball can step clean over the region in which it would have overlapped. So: how wide is that region? Working along x, the ball's box overlaps the paddle's box while the ball's left edge satisfies ball.x < paddle_right and ball.x + ball_size > paddle_left. With our numbers that is ball.x ∈ (6, 14) — an interval whose width is exactly:

\[ w_{\text{window}} = \textit{paddle\_width} + \textit{ball\_width} \] window width = paddle_width + ball_width

In prose: the window in which two boxes overlap along an axis is as wide as the two boxes put together. (Which is a small, generally useful fact: it is the same reason the "Minkowski sum" trick turns box-versus-box into point-versus-bigger-box, and Module 7 will use it by name.)

The ball crosses that window at |vx| pixels per second and gets one sample every h seconds. So the test is guaranteed to catch the overlap when:

\[ |v_x| \, h \;<\; \textit{paddle\_width} + \textit{ball\_width} \] |vx| * h < paddle_width + ball_width

In prose: a discrete overlap test is reliable only while the object moves less than the overlap window's width in a single step. Above that it may miss, depending on where the samples happen to land; well above it, it misses essentially always. Note it is |vx| and not the speed — a ball moving steeply is crossing the window slowly no matter how fast it is going.

Worked example — put our numbers in and get a nasty surprise

Our paddle is 4 px wide and our ball is 4 px, so the window is 8 px. Rearranged, the test is safe while |vx| < 8 / h:

Sim ratehSafe up toBall's top speed is 260 px/s…
120 Hz0.00833 s960 px/snever fails — not even close
60 Hz0.01667 s480 px/snever fails — the bug is unreachable
30 Hz0.03333 s240 px/sfails once a rally passes ~20 hits
10 Hz0.1 s80 px/sfails on the opening serve, at 105 px/s

Read the 60 Hz row again. The safe ceiling is 480 px/s and the ball is capped at 260. There is no sequence of play — no rally length, no lucky angle, no amount of testing — that can make this bug appear at 60 Hz. It is not merely unlikely. It is impossible to observe, and it is definitely there.

The engineering lesson, which is bigger than collision

A test suite that passes tells you the bug is not reachable under the conditions you tested. It does not tell you the code is correct. Here the difference is stark and quantifiable: the naive test is correct for all |vx|h < 8 and wrong above it, and your machine simply never leaves the correct region.

This is why "it works on my machine" is not a joke about laziness — it is a precise statement about a parameter space. The habit worth building is the one this lesson is modelling: when you write a discrete test of a continuous thing, work out the condition under which it stops being valid, and write that condition down in a comment, even when today's numbers are safely inside it. Tomorrow's numbers will not be.

The bounce angle, and a deliberate lie

One piece left: what happens when the ball meets a paddle, as opposed to a wall.

The honest answer is: exactly what happens at a wall. A paddle is a flat vertical surface, its normal is (±1, 0), and reflect with that normal negates the x velocity and leaves y alone. It is physically right. Build the game that way — Exercise 1.8.1 asks you to — and it dies on the spot.

Here is why, and it is worth understanding rather than just accepting. If a paddle only ever negates vx, then vy is conserved for the entire match. Nothing in the game can ever change it: walls negate it, paddles ignore it. The ball is condemned to whatever vertical rate its serve gave it, forever, and both players are reduced to standing where the ball already is. There is no way to place a shot, and therefore no skill and no game.

So the 1972 original cheats, and every version since has kept the cheat. The paddle is treated as though it were curved: where you strike it chooses the outgoing angle (Figure 6). Hit dead centre and the ball returns flat; hit near the top and it leaves steeply upward. The paddle stops being a wall and becomes a control.

Mechanically: measure how far from the paddle's centre the ball struck, as a fraction from −1 (top edge) to +1 (bottom edge). Multiply by the steepest angle we allow. That is the outgoing direction; the speed is chosen separately.

\[ \textit{offset} = \frac{y_{\text{ball}} - y_{\text{paddle centre}}}{\textit{paddle\_height}/2}, \qquad \theta = \textit{offset} \times \theta_{\max} \] offset = (ball_centre_y - paddle_centre_y) / (paddle_height / 2); theta = offset * max_angle

Then the direction is (cos θ, sin θ), pointed left or right as appropriate. That pair needs no normalising: cos²θ + sin²θ = 1 for every θ, so it is a unit vector by construction. Calling normalised() on it would buy a square root and no correctness — a small thing, but exactly the kind of small thing that fills an inner loop.

Worked example — the hit from §3.3, finished

The ball struck at y = 70 with the paddle's top edge at 60. The ball is 4 tall, so its centre is at 70 + 2 = 72. The paddle is 30 tall, so its centre is at 60 + 15 = 75. Our steepest angle is max_bounce_angle = 1.0472 rad = 60°.

  • offset = (72 − 75) / (30/2) = −3/15 = −0.2 — struck slightly above centre.
  • θ = −0.2 × 1.0472 = −0.20944 rad = −12.0°
  • direction = (cos(−12°), sin(−12°)) = (0.978148, −0.207912), then flipped to point right, which it already does.
  • Check it is unit length: 0.978148² + 0.207912² = 0.956774 + 0.043227 = 1.000001 ✓ (the last digit is float rounding).
  • New speed is 105 + 7 = 112 px/s, so velocity = 112 × (0.978148, −0.207912) = (109.5525, −23.2861).

Negative y is upward on screen, so a ball striking just above the paddle's centre leaves gently upward. That matches the intuition, and it matches what the running program reports to four decimal places.

4Implementation

The full files are in §5; this section is for understanding how they got that shape. We build from the outside in: the boundary first, then the data, then the rules.

A new directory, and what it means

Create src/game/. The header that goes in it opens with a line worth pausing on:

draw() takes a framebuffer&, so this header must be able to name the type — but it never dereferences one, never calls a method, never asks how big it is. For that, the compiler needs to know only that the name refers to a class. A forward declaration says exactly that and nothing more.

C++ in place — headers, and why an #include is expensive

An #include is a literal text paste, performed before the compiler proper ever runs. Include framebuffer.hpp here and every file that includes pong.hpp also compiles the whole of framebuffer.hpp — and colour.hpp and <vector>, which it in turn includes. The cost compounds: a header included by fifty files is compiled fifty times.

Worse than the time is the coupling. Every file that pastes in framebuffer.hpp must be rebuilt when it changes, whether or not it used anything that changed. The rule that follows: in a header, include what you use; forward declare what you merely mention. Pointers, references, and function parameters/returns you do not call through are all "merely mention". By-value members, base classes, and anything whose size or members you need are "use".

pong.cpp includes the real header, because it actually draws. That is where the cost belongs: one file, paid once. Module 5 revisits this under its proper name, physical design, and it is a large part of why big C++ codebases are fast or slow to build.

The state, the intent, and the seed

The state struct is the game. Every number that says what is currently true lives in it, and — this is the constraint that makes everything else work — nothing else does:

Plain members, no pointers, no owned resources, no virtual anything. That is not tidiness for its own sake; three concrete properties fall out of it. It can be copied with a single assignment, which is what makes previous = current in the fixed-step loop cheap enough to do every step. It can be compared field by field, which is how the determinism check in §6 works. And in Module 8 it can be written to disk in one call, because there are no pointers in it to be meaningless when read back.

The intent struct is the other half of the contract. The simulation never touches the keyboard; it is handed two numbers:

A scancode is a fact about hardware; "the left player wants to go up" is a fact about the game. Keeping the translation in main.cpp means the simulation can be driven by a keyboard, a replay file, a network packet, or a test harness without knowing the difference — and it is the seam that Module 5's action-mapping system will widen rather than cut.

Then the seed, which is where a subtle bug would otherwise live:

That is xorshift32 — three shifts, three xors, a period of 2³²−1. It is not a good generator by any serious standard, and it does not need to be: it picks serve angles. What it is is stateless apart from the word we hand it, and that is the property that matters.

SDL does provide SDL_srand, SDL_rand and SDL_randf, and they would work fine. We do not use them here for one reason: they keep their state in a hidden global. A simulation that reads a global is not a function of its inputs, which means two copies of it cannot be compared and a recorded match cannot be replayed. Putting the seed in the struct keeps state honest — it really does contain the entire game.

The shift in next_unit is also deliberate. A float has 24 bits of mantissa; taking the top 24 bits and scaling by 2⁻²⁴ hits every representable value in [0,1) exactly once with no rounding. Dividing the full 32 bits by 2³² would round — and the bits doing the rounding would be the low ones, which in xorshift are the worst ones.

The one seed that breaks it

Every operation in xorshift maps 0 to 0. Seed it with zero and it returns zero forever: every serve leaves at the identical angle and the match becomes a metronome. It is a genuinely confusing bug because the code is not wrong and nothing crashes. make_state substitutes a constant when handed 0, and that one line is worth its comment.

Three details are doing work here. teleported is cleared at the top, because the flag describes this step — a stale one left set would make the renderer refuse to interpolate for the rest of the match. Paddles move during the serve countdown, which is most of what makes the pause between points feel like a breath rather than a lockout. And the signature takes h, so there is nowhere to pass a frame duration in even by accident: Lesson 1.4's separation, enforced by the type system rather than by remembering.

Moving the ball

Walls first. Two things must happen at a bounce, and forgetting either produces a distinct, recognisable bug:

Fix only the velocity and the ball spends a step outside the court before turning round — a visible flicker past the edge. Fix only the position and the ball sticks to the wall, because next step it is still moving into it and collides again, and again. Both together: the ball turns around and the overshoot is mirrored back inside, which is also what a real bounce does — the ball does not pause at the wall waiting for the next tick.

The normals are written explicitly and the general reflect is used even though a sign flip would do. It costs nothing — reflect is constexpr and folds to a negation for these constants — and it is the version that still works when Exercise 1.8.4 tilts a wall. Note the direction of each normal: it points into the court. The ceiling at y = 0 has normal (0, +1) because +y is down.

Then the paddle, with both rules side by side so the failure can be switched on:

Look at what the naive branch sets: impact_t = 1.0f and impact_y = to.y. It has no idea when contact happened, so it assumes the end of the step — which is the honest expression of its limitation. Everything downstream is shared, so the only difference between a correct game and a broken one is which question got asked.

Note also that we test only the paddle the ball is moving toward. That halves the work, but more usefully it makes it impossible to "collide" with the paddle the ball has already gone past.

The response then applies the bounce and spends the remainder of the step:

Speed and direction chosen independently, exactly as Lesson 1.7 insisted — which is why the ball can get faster over a rally without the rally getting flatter. And the remainder is spent rather than discarded: without that last line the ball loses a fraction of a step's motion at every single bounce, which is invisible once and a measurable drag over sixty hits.

A simplification, stated plainly

We resolve one impact per step. A ball that would strike a paddle and then a wall inside the same step gets the wall on the following step instead — half a millisecond late at 60 Hz, and never noticeable. This is the ninety-percent picture; the missing ten percent is a solver that loops until the whole step is consumed, and Module 7 writes it. Exercise 1.8.3 asks you to write it early.

Three handicaps

The opponent is fifteen lines, and every one of them is a deliberate weakness. A paddle that simply matches the ball's y is unbeatable, and an unbeatable opponent is not an opponent:

  • It moves at ai_skill = 0.82 of full paddle speed, so a steep return outruns it. This is the handicap that makes the angle-choosing paddle from §3.5 into an actual weapon.
  • It only chases while the ball is heading its way, and drifts home otherwise. That drift is what gives the player an opening: the AI is out of position at the start of every return.
  • It has a 3-pixel deadzone. Without one it oscillates one pixel either side of the ball forever, which looks broken even though it is technically tracking perfectly.

That last one is worth remembering beyond Pong. Any controller that steers toward a target needs either a deadzone or damping, or it will hunt around it — the same phenomenon that makes a badly tuned thermostat click on and off all night.

Drawing, and what must not be interpolated

draw takes both states and alpha, and the interesting decision is which fields get the interpolation:

A position is a sample of something continuous, so a value between two samples is meaningful — that is the entire premise of Lesson 1.4's interpolation. A score is a discrete fact. There is no such thing as being 0.4 of the way to a point, and lerping one would put a number on screen that was never true. The general rule: interpolate quantities that were continuous before you sampled them, and nothing else.

C++ in place — std::lerp

std::lerp(a, b, t) arrived in C++20, in <cmath>. It is worth preferring over the obvious a + (b - a) * t because the standard requires two properties that expression does not have in floating point: it must return exactly b when t == 1, and it must be monotonic. The naive form can overshoot slightly near the ends. Free correctness for the same typing.

Our engine::lerp for vec2 still uses the simple form, which is a small inconsistency worth knowing about rather than hiding — for a ball position the difference is far below a pixel, and Module 2 revisits it when interpolation starts driving texture coordinates, where it matters much more.

The scores are drawn from a 3×5 bitmap font defined as ten integers. Fifteen bits per glyph, written top row first, so the literal is a picture of the digit:

That is the whole idea behind a bitmap font: a glyph is a rectangle of on/off pixels plus a rule for reading it. Real text — variable widths, kerning, anti-aliasing, an atlas, and stb_truetype — is Module 6. This is the cheapest thing that works for two scores, chosen knowingly rather than by default.

Very little, which is the point. main.cpp opens a window, owns a framebuffer, reads the keyboard into an intent, runs the loop, and uploads pixels. The rules of Pong are not in it and could not be. The one part worth reading closely is the step loop:

  3. The ball goes straight through your paddle. Not sometimes. Every time, on the opening serve, at 105 px/s — because 105 × 0.1 = 10.5 px per step against an 8 px window, exactly as §3.4 predicted.

7Common Pitfalls & Debugging

The ball sticks to a wall and vibrates — it slides along the edge
Symptom
On touching the top or bottom wall the ball stops bouncing and shudders along it, sometimes escaping after a while, sometimes not.
Cause
The velocity was reflected but the position was not corrected, so the ball is left slightly outside the wall. Next step it is still outside, the wall test fires again, and the velocity flips again — every step. The ball is being reflected many times per second, so it makes no net progress.
Fix
Mirror the position as well as the velocity: to.y = -to.y for the ceiling, and subtract twice the overshoot for the floor. The test to remember: after resolving a collision, the object must no longer be inside the thing it hit.
The ball passes through a paddle — tunnelling
Symptom
The ball ignores a paddle entirely. Reliably at low simulation rates or high speeds; intermittently in between.
Cause
A discrete overlap test being asked about a continuous path. The ball's per-step motion |vx|·h exceeds the overlap window paddle_width + ball_width, so no sampled position ever lands inside it. §3.4.
Fix
The swept test: find the fraction t of the step at which the ball's leading edge crosses the paddle's face, and check the vertical overlap at that instant. To confirm the diagnosis before fixing it, print |vx|·h beside the window width — if the first exceeds the second, it is this and not something else.
The ball streaks across the court when a point is scored — a one-frame smear
Symptom
On the frame a point is awarded, the ball appears somewhere between its old position and the centre, or seems to fly back across the court at impossible speed.
Cause
Interpolating across a teleport. previous holds the ball just off the edge of the court, current holds it at the centre, and the renderer dutifully draws a point alpha of the way between two positions that were never connected by motion. Lesson 1.4 §5.4.
Fix
Snap rather than interpolate: when a step teleports something, set previous = current immediately after it, so that frame's lerp is a no-op. Our state::teleported flag exists for exactly this and is cleared at the top of every step so it can never go stale.
Bounces gain or lose energy — the ball speeds up or dies at walls
Symptom
Every wall bounce makes the ball faster (soon a blur) or slower (soon crawling), with paddle hits behaving normally.
Cause
An unnormalised normal passed to reflect(). The formula v − 2(v·n̂)n̂ assumes |n̂| = 1; a normal of length k scales the correction by k². Length 2 quadruples it; length ½ quarters it.
Fix
Normalise the normal at its source, and check with the invariant a reflection must satisfy: length(reflect(v, n)) == length(v). That one assertion catches every instance of this bug immediately.
Every serve leaves at the same angle — the match is a metronome
Symptom
The randomised serve is not random. Every serve is identical, and the game plays the same way every time, with no crash and no warning.
Cause
The xorshift generator was seeded with 0. Every one of its operations maps 0 to 0, so the state can never leave zero — it is a fixed point, not a slow start.
Fix
Substitute a non-zero constant when a seed of 0 is supplied, as make_state does. More generally: when adopting any small PRNG, look up its degenerate states before using it. Most have at least one.
The AI paddle jitters constantly — a one-pixel buzz
Symptom
The opponent's paddle vibrates rapidly around the ball's height instead of settling.
Cause
No deadzone. The controller asks "is the ball above or below me?" every step and always gets an answer, so it overshoots by a fraction of a pixel and corrects, forever.
Fix
Return zero when the error is smaller than a few pixels. Any controller steering toward a target needs a deadzone or damping; without one it hunts.
Undefined symbols game::step / game::draw — it compiles but will not link
Symptom
Every file compiles; the link step fails naming game::step and game::draw.
Cause
src/game/pong.cpp is not in the add_executable list. The header declared the functions — which is all the compiler needs to accept the calls — but no translation unit ever defined them.
Fix
Add the file to CMakeLists.txt and reconfigure. This failure mode is the direct cost of listing sources explicitly rather than globbing them, and it is a cost worth paying: a glob would have silently included the file and silently excluded the next one, depending on when CMake last ran (Lesson 0.4).

8Exercises

1.8.1Make the paddle honest, and watch the game diewarm-up

Replace bounce_direction() with a physically correct mirror reflection: use engine::reflect with the paddle's true normal, (±1, 0). Play five points. Describe precisely what goes wrong and why — the answer should mention a quantity that is now conserved for the whole match.

Hint

Ask what can change ball_vel.y once the paddles stop touching it. Walls negate it. Paddles now ignore it. What is left?

Solution

ball_vel.y becomes an invariant of the match: nothing can alter its magnitude ever again. The ball settles into a fixed zig-zag whose vertical rate was decided by the serve, and since neither player can influence it, both are reduced to moving to wherever the ball already is. Rallies become endless and identical.

This is the useful form of the insight: a game needs the player to be able to change the state in ways the opponent must respond to. Physical accuracy that removes the only channel of influence makes a worse game, and 1972 was right.

1.8.2Give the paddle spinmedium

Real Pong players expect a moving paddle to drag the ball. Add it: track each paddle's velocity across the step and blend a fraction of it into the ball's outgoing y velocity, then renormalise so the ball's speed is unchanged — only its direction should shift.

Hint

You need the paddle's velocity, which the state does not currently store. Either add a left_vy field written by move_paddle, or record the previous y and difference it. Then in bounce_direction, add paddle_vy * k to the direction's y before normalising, with k around 0.15 to start. This is the first time normalised() is genuinely required rather than avoidable — after adding the spin the direction is no longer unit length.

Solution sketch

Storing the velocity is cleaner than differencing, because it keeps step() a function of the state alone. Clamp the total angle afterwards, or a fast paddle sweep can send the ball almost vertically and it will bounce between the walls for a long time before crossing the court. Watch out for one interaction: clamping the angle after adding spin changes the direction, so normalise after clamping, not before.

1.8.3Resolve every impact in a step, not just the firstmedium

We resolve one collision per step and let a second wait for the next one. Fix it properly: restructure advance_ball as a loop that consumes the step's time budget — find the earliest impact in the remaining time, resolve it, subtract the time used, and repeat until no impact remains or the budget is spent.

Hint

Track remaining_h rather than a fraction. Each pass computes the candidate impacts (walls and the approaching paddle), takes the one with the smallest t, advances the ball to it, applies the response, and sets remaining_h -= t * remaining_h. Cap the iterations — five is plenty — because a ball wedged in a corner can otherwise produce an unbounded loop inside one frame.

Solution sketch

The iteration cap is the part worth thinking hardest about, and it is the same reasoning as Lesson 1.4's spiral-of-death guard: an unbounded loop inside a frame is far worse than a slightly wrong answer, because the first hangs and the second is a pixel out of place. When the cap is hit, the honest move is to zero the remaining time and let the ball resume next step — and to count the occurrences so the situation is visible rather than silent.

This structure — earliest impact, resolve, repeat, capped — is what Module 7's collision solver looks like. Writing it here, with two objects and one axis, is the cheapest possible introduction to it.

1.8.4Put an angled obstacle in the middle of the courtopen

Add a fixed diagonal wall — a 45° bar near the centre — that the ball bounces off. This is where reflect()'s general form earns its keep, because no amount of negating a single component will do.

Hint

Represent the bar as a line segment with a precomputed unit normal. For the collision, the cheapest approach that stays honest is to sweep the ball's centre against the segment's line: compute the signed distance from the centre to the line before and after the step (that is dot(p − point_on_line, n̂)), and if the sign changed, the ball crossed it. Solve for t exactly as in §3.3, check the contact point lies within the segment's extent, and reflect.

Solution sketch

The signed-distance trick generalises everything in this lesson: the paddle's face test is the special case where n̂ = (±1, 0), and the wall test is where n̂ = (0, ±1). Writing the general version and then noticing the three tests collapse into one is the single most valuable outcome of this exercise.

Two things will bite. First, treating the ball as a point ignores its 4-pixel size, so it visually clips the bar — the fix is to offset the line by the ball's radius along the normal, which is the Minkowski trick from §3.4 again. Second, decide what the bar does to the rally: it makes the game much harder to read, and you may want it smaller than your first instinct says.

1.8.5Record a match and replay it exactlyopen

Prove the determinism claim rather than trusting it. Record the seed and every step's intent to a file; add a replay mode that reconstructs the state from the seed and feeds the recorded intents back in. The replayed match must be identical — same rallies, same angles, same final score.

Hint

An intent is two floats and a bool, and there is one per step, not per frame — which is exactly why the recording is small and why it only works because the simulation is stepped at a fixed rate. Write the seed as a header, then the intents. On replay, ignore the keyboard entirely and pull the next intent each step.

Solution sketch

The interesting part is what breaks it, because that is what teaches the limits of the word "deterministic". Reading the clock inside step() breaks it. Calling rand() breaks it. Storing anything the simulation depends on outside the state breaks it. If your replay diverges, the divergence point tells you exactly which line reached outside the state — and finding it that way is more instructive than any amount of reading.

Determinism here means: same binary, same machine, same seed, same intents, same step size. It is not cross-platform — floating point differs across compilers and architectures through FMA contraction, x87 excess precision, and libm implementations — and it is not even cross-step-size. Change the sim rate to 120 Hz and the replay diverges, because h is an input too. That nuance is what makes networked lockstep games hard, and it is worth meeting on a toy.

9Recap & Next — Module 1 complete

What the engine can do now that it could not before

  • It runs a game. Not a demo of a subsystem — a complete, winnable game with rules, an opponent, a score and an ending, assembled from parts built one at a time.
  • Vectors can bounce. reflect() derived from projection, valid for a surface at any angle, with its unit-normal precondition understood rather than memorised.
  • Collision is asked about intervals, not instants. The swept test, the time of impact, and the remainder of the step — the structure Module 7 scales up.
  • The boundary between engine and game exists and has never been crossed, which is what will make the Module 5 refactor a rename instead of a rewrite.
  • The simulation is a pure function of its state and its inputs, seed included — the practical form of determinism, and the thing that makes replays, tests and (much later) networking possible at all.

And one habit that outlives all of it: when you write a discrete test of a continuous thing, work out the condition under which it stops being valid and write it down. The naive overlap test was correct for every case we could reach and wrong for cases we could not. That gap between "my tests pass" and "this is correct" is where the expensive bugs live.
