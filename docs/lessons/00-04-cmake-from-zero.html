CMake From Zero

One hand-typed compiler command was fine for one file. It collapses the instant you add a second file, a third compiler, or a library like SDL. CMake lets you describe the build once and get it everywhere — and by the end of this lesson it will fetch, build, and link SDL3 for you.

1The Problem

In 0.3 you compiled hello.cpp with one command you typed by hand. That worked because the situation was the simplest one possible: one file, no dependencies, one compiler, one platform. Every one of those is about to stop being true.

The engine will grow to hundreds of source files. It must build with MSVC, GCC, and Clang, on Windows, Linux, and macOS. And it depends on SDL3 — which is itself a large C project with its own build, hundreds of its own source files, and platform-specific pieces that differ on every OS. Managing all of that with hand-typed commands is not merely tedious; it is a full-time job that no one does correctly for long.

Here is the wall, and you can hit it right now. In 0.4 we extend hello.cpp to ask SDL its version — one #include. Try the 0.3 hand-command on it:

The compiler cannot find SDL's headers, because you did not tell it where they are — and you could not have, because SDL is not installed anywhere yet. To make this build by hand you would need to: download SDL, build it (correctly, for your platform, with the right options), find where its headers and its compiled library landed, and add -I and -L and -l flags pointing at exactly those places — and then write it all down differently for the next person on the next OS.

The failure mode: the build that only works on one machine

Teams that manage this by hand end up with a build that works on precisely one person's computer, held together by paths hard-coded to their home directory and steps that live only in someone's memory. A new hire loses a day to it. A CI server loses a week. "Works on my machine" is not a joke about carelessness — it is the natural end state of a build with no single, portable description. CMake exists to be that description.

2Building Intuition

2.1 CMake generates builds; it is not a build

The most common misconception about CMake is that it compiles your code. It does not. CMake never invokes a compiler to build your program. It is a build-system generator: it reads your description and writes out the input files for some other build tool — which then does the compiling.

The analogy that makes this click: CMake is not the cook, and it is not the recipe either. It is a translator who writes the recipe out in the local kitchen's language. You hand it one description of the dish; on a Linux machine it writes instructions for make or ninja; on Windows it writes a Visual Studio solution; on a Mac it can write an Xcode project. Same dish, translated for whatever kitchen it finds. The translation is the whole value: you describe the build once, in CMake's language, and every platform gets native build files it already knows how to run.

That indirection is why CMake feels like a layer of ceremony at first. It is buying you something real, though — the alternative is writing and maintaining a Makefile and a Visual Studio project and an Xcode project by hand, keeping all three in sync forever. Figure 1 shows the shape of what CMake does.

2.2 Targets are the nouns

Old CMake — the kind you will find in most Stack Overflow answers — was written in terms of global variables: set a flag here, and it silently affects everything, everywhere, forever. That approach does not scale, and it is the source of most CMake horror stories.

Modern CMake, which is all we will write, is built on one noun: the target. A target is a thing you build — an executable, or a library. Everything a target needs to be built — its source files, the standard it compiles to, the warnings it wants, the libraries it links, the headers those libraries expose — is a property of that specific target, not a global setting. You configure targets, and targets carry their configuration with them.

This matters enormously the moment SDL enters, and it is worth stating the payoff before we get there: when our program's target links SDL's target, it inherits everything SDL says a consumer needs — its include directories, its own dependencies, its platform frameworks — automatically, because that information lives on SDL's target and travels across the link. You never type an -I path for SDL. You just say "my program uses SDL's target", and the properties flow. Hold that thought; §3.2 makes it concrete.

3The Theory

3.1 The two phases

Every CMake build has exactly two phases, and knowing which one you are in tells you what any given error means.

  1. Configure (and generate). CMake reads CMakeLists.txt, inspects your machine (which compilers exist, what they support, where things are), and writes native build files into a build directory. This is when FetchContent downloads SDL, and when a typo in your CMake logic surfaces. You run it with cmake -S . -B build.
  2. Build. The native tool CMake generated compiles and links your code. This is when a C++ syntax error or a missing symbol surfaces — the four-stage build from 0.3 §3.1, now driven by Ninja or Visual Studio instead of your fingers. You run it with cmake --build build.

Notice the split: a configure error is about your CMake description or your environment; a build error is about your C++. Same reflex as the compile-versus-link distinction from 0.3 — knowing which phase spoke to you halves the search before you read the message.

Both commands take a build directory (build/), and it is separate from your source on purpose. This is an out-of-source build: every generated file, every object file, and the entire fetched-and-built copy of SDL live under build/, touching none of your tracked source. Deleting build/ throws away nothing you wrote and forces a clean rebuild — which is why build/ is in our .gitignore and never committed.

3.2 Properties travel with targets

The keyword that makes target properties flow is the visibility marker on target_link_libraries and its siblings: PRIVATE, PUBLIC, or INTERFACE. It answers "who is this property for — me, my consumers, or both?"

KeywordApplies to the target itself?Applies to things that link it?Use for…
PRIVATEyesnoan implementation detail — our warning flags
PUBLICyesyessomething in your headers a consumer also needs
INTERFACEnoyesa header-only library's requirements

So when SDL declares its include directories PUBLIC, and we write target_link_libraries(engine_smoketest PRIVATE SDL3::SDL3), SDL's public properties flow across the link onto our target: our compiler is silently handed SDL's include path, so #include <SDL3/SDL_version.h> just works — no -I from us. Meanwhile our PRIVATE warning flags stay on our target and never touch SDL. That is the entire reason we hold ourselves to -Wall -Wextra without drowning in warnings from SDL's hundreds of C files: the flags are private to the target that wanted them. This is the modern-CMake payoff, and it is worth pausing on, because it is exactly the "third-party is not our code" rule from style §9, enforced by the build system.

3.3 FetchContent, and the pin

FetchContent is CMake's built-in way to bring another CMake project into yours. At configure time it downloads the dependency's source and adds its CMakeLists.txt to your build, so the dependency is built from source alongside your code, and its targets — like SDL3::SDL3 — become available to link. No system install, no manual download, no hunting for headers. A fresh clone of your repo builds on any machine with a compiler and CMake, and nothing else. Figure 2 traces the whole flow, including the property inheritance from §3.2.

The one non-negotiable rule is the pin. We fetch SDL at an exact release tag, release-3.4.12, never a branch name like main. The reason is reproducibility, and it is worth stating as a principle:

Pin every dependency to an immutable reference

A branch moves. If you fetched main, then your machine and a teammate's machine — cloned a week apart — would silently build different SDL source, and a bug that appears for one of you but not the other becomes impossible to reason about. A tag is immutable: release-3.4.12 is the same bytes forever. Pinning turns "it built yesterday" into a guarantee rather than a hope. When you deliberately upgrade, you change the tag in one line, on purpose, and everyone moves together.

Worked example — decoding the version integer

SDL reports its version as a single int, packed by the macro SDL_VERSIONNUM(major, minor, patch) = major·1000000 + minor·1000 + patch. For the 3.4.12 we pinned:

  • SDL_VERSIONNUM(3, 4, 12) = 3·1000000 + 4·1000 + 12 = 3000000 + 4000 + 12 = 3004012.

So SDL_GetVersion() returns 3004012. The extraction macros unpack it with integer division and remainder:

  • major = 3004012 / 1000000 = 3  (integer division drops the remainder)
  • minor = (3004012 / 1000) % 1000 = 3004 % 1000 = 4
  • micro = 3004012 % 1000 = 12

Giving 3.4.12 — which is exactly what our program prints below. Packing three small numbers into one integer with fixed decimal "slots" is a trick you will see again; it works only because each field is guaranteed under 1000, and SDL enforces that. (This is why the minor extraction needs both the divide and the % 1000: the divide shifts 12 out, the remainder shifts 3 out.)

4Implementation: build up the CMakeLists

We build the CMakeLists.txt in two parts. First, the minimum that builds hello.cpp exactly as 0.3 left it — proving CMake reproduces your hand-command. Then we add SDL.

4.1 Part 1 — build hello.cpp with CMake

Create CMakeLists.txt next to hello.cpp. Every CMake file opens with two lines: the minimum CMake version it needs, and the project's name. We require 3.24 because it is new enough for the modern behaviour we rely on — and demanding it means an ancient CMake fails loudly and early, rather than misbehaving subtly:

Next, the C++ standard — set once, for every target in the project. Three settings work together: pick C++20, make it required (an error, not a silent downgrade, if the compiler cannot do it), and turn off compiler extensions so we get portable -std=c++20 rather than -std=gnu++20:

What those three settings actually emit

On GCC/Clang they produce -std=c++20; on MSVC, /std:c++20 — the exact flags you typed by hand in 0.3, now chosen per-compiler for you. Note they do not add MSVC's /Zc:__cplusplus; we add that ourselves below, because hello.cpp's guard reads _MSVC_LANG but SDL reads __cplusplus (the trap from 0.3 §3.3).

Finally, declare the executable and, for now, nothing else. add_executable is the line that creates a target: a thing named engine_smoketest, built from hello.cpp:

With just those lines — and hello.cpp reverted to its dependency-free 0.3 form — cmake -S . -B build then cmake --build build would produce a working engine_smoketest. That is CMake reproducing your hand-command, portably. Now let us give it the job the hand-command could not do.

4.2 Part 2 — fetch and link SDL3

Between the standard settings and the executable, we bring in SDL. First, one SDL build option — we only need the library, not SDL's own test helpers, and skipping them shaves real time off a from-source build. SDL reads its options before it is configured, so this must come first:

FetchContent_Declare describes where the dependency is and which exact commit to take; FetchContent_MakeAvailable is what actually downloads it and folds its build into ours. After that line, SDL's targets exist and we can link them. GIT_SHALLOW TRUE fetches only the pinned tag's snapshot rather than SDL's entire history — faster, and we do not need the history.

Now link SDL onto our target, and apply our warnings. Both use the target-property model from §3.2:

SDL3::SDL3 is the alias SDL exports for "the SDL library, however it was built". Linking it hands our target SDL's headers and platform frameworks automatically — the property flow from §3.2 in action. The warning flags are PRIVATE, so they land on our target and never on SDL's C files. That single word is the difference between a clean build and a wall of warnings you did not write and cannot fix.

And hello.cpp gains the SDL query it needs — a version print, using the API from §3.3:

⚠ Verify

The SDL3 version API used here is verified against SDL3/SDL_version.h at release-3.4.12: int SDL_GetVersion(void) returns the packed integer, and SDL_VERSIONNUM_MAJOR / _MINOR / _MICRO unpack it. The compile-time SDL_MAJOR_VERSION family gives the version of the headers you built against. If you pin a different SDL tag, re-verify against that tag's header — the API is stable across SDL3, but this course does not ask you to take that on faith.

6Build & Run

The two-phase incantation is the same on every platform:

-S . is the source directory (where CMakeLists.txt is); -B build is the build directory to create. Then run it:

The first configure is slow — that is not a hang

The very first cmake -S . -B build downloads SDL and the first cmake --build build compiles it from source — SDL is a large C library, so this takes a few minutes and prints a great many lines. This happens once; afterwards SDL is cached under build/_deps/ and only your own code rebuilds, in seconds. Do not Ctrl-C it thinking it has frozen.

What you should see

After the build finishes, running the program prints the toolchain report from 0.3 plus two new SDL lines:

Checkpoint

Success is the two SDL lines both reading 3.4.12 — the tag you pinned. That they match is expected here: FetchContent built SDL from the same source your headers came from, so compile-time and runtime versions are identical. They would only diverge if you linked a system-installed SDL of a different version, which we deliberately never do. If you got this far, CMake fetched, built, and linked a real dependency, and your program ran against it — the entire build pipeline, working, on your machine.

7Common Pitfalls & Debugging

CMake refuses to configure — "CMake 3.24 or higher is required"
Symptom
Configure fails immediately, complaining about the CMake version.
Cause
Your CMake is older than our cmake_minimum_required floor. This is the floor working as intended — an old CMake would misbehave in subtle ways, so we reject it up front.
Fix
Install a current CMake. Do not lower the cmake_minimum_required line to match an old CMake — that trades a clear failure now for a mysterious one later. On Windows, the Visual Studio installer ships a recent CMake; on macOS, brew install cmake; on Linux, prefer the official CMake over an old distro package if yours is ancient.
SDL3::SDL3 not found — "Target SDL3::SDL3 not found" at link
Symptom
Configure or the link step reports that the target SDL3::SDL3 does not exist.
Cause
Either FetchContent_MakeAvailable(SDL3) is missing or comes after the target_link_libraries that uses it, or the fetch itself failed (see the next pitfall). SDL's targets only exist after MakeAvailable has run.
Fix
Ensure the fetch block runs before you link the target, and that MakeAvailable completed without error. Read the configure output from the top — the real failure is usually earlier than the "target not found" line.
The fetch fails — git or network errors during configure
Symptom
Configure dies partway with a git clone error, a timeout, or a proxy/SSL complaint.
Cause
FetchContent runs git clone at configure time, so it needs git installed and network access to GitHub. Corporate proxies, offline machines, and a missing git all break it.
Fix
Confirm git --version works and you can reach GitHub. Behind a proxy, configure git's proxy settings. Fully offline, the alternative is a vendored copy of SDL — the submodule approach in the sidebar below — which needs the network only once, when you first add it.
Windows: the exe is not in build/ — "cannot find engine_smoketest.exe"
Symptom
The build succeeds, but the executable is not where you expected on Windows.
Cause
Visual Studio is a multi-config generator: it puts binaries in a per-configuration subfolder — build\Debug\, build\Release\ — not directly in build\. Single-config generators (Ninja, Make) do put it in build/, which is why Linux/macOS differ.
Fix
Run .\build\Debug\engine_smoketest.exe. To choose a configuration explicitly, build with cmake --build build --config Release. We return to Debug-vs-Release properly in the profiling lessons.
Warnings pour out of SDL — a wall of warnings you did not write
Symptom
The build drowns in warnings from files under build/_deps/sdl3-src/.
Cause
You applied warning flags globally (e.g. via CMAKE_CXX_FLAGS or an add_compile_options at the top) instead of PRIVATE on your own target. Global flags hit SDL too.
Fix
Use target_compile_options(engine_smoketest PRIVATE ...), exactly as the listing does. Warnings belong to the target that asked for them. This is the §3.2 lesson biting back if ignored.

8Exercises

0.4.1Rebuild is instantwarm-up

After your first successful build, change one printf string in hello.cpp and run cmake --build build again. Time it. Why is it a hundred times faster than the first build?

Solution

The second build takes a second or two, versus minutes. SDL is already compiled and cached under build/_deps/, and the native build tool tracks dependencies: only hello.cpp changed, so only hello.cpp recompiles and the program re-links. Nothing about SDL is touched. This incremental behaviour is a large part of why we tolerate CMake's up-front ceremony — you pay the full cost once.

0.4.2Change the pin, watch the versionguided

Change GIT_TAG release-3.4.12 to an older SDL release tag — for instance release-3.2.30 — then delete build/ and configure and build again. What do the two SDL lines print now, and why did you have to delete build/?

Hint

The fetched source is cached. FetchContent will not re-clone a different tag into a build tree that already has the old one.

Solution

Both SDL lines now print 3.2.30 — you built against, and linked, a different SDL. You had to delete build/ because the old SDL source was already cached under build/_deps/, and a plain reconfigure will not swap it out. This is the pin from §3.3 made tangible: the tag, and only the tag, decides which SDL you get — and changing it is a deliberate one-line act, not an accident of when you happened to clone.

0.4.3Prove the two phases are separateguided

Introduce a deliberate CMake error (misspell add_executable as add_exectuable) and run configure. Then fix it, and introduce a deliberate C++ error (delete a semicolon in hello.cpp) and run build. Which phase caught each, and how did the error messages differ?

Solution

The misspelled command fails during configure — CMake reports "Unknown CMake command", and no compiler ever ran. The missing semicolon fails during build — the compiler reports a syntax error at a line in hello.cpp, exactly as in 0.3, because CMake configured fine and handed the job to the native tool. This is §3.1 made concrete: configure errors are about your build description; build errors are about your code. Knowing which phase failed tells you which file to open.

0.4.4The submodule alternativeopen

The master plan says to teach both dependency-acquisition methods once, then standardize. The other method is a git submodule: instead of CMake cloning SDL at configure time, you commit a reference to SDL's repo inside yours (git submodule add under third_party/) and add_subdirectory(third_party/SDL) in CMake. Research both, and write down two concrete advantages of each. Which would you choose for this course, and why?

Solution sketch

Submodule advantages: the dependency's source is vendored, so builds work fully offline after the first checkout; and the exact commit is recorded in your git history, auditable alongside your code. FetchContent advantages: a fresh clone needs no git submodule update --init ritual — the single most common way submodule projects fail to build for a newcomer — and the dependency is not duplicated into your working tree.

We standardize on FetchContent for this course, precisely because the "forgot to init the submodule" failure is so common and so confusing for someone new, and because a bare git clone of the repo builds with nothing extra. The tradeoff we accept is the one-time source build and the network requirement — both acceptable, and both mitigated (caching handles the first, the submodule fallback handles the second). If you were shipping in a locked-down offline environment, the submodule's guarantees might tip the balance the other way — which is the honest answer: the choice follows the constraints, as it always does.

9Recap & Next

What the engine can do now that it could not before

  • It has a portable build. One CMakeLists.txt describes the whole thing, and CMake turns it into native builds for Windows, Linux, and macOS.
  • It acquires its own dependencies. A bare git clone plus cmake -S . -B build fetches, builds, and links SDL3 — pinned to an exact, reproducible version — with no manual install.
  • You think in targets. Standard, warnings, and links are properties of the engine_smoketest target, and SDL's properties flow to it across the link. No global state, no hand-managed include paths.

The idea to carry forward: CMake generates; the native tool builds. Two phases, two kinds of error, one portable description. When a build breaks later, the first question is "configure or build?" — and you already know how to tell.
