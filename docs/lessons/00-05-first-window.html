1The Problem

In 0.4, SDL built and linked, and our program proved it by printing a version number. That is a fine milestone for a build, but it is not yet an engine. An engine, from 0.1, is a thing with a heartbeat: it owns a loop and produces frames. We have no window and no loop. It is time to grow both.

Opening a window turns out to be the easy half. The hard half — the half that separates a program from an application — is the loop, and the reason is a failure mode you have certainly seen, probably cursed at, and maybe never understood:

The failure mode: the window that will not die

You have met an unresponsive application: the window is there, but it will not move, will not close, and the OS greys it out and offers to kill it — the spinning beach ball on macOS, "Not Responding" in the Windows title bar. It looks like a crash. It usually is not.

What has actually happened is that the application stopped talking to the operating system. The OS is constantly sending your window messages — "the user clicked", "you were resized", "please close", "are you still alive?" — and it expects you to collect them, promptly, forever. An app that opens a window and then does anything else — a long computation, or simply nothing — leaves those messages piling up unanswered, and after a moment the OS concludes the app is hung and marks it dead. The window is not frozen because the app crashed; it is frozen because the app went quiet.

So the event loop is not boilerplate you copy and forget. It is the conversation that keeps your window alive, and understanding it is understanding why every interactive program on every OS is shaped the way it is. Let us build the intuition before the code.

2Building Intuition

2.1 The OS owns the window; you rent it

A window is not really yours. It is an operating-system object — the OS draws its border, its title bar, its close button; the OS decides where it sits and when it is covered by another window. What SDL hands you is a handle to that OS object: a way to refer to it and ask things of it. You are a tenant, not the owner, and the landlord has rules.

The chief rule is the one from §1: the OS will send you a steady stream of messages about your window, and you must collect them. Think of it as a mailbox the OS fills and you must empty. If you stop emptying it, mail piles up, and eventually the landlord assumes you have moved out. Emptying the mailbox — pumping events — is the rent you pay to keep the window responsive, and it comes due many times a second, every second, for as long as the window lives.

This is why an engine must have a loop, and why the loop must run fast enough to check the mailbox often. It also quietly explains a rule from 0.1: the frame budget is not only about drawing pretty pictures on time — it is also about answering the OS on time. Miss the budget badly enough and the window stops responding, picture or no.

3The Theory

3.1 Initialise, and owned resources

SDL_Init(SDL_INIT_VIDEO) starts up SDL's video subsystem (and, as a bonus, the event subsystem it depends on). Everything after it depends on it having succeeded, which is why it is the first thing we check.

SDL_CreateWindow and SDL_CreateRenderer each return a pointer to an object SDL created and now lets you use: an SDL_Window* and an SDL_Renderer*. These are owned resources — SDL allocated them for you, and you are responsible for handing them back, with SDL_DestroyWindow and SDL_DestroyRenderer, when you are done. Forget, and you leak: the OS resources stay reserved until your process dies.

C++ in place — a pointer that means "not null, and mine to free"

SDL is a C library, so it speaks in raw pointers. An SDL_Window* here is an owning pointer by convention: SDL made it, and this code must free it. That directly contradicts our own style rule that "a raw pointer never owns" — and that tension is the point. Manually pairing every Create with a Destroy, on every exit path, is exactly the error-prone bookkeeping that RAII exists to abolish. In Module 5 we wrap these in small types whose destructors call Destroy automatically. For now we do it by hand, so that when RAII arrives you will feel precisely what it is saving you from — watch how the error paths below have to repeat the cleanup.

3.2 The event queue

The mailbox from §2.1 is, concretely, a queue: the OS (through SDL) appends events to the back as they happen, and SDL_PollEvent removes one from the front each time you call it. You call it in a tight inner loop — while (SDL_PollEvent(&event)) — to drain every pending event each frame, because more than one can arrive between frames: a key press, a mouse move, and a resize might all be waiting.

Each event carries a type, and we branch on it. Today we care about two: SDL_EVENT_QUIT (the user asked to close the window) and SDL_EVENT_KEY_DOWN (a key went down). Everything else we let fall through, ignored, for now — but crucially, polled. Draining an event you do not handle still empties the mailbox; that is what keeps the window responsive even before we act on most messages.

Worked example — tracing two ways to quit

The user clicks the close button. The OS posts one event; on the next frame:

  • SDL_PollEvent returns true, filling event with type == SDL_EVENT_QUIT.
  • Our first branch matches → running = false.
  • The inner while polls again, gets false (queue empty), and ends.
  • We draw one last frame, the outer while (running) test now fails, and control falls to beat 4: destroy renderer, destroy window, SDL_Quit, return 0.

The user presses Esc. Same shape, different event: type == SDL_EVENT_KEY_DOWN, and event.key.key == SDLK_ESCAPE, so the second branch fires and sets running = false. Both routes converge on the same clean shutdown — which is exactly what we verified actually happens before publishing this lesson.

3.3 SDL3's honest return values, and the SDL2 traps

SDL3 is a major revision, and if you have ever written SDL2 — or copy a snippet that was — a few changes will bite. This course uses SDL3 only, and it is worth naming the traps once, because they are exactly the kind of thing that compiles in your head and fails on the machine:

ConcernSDL2 (do not use)SDL3 (verified against 3.4.12)
Init resultSDL_Init(...) == 0 meant successbool SDL_Init(...) — true is success. Test if (!SDL_Init(...)).
Poll resultint SDL_PollEventbool SDL_PollEvent
Keyboard keyevent.key.keysym.symevent.key.key — the nesting is gone
Window creationSDL_CreateWindow(title, x, y, w, h, flags)SDL_CreateWindow(title, w, h, flags) — no x/y
Renderer creationSDL_CreateRenderer(win, index, flags)SDL_CreateRenderer(win, name) — nullptr for default

The pattern in that table is that SDL3 made its API honest: functions that can fail return a plain bool you check directly, and the confusing nested structs and index-based selectors are gone. Whenever a function returns false or a null pointer, the human-readable reason is waiting in SDL_GetError() — so every failure check in our code logs it. That habit costs a line now and saves an hour later, when a window silently fails to create on someone else's machine and the error string tells you exactly why.

⚠ Verify

Every SDL3 signature and field used in this lesson was verified against the headers at release-3.4.12 — the tag we pin in 0.4: bool SDL_Init(SDL_InitFlags), SDL_CreateWindow(title, w, h, flags), SDL_CreateRenderer(window, name), bool SDL_PollEvent(SDL_Event*), and event.key.key. If you pin a newer SDL, the official SDL2→SDL3 migration guide is the authority on any difference — this course does not ask you to trust a signature you cannot check.

4Implementation: src/main.cpp

This is the first file of the real engine, and it earns a real home: src/main.cpp. The throwaway hello.cpp at the repo root is retired — its job was to prove the toolchain, and it is done. From here, engine code lives under src/, the library-shaped layout that the Module 5 refactor will grow into.

We include the umbrella header and — separately — SDL's entry point. This is a genuine SDL3 gotcha worth stating plainly:

<SDL3/SDL.h> pulls in almost all of SDL, but not <SDL3/SDL_main.h> — that one is special, and you include it exactly once, in the file with main. It lets SDL adapt your main to each platform's real entry point (a plain main on Linux/macOS, but the machinery behind WinMain on Windows), so the same source runs everywhere. Skip it and your program may fail to start on Windows with a baffling linker error.

7Common Pitfalls & Debugging

The window opens then instantly closes — a flash and it's gone
Symptom
The window appears for a split second and the program exits.
Cause
Almost always a missing or broken loop: no while (running), so main runs straight to cleanup and returns. A subtler version: setting running = false unconditionally, or handling a key-up as a quit so the very first input closes it.
Fix
Ensure the outer while (running) wraps the draw, and that running only flips on QUIT or Esc-down. If it still flashes, this is a job for the debugger — which is exactly what 0.6 is about.
SDL2 code that won't compile — 'keysym' is not a member; too many arguments
Symptom
Errors like "no member named keysym", or "too many arguments to SDL_CreateWindow", or comparing SDL_Init's result to 0 behaving backwards.
Cause
You (or a tutorial you copied) wrote SDL2. SDL3 changed these: no event.key.keysym.sym (it is event.key.key), no x/y in SDL_CreateWindow, and bool returns instead of int. See the table in §3.3.
Fix
Use the SDL3 forms. When in doubt, the migration guide lists every change. The compiler is your friend here — an SDL2 idiom that fails to compile is a bug caught early, not a runtime surprise.
Windows: link error about WinMain — unresolved external symbol WinMain
Symptom
On Windows, the build compiles but fails to link, complaining about WinMain or the entry point.
Cause
You did not #include <SDL3/SDL_main.h>. On Windows a GUI app's real entry point is WinMain, not main; SDL_main.h provides the shim that bridges them. <SDL3/SDL.h> does not include it for you.
Fix
Add #include <SDL3/SDL_main.h> in the file with main, exactly once, as the listing does.
The window is unresponsive — beach ball / "Not Responding"
Symptom
The window shows but will not move or close, and the OS marks it hung.
Cause
You are not draining events — no SDL_PollEvent loop, or something blocks inside the frame (a sleep, an infinite inner loop) so the poll never runs. The mailbox from §3.2 overflows and the OS gives up on you.
Fix
Make sure every frame runs the while (SDL_PollEvent(&event)) loop, and that nothing blocks the frame. Responsiveness is a property of a loop that keeps moving.
macOS: nothing happens, or an assertion about the main thread — video must init on the main thread
Symptom
On macOS, the window never appears, or SDL complains about the thread.
Cause
macOS requires window and event handling on the main thread. Calling SDL_Init(SDL_INIT_VIDEO) or creating the window from another thread violates that. For us this cannot happen — everything is in main — but it is a real constraint the moment you add threads (Module 8).
Fix
Keep window creation and the event loop on the main thread. When we add a job system later, worker threads do computation, never windowing.

8Exercises

0.5.1Make it yourswarm-up

Change the window title, its size, and the clear colour. Rebuild and confirm each change shows. What happens to the aspect ratio of the coloured area if you make the window very wide?

Solution

Title and size are the arguments to SDL_CreateWindow; the colour is the SDL_SetRenderDrawColor RGBA. The clear fills whatever the window's current size is, so a very wide window is simply a wide slab of colour — the renderer has no notion of a fixed aspect ratio yet. Aspect-ratio handling is a rendering concern we meet properly once there is something in the window to distort (Module 2's projection, and the viewport).

0.5.2React to a resizeguided

Add a branch that, on SDL_EVENT_WINDOW_RESIZED, logs the new size. The new dimensions live in the event — inspect the SDL_WindowEvent members (or the wiki) to find where. Confirm you get a line each time you drag the window's edge.

Hint

A window event's payload is in event.window. For a resize, the new width and height arrive as its two integer data fields. ⚠ VERIFY their exact names (data1/data2) against SDL3/SDL_events.h for your pinned tag before relying on them — this is the header-reading skill 0.6 makes routine.

Solution sketch

Add else if (event.type == SDL_EVENT_WINDOW_RESIZED) SDL_Log("resized to %dx%d", event.window.data1, event.window.data2);. You will see a burst of lines as you drag — resize events fire continuously, which is your first hint that some events are cheap and frequent while others are rare, a distinction that matters for performance later.

0.5.3A pulsing backgroundguided

Make the clear colour change over time — a slow pulse. SDL_GetTicks() returns milliseconds since start; feed it through a sine to drive one colour channel. Why does the pulse look smooth even though your loop has no timing control yet?

Solution sketch

Something like Uint8 g = (Uint8)(128 + 127 * SDL_sin(SDL_GetTicks() * 0.002)); then use g as the green channel. It looks smooth because the colour is a function of wall-clock time, not of frame count — so however fast or unevenly the loop runs, the pulse tracks real seconds. That is a quiet preview of the deepest idea in Module 1.4: tie change to time, not to frames, and motion stays correct regardless of frame rate. You have just done, by accident, the right thing.

0.5.4Feel the leakopen

Delete the three cleanup lines in beat 4 and rebuild. The program still runs and closes — seemingly fine. So why is removing them a bug? What tool would prove it, and how does this motivate the RAII wrappers promised for Module 5?

Solution sketch

It "works" because the OS reclaims all of a process's resources when it exits, so a leak at the very end of main is invisible. But that is luck, not correctness: the same forgotten Destroy inside a function that runs thousands of times — creating and dropping windows, textures, or buffers — leaks steadily until the program dies. A leak checker (Valgrind, ASan's leak detector, or Visual Studio's diagnostics) would flag the un-freed SDL_Window. The lesson: manual pairing of create/destroy is correct only if you never forget, on any path — and humans forget. RAII makes the destructor remember for you, which is why Module 5 wraps every one of these. You just felt the problem; that solution will feel earned.

9Recap & Next

What the engine can do now that it could not before

  • It has a window and a heartbeat. The engine opens a real OS window, runs an event loop, responds to input, and shuts down cleanly.
  • It has the skeleton. Init → create → loop → destroy is the shape of every engine, and beat 3 — the loop — is where all future work lives.
  • It has a real home. src/main.cpp is the first engine source file; hello.cpp is retired. The library-shaped layout begins here.

The idea to carry forward: a window stays alive only as long as you answer the OS. The event loop is not ceremony — it is the conversation that keeps the window from freezing, and it is the reason engines are loop-shaped.
