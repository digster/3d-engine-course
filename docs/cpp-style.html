C++ Style Guide

The rules this codebase obeys, and the reasoning behind each. A style guide's job is not to be right in the abstract — it is to be consistent, so that reading the tenth file costs less than reading the first.

The one rule behind all the others

Optimise for the reader. You will read this code far more often than you write it, and in six months you are the reader — with none of today's context. Where a rule below seems fussy, it is buying clarity at that future moment.

1Language and dialect

StandardC++20, no compiler extensions
CompilersMSVC, Clang, GCC — all three, all the time
ExceptionsOff in the engine core (§5)
RTTIOff in the engine core
Warnings-Wall -Wextra / /W4, all fixed (§9)

C++20 buys us four things we actually use: concepts (readable template constraints instead of SFINAE), std::span (the honest way to pass a non-owning view of a contiguous range), designated initializers (which make SDL_GPU's large create-info structs legible), and constexpr reaching far enough to do real work at compile time.

Modules are not used. The tooling is still uneven across the three compilers we support, and a course that fails to build on someone's machine has failed. Header/source it is — and translation units get taught properly instead of hidden.

2Naming

We follow the standard library's conventions rather than inventing our own. snake_case everywhere for types and functions means our code and std:: read as one language, and there is never a moment's thought about which casing a given name wants.

The trailing underscore on private members exists to kill the constructor shadowing problem without this-> noise: width_(width) is unambiguous where width(width) is a coin flip.

Macros are always prefixed ENG_. Macros ignore namespaces — that is their whole nature — so the prefix is the only namespace they get. We use exactly two families: assertions and platform detection.

C++ in place — scoped enums

Always enum class, never plain enum. A plain enum leaks its enumerators into the surrounding scope and implicitly converts to int, which means cull_mode::back and fill_mode::line could silently compare equal. Scoped enums cost one :: and remove an entire category of bug.

Names carry units

timeout is a bad name; timeout_seconds is a good one. Wherever a number has a unit, the name says so. This is the same discipline as Conventions §8, enforced at the identifier level, and it is nearly free.

3Files, headers, and include order

  • .hpp for headers, .cpp for sources. Never .h for C++.
  • #pragma once, not include guards. Supported by all three compilers, and impossible to typo.
  • One primary type per header, and the file is named after it.
  • A header must be self-sufficient: including it first, alone, must compile.

Include order

Ordered most-local to most-global. The reason is not aesthetic: putting a file's own header first means that if that header forgot an include, its own translation unit fails immediately, rather than silently working because some other header happened to include the missing piece first.

Project includes use "quotes" and paths relative to the include root; everything external uses <angles>. So "gfx/vertex_buffer.hpp", never "../gfx/vertex_buffer.hpp" — relative-to-file paths break the moment a file moves.

C++ in place — headers and translation units

#include is textual paste, nothing cleverer. Each .cpp plus everything it pastes in becomes one translation unit, compiled alone, then joined by the linker. This explains almost every confusing C++ build error you will meet: "undefined reference" means the linker never found a definition; "redefinition" means it found two. Lesson 0.4 makes this concrete.

Public vs private headers

After the Module 5 refactor the boundary is physical, and enforced by where a file lives:

LocationMeaning
engine/include/engine/…Public API. Demos may include these. Changing one is a breaking change.
engine/src/…Private. Free to change without telling anyone.

A public header may only include other public headers. If a public header needs a private type, the design is wrong — not the rule.

4Ownership and RAII

Every resource has exactly one owner, and that owner is an object whose destructor releases it. This is the single most important rule in the guide.

IntentUseNever
Sole ownershipa value, or std::unique_ptr<T>owning raw pointer
Shared ownershipstd::shared_ptr<T> — rare, and justified in a commentas a default
Non-owning referenceT&, or T* if nullableshared_ptr "just in case"
Non-owning rangestd::span<T>T* + separate count
Non-owning stringstd::string_viewconst char* + length
Engine resourcea handle (Module 5)pointer into a vector that may reallocate

A raw pointer never owns. If you see T* in this codebase it means "maybe null, definitely not mine, and it will outlive this use." That single convention makes delete outside a destructor a code smell visible at a glance.

C++ in place — RAII

Resource Acquisition Is Initialization: tie a resource's lifetime to an object's. Acquire in the constructor, release in the destructor, and the compiler emits the cleanup on every exit path — early return, exception, or the end of the block. It is the reason C++ can be safe without a garbage collector, and it is why we almost never write cleanup code.

The corollary is the rule of zero: if every member manages itself, your class needs no destructor, no copy constructor, and no assignment operator. Most classes here declare none of the five. The best special member function is the one you did not write.

Move semantics

Engine resources — GPU buffers, textures, file handles — are movable but not copyable. Copying a 200 MB texture because a function took it by value is a bug you should not be able to write, so we delete the copy operations rather than trusting discipline.

Move operations are noexcept. This is not decoration: std::vector will silently copy your elements when it grows if their move constructor is not noexcept, because it cannot otherwise guarantee the strong exception guarantee. A missing noexcept is a silent performance cliff.

5Errors without exceptions

The engine core is built with exceptions and RTTI disabled. This is a real tradeoff with real costs, and it deserves an honest accounting rather than an assertion that it is obviously correct — because it is not obviously correct.

What we gainWhat we give up
Predictable control flow — every failure path is visible in the signature. No hidden unwinding on a frame budget. Smaller binaries, no exception tables. Interop with C APIs (all of SDL) that cannot propagate exceptions anyway. The console and embedded platforms where this is often mandatory. Constructors cannot fail. That is the big one, and it forces the two-phase create() pattern below — which is genuinely uglier than RAII with a throwing constructor. Errors must be checked by hand, and [[nodiscard]] becomes load-bearing rather than advisory. Some of the standard library becomes off-limits.

Honest summary: for a general-purpose application, exceptions are usually the better default, and plenty of excellent engineers would argue for them here. We disable them because engines live in the world of frame budgets, C interop, and platforms that ban them — and because being explicit about failure is a discipline worth practising while you learn. Module 5.3 makes this argument in full.

The rules

  • Programmer error → ENG_ASSERT. A precondition violation is a bug; crash loudly and immediately, in debug.
  • Expected runtime failure (file missing, shader won't compile, device lost) → return a value that says so.
  • Impossible states → make them unrepresentable in the type, so there is nothing to check.

[[nodiscard]] goes on every function returning a status or a value whose loss is a bug. Without exceptions, an ignored return is a swallowed error — so we let the compiler be the reviewer who never gets tired.

6const-correctness and interfaces

  • const on every member function that does not mutate. Not optional.
  • Parameters: take by value for cheap types (int, float, vec3, handles), const& for expensive ones, by value + std::move when you intend to keep it.
  • Prefer constexpr wherever the compiler will accept it.
  • No output parameters. Return a value, or a struct of values — that is what structured bindings are for.
C++ in place — const& vs by value

A vec3 is 12 bytes and fits in registers; const vec3& is a pointer the compiler may have to dereference, and can actively hurt. The rule of thumb: pass by value up to roughly two pointers' worth of trivially-copyable data, and const& beyond that. So vec3 by value, const mat4& by reference (64 bytes), const std::vector& always by reference.

7Comments that teach

Comments explain why, and the non-obvious how. Never the what. The code already says what it does; if it does not, fix the code rather than annotating it.

BannedWanted
i++; // increment i The constraint, the reason, the thing that bit you
// loop over lights // Lights are sorted by distance so the shader can early-out.
// set the flag // SDL_GPU zero-inits cull_mode to NONE, so we must be explicit or
// we silently pay for back faces. See conventions.html §7.

Every public entity gets a doc comment using ///: what it does, what it costs the caller, what it requires, and what it returns. Preconditions get stated explicitly, because without exceptions the caller is the error handler.

Comments that explain the course rather than the code belong in the lesson prose, not the source. The one exception: a comment marking a decision the reader would otherwise assume is a mistake — like the duplicated CSS in our lesson template — earns its place, because it prevents someone "fixing" a deliberate choice.

8Formatting

Formatting is not worth arguing about, so we do not: .clang-format decides, and the answer to every formatting question is "run the formatter". The settings below exist only so the file is not a black box.

Indent4 spaces, never tabs
Line length100 columns
BracesAllman for types and functions; K&R for control flow
Braces on ifAlways, even for one statement
Pointer/referenceBound to the type: int* p, const vec3& v
using namespaceNever at file scope in a header. Function scope in a .cpp is fine.

Braces on every if is the one formatting rule with a bug-prevention argument rather than a taste argument, so it survives the "do not argue about formatting" principle. goto fail; is the famous example, and it cost real people real security.

9Warnings

-Wall -Wextra on Clang and GCC, /W4 on MSVC. All warnings are fixed, not suppressed.

A warning left standing is a broken window: once one is normal, the next is invisible, and the one that mattered scrolls past unread. The compiler is the most patient reviewer you will ever have, and it works for free.

We do not set -Werror in normal development builds — it turns an unrelated compiler upgrade into a wall you cannot get past to do today's work. CI turns it on. That way warnings cannot be ignored, but they also cannot block you mid-thought.

Third-party headers are not our code

SDL, ImGui, and stb are included as system headers (SYSTEM in CMake's target_include_directories) so their warnings never reach our build. We hold ourselves to /W4; holding other people's libraries to it produces noise we cannot act on — which is exactly the broken window we are trying to avoid.

The guide in nine lines

  • C++20, no extensions, three compilers, all warnings fixed.
  • snake_case like the standard library; private members get a trailing underscore.
  • .hpp + #pragma once; own header first; headers are self-sufficient.
  • One owner per resource, and it is an object. Raw pointers never own.
  • Engine resources are movable, not copyable; moves are noexcept.
  • No exceptions in core: assert on bugs, return values on expected failures, [[nodiscard]] everywhere it matters.
  • const by default; no output parameters.
  • Comments explain why. Doc comments on every public entity.
  • The formatter decides formatting. Do not argue with it.
