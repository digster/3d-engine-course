Conventions

The handful of arbitrary-but-binding choices this course makes about direction, order, and range. Convention mismatches are the single largest source of silent rendering bugs — a wrong choice does not crash, it just quietly draws the mirror image.

How to use this page

Everything here is decided, not discovered — except the parts marked fixed by SDL_GPU, which are facts we must obey. Where a choice was genuinely open, this page says why we chose as we did, because a convention you understand is one you can debug.

1Why conventions deserve their own page

Most graphics bugs announce themselves. You get a crash, a validation error, a black screen. Convention bugs do not. They produce a picture that is almost right — mirrored, upside-down, inside-out, or depth-sorted backwards — and every individual piece of code looks correct in isolation. The bug lives in the disagreement between two files, not inside either one.

The cure is to decide once, write it down, and never deviate. That is this page. If a lesson ever contradicts it, the lesson is wrong.

2World space: right-handed, Y-up, −Z forward

Point your right hand's index finger along +x and your middle finger along +y; your thumb now points along +z, out of the screen toward you. That is a right-handed system, and it is what this course uses everywhere in model, world, and view space. Figure 1 is the picture to keep in your head.

Worked example — proving the handedness by cross product

The cross product of the first two basis vectors must give the third. With x = (1, 0, 0) and y = (0, 1, 0):

  • (x × y).x = x.y·y.z − x.z·y.y = 0·0 − 0·1 = 0
  • (x × y).y = x.z·y.x − x.x·y.z = 0·0 − 1·0 = 0
  • (x × y).z = x.x·y.y − x.y·y.x = 1·1 − 0·0 = 1

So x × y = (0, 0, 1) = +z. Right-handed, confirmed. In a left-handed system the same arithmetic would have to yield −z, which is why you cannot mix the two without noticing — eventually.

Why right-handed, when SDL_GPU's clip space is left-handed?

This is a fair objection and worth answering directly. Two reasons decided it:

  • glTF 2.0 is right-handed, Y-up. Module 6 loads glTF scenes. Matching it means zero axis conversion on import — and axis-conversion code is a permanent tax you pay on every asset, forever.
  • Every reference this course sends you to is right-handed. Scratchapixel, Real-Time Rendering, LearnOpenGL. When you read a derivation there and type it here, it should just work.

The cost is one sign flip inside the projection matrix — a single line, written once, in §5. That is a bargain compared to converting every mesh you ever load.

3The space chain

A vertex makes six stops between the model file and the pixel. Every renderer in this course — the CPU rasterizer in Modules 2–3 and the SDL_GPU renderer from Module 4 — walks exactly this chain, in this order. Figure 2 names each hop and the matrix responsible.

Written as one expression, with our column-vector convention (§6), a vertex reaches clip space like this:

\[ \mathbf{v}_{clip} = P \cdot V \cdot M \cdot \mathbf{v}_{model} \] v_clip = P * V * M * v_model

Read it right to left — the model matrix touches the vertex first, the projection matrix last. That right-to-left reading is a direct consequence of using column vectors; if you ever see an engine write v * M * V * P, it uses row vectors, and its matrices are the transposes of ours.

Naming the hops, and the model matrix's internal order

In prose and diagrams we abbreviate the three matrices M, V, P. In code, name each one a_from_b — the space it produces, then the space it consumes: world_from_model, view_from_world, clip_from_view. Then a product's adjacent labels must match (view_from_world · world_from_model), so a wrong-ordered composition is a spelling mistake you can see before running anything. This convention is introduced and justified in Lesson 2.8 §3.1.

The model matrix itself is always M = T · R · S — scale first, rotate, translate last. The order is derived (Lesson 2.8 §3.2), not conventional: it is the only order that scales an object along its own axes and rotates it about its own centre. Its columns are the object's own axes in world space, scaled by its size, with the position in column 3.

The view matrix (the V above) is view_from_world = inverse(world_from_camera) — a camera is an object with a placement, and looking through it is undoing that placement (Lesson 2.9). Because a camera has no scale, its placement is rigid and the inverse is free: affine(transpose(R), −transpose(R)·eye), no general 4×4 inversion. Transposing the orthonormal rotation puts the camera's right, up, backward axes into the view matrix's rows — the fastest way to read a camera off its matrix. The camera looks down its own −z, so its backward (+z) axis is eye − target, consistent with the world-space −Z-forward convention above.

The projection matrix (the P above) is perspective(fovy, aspect, near, far) (Lesson 2.10). It carries view space to clip space, and its defining trick is that it writes −z — the depth — into w, so the later perspective divide (÷w) shrinks distant things. This is where w stops being 1, where clip space and NDC become distinct (before and after the divide), and — per §5 below — where the right-handed → left-handed handedness flip happens. It targets SDL_GPU's clip space exactly: x, y in [−1,1] and depth in [0,1] (§4), so the software rasterizer and the Module 4 GPU renderer share one projection.

The viewport transform is the final hop, NDC → framebuffer pixels (Lesson 2.11), and it is an engine::viewport mirroring SDL_GPUViewport (x, y, w, h, min_depth, max_depth). It is three independent affine maps — and it is where the +y flip lives. NDC's +y points up; the framebuffer counts rows downward from the top (§4's "+Y up in NDC, +Y down in viewport" note), so y — and only y — reverses. That reversal belongs in exactly one function, viewport::to_screen; a scene rendered upside down is almost always this sign, missing or applied twice.

4Clip space and NDC — fixed by SDL_GPU

Not our choice

Everything in this section is dictated by SDL_GPU and verified against the SDL wiki and headers — see §10. We do not get a vote. Getting this wrong produces an upside-down or depth-inverted image, silently.

After the perspective divide, a vertex lands in normalized device coordinates. SDL_GPU defines that space precisely, and Figure 3 draws it:

  • x runs from −1 (left) to +1 (right).
  • y runs from −1 (bottom) to +1 (top) — +Y is up.
  • z runs from 0 (near plane) to 1 (far plane).

+Y is up in NDC, but +Y is down in viewport and texture space

This looks like a contradiction and is not. They are different spaces with different jobs:

SpaceOrigin+Y directionWhat it addresses
NDCcentreupvertex shader output
Viewporttop-leftdownpixels
Texture (UV)top-leftdowntexels

NDC is the geometric output of your maths. Viewport and UV space are addressing schemes for rectangular grids in memory, and memory rows run top to bottom — hence +Y down. The viewport transform is what bridges them, and the Y flip lives there.

Worked example — NDC to screen, with real numbers

For an 800 × 600 viewport, the transform is:

  • x_screen = (x_ndc * 0.5 + 0.5) * 800
  • y_screen = (0.5 − y_ndc * 0.5) * 600  ← note the minus: this is the flip

Push three points through:

  • Centre (0, 0) → x = (0 + 0.5)·800 = 400, y = (0.5 − 0)·600 = 300 → (400, 300). The middle of the screen. ✓
  • NDC top-left (−1, +1) → x = (−0.5 + 0.5)·800 = 0, y = (0.5 − 0.5)·600 = 0 → (0, 0). The top-left pixel. ✓
  • NDC bottom-right (+1, −1) → x = (0.5 + 0.5)·800 = 800, y = (0.5 + 0.5)·600 = 600 → (800, 600). The bottom-right corner. ✓

NDC's top (y = +1) maps to screen row 0. That is the whole content of "+Y up becomes +Y down", and it costs exactly one sign.

You do not flip Y in your shaders

Vulkan's native NDC has +Y down, unlike D3D12 and Metal. SDL_GPU papers over this: per the SDL wiki, if the backend differs from this convention, SDL converts the coordinate system behind the scenes, so you do not need to perform any coordinate flipping logic in your shaders. Write your shaders once, against the convention above, and let SDL handle the backend. This is one of SDL_GPU's genuinely nice guarantees.

5Where the handedness flip happens

Our world is right-handed. SDL_GPU's clip space is left-handed. Something must reconcile them, and that something is the projection matrix — as Figure 2 marks. This is normal and correct; it is one of the projection matrix's jobs, and hiding it only makes the inevitable "my scene is mirrored" bug harder to diagnose.

With vertical field of view fovY, aspect ratio a = width/height, and near/far planes n and f, let t = 1/tan(fovY/2) and R = f/(n−f). Our projection matrix is:

\[ P = \begin{bmatrix} t/a & 0 & 0 & 0 \\ 0 & t & 0 & 0 \\ 0 & 0 & R & R\,n \\ 0 & 0 & -1 & 0 \end{bmatrix} \] row0 = [t/a, 0, 0, 0]; row1 = [0, t, 0, 0]; row2 = [0, 0, R, R*n]; row3 = [0, 0, -1, 0] where t = 1/tan(fovY/2), R = f/(n-f)

The flip is the −1 in the bottom row. It sets w_clip = −z_view. In right-handed view space the camera looks down −z, so anything in front of the camera has a negative z_view, and negating it yields a positive w — which is what the perspective divide requires. Depth then increases with distance, and increasing-depth-into-the-screen is exactly what left-handed means.

Expect this matrix to differ from the tutorials

Most online derivations (LearnOpenGL especially) target OpenGL's z ∈ [−1, 1] and produce a different third row. Ours targets z ∈ [0, 1] — the D3D-style matrix — because that is what SDL_GPU requires. The t/a and t terms are identical; only the depth row differs. We derive all of this from similar triangles in Module 2, so it will not be a matrix you take on faith.

Worked example — verifying the near and far planes map to 0 and 1

Take n = 0.1 and f = 100. Then R = f/(n−f) = 100/(0.1−100) = 100/(−99.9) ≈ −1.001001, and R·n ≈ −0.1001001.

A point on the near plane sits at z_view = −0.1:

  • z_clip = R·z_view + R·n = (−1.001001)(−0.1) + (−0.1001001) = 0.1001001 − 0.1001001 = 0
  • w_clip = −z_view = 0.1
  • z_ndc = z_clip / w_clip = 0 / 0.1 = 0 ✓ — the near plane maps to 0.

A point on the far plane sits at z_view = −100:

  • z_clip = (−1.001001)(−100) + (−0.1001001) = 100.1001 − 0.1001 = 100.0
  • w_clip = −z_view = 100
  • z_ndc = 100 / 100 = 1 ✓ — the far plane maps to 1.

Both endpoints land exactly where SDL_GPU expects. Notice w_clip came out positive in both cases despite z_view being negative — that is the −1 doing its job.

6Matrices: column vectors, column-major

Two independent choices hide behind the word "convention" here, and conflating them causes endless confusion. Keep them separate:

QuestionOur answerConsequence
Notation — how do we write a transform? Column vectors: v' = M · v Compose right-to-left: P·V·M
Storage — how do the 16 floats sit in memory? Column-major: m[0..3] is column 0 Uploads to HLSL unchanged

A matrix is where the basis vectors land — that is the mental model this course builds in Module 2, and it is what makes column-major storage natural. Each column of M is the image of one basis vector, so storing column-by-column stores "where x went, where y went, where z went, where the origin went," in that order. Figure 4 shows the bytes.

The transpose trap

"Column-major" (storage) and "column vectors" (notation) are independent. Because we chose both, the two effects cancel and our in-memory layout matches HLSL's default — no transpose on upload. Choose only one and you need a transpose somewhere. If a matrix ever produces a transformation that looks like the transpose of what you wanted, this is the first place to look.

7Winding and culling

Counter-clockwise winding, viewed from the front, means front-facing. We cull back faces. Everywhere: in the software rasterizer and on every GPU pipeline.

Winding is not an SDL-wide rule — it is per-pipeline state

Any claim that "SDL_GPU uses counter-clockwise" is wrong. Winding is a field you set on every graphics pipeline you create, via SDL_GPUFrontFace in SDL_GPURasterizerState. SDL has no opinion; we do.

Both relevant enums have their first entry at zero — SDL_GPU_FRONTFACE_COUNTER_CLOCKWISE and SDL_GPU_CULLMODE_NONE — so a zero-initialised rasterizer state means "CCW is front, cull nothing." Forgetting to set cull_mode therefore manifests as no culling at all, not as an error. Nothing warns you; you just quietly pay for the back faces. We set both fields explicitly, every time.

Determining winding is a signed-area computation, and its sign is the same cross product that defined handedness in §2. We derive this properly in Module 2 when we build the edge-function rasterizer, and reuse it for back-face culling in Module 3 — one idea, paid for once, used twice.

In the framebuffer, the sign is reversed — and that is not a contradiction

"CCW = front" is a statement about NDC, where +y points up. The viewport transform flips y on the way to pixels (§4), and flipping one axis reverses orientation. So in framebuffer coordinates a triangle that looks counter-clockwise on screen has a negative signed area, by the usual cross-product formula.

Measured, not assumed: the vertices (5,0) → (0,10) → (10,10), which trace counter-clockwise on screen, give −100; reversed they give +100.

Module 2's software rasterizer therefore does not cull by winding — fill_triangle accepts either and orients its test to match. Culling is a decision made in a specific space for a specific reason, and Lesson 3.4 makes it there.

"1 unit = 1 metre" sounds like bookkeeping until physics arrives. Gravity is −9.81 in metres per second squared; if your units are secretly centimetres, everything falls in a way that looks subtly like slow motion, and no single line of code is wrong. Decide the unit once, at the start, and the whole engine agrees.

Angles in radians is the same discipline. std::sin takes radians; every conversion you skip is a bug you cannot have.

9The simulation contract

Settled across Lessons 1.3, 1.4 and 1.8, and binding from here to the capstone. Every system added later — physics, animation, the ECS, networking — plugs into this shape rather than negotiating its own.

RuleWhy
The frame order never changes: drain events → clk.tick() → in.update() → N fixed steps → render Draining first is what refreshes SDL's state arrays (SDL_PollEvent pumps). Ticking each subsystem exactly once means the whole frame sees one coherent snapshot.
Simulation advances by a fixed h, never by a frame duration. Default h = 1/60 s. A frame-rate-dependent simulation gives different answers on different machines. Enforced by signature: step() takes h, so there is nowhere to pass dt in.
Rendering interpolates between the two most recent states by alpha ∈ [0,1). Trades a lag that swings between 0 and h for one that is always exactly h. Smoothness is consistency, not immediacy.
Interpolate quantities that were continuous before sampling. Nothing else. A position between two samples is meaningful; a score between two values was never true. Lesson 1.8 §4.6.
Never interpolate across a teleport — snap previous = current instead. The two states are not connected by motion, so the lerp draws a streak at a speed the object never had. Whatever teleports must say so.
The simulation is a pure function of (state, input, h). No globals, no clock reads, no hidden RNG — the seed lives in the state. This is what makes replays, deterministic tests, and (much later) lockstep networking possible at all. A single global read destroys it silently.
What "deterministic" does and does not promise

Determinism here means: same binary, same machine, same seed, same inputs, same step size produce a bit-identical run. It is not cross-platform — floating point differs across compilers and architectures through FMA contraction, x87 excess precision, differing libm implementations, and auto-vectorisation. It is not even step-size-independent: change h and the run diverges, because h is an input too. (Measured in Lesson 1.8: the same seed at 60 Hz and 120 Hz diverges within two simulated seconds.)

Sampling a continuous thing is where the artifacts come from

A discrete test of a continuous process is valid only within a bound, and that bound is worth computing and writing down every time. Lesson 1.8's case: an overlap test catches a moving object only while |v|·h < size_a + size_b. The same shape recurs as aliasing on triangle edges (Module 2), texture shimmer (Module 3), and shadow acne (Module 6) — one idea, four artifacts.

10Colour and diagram conventions

Every diagram in this course obeys these. They are as binding as the maths, because a reader should be able to identify an axis by colour alone, in any lesson, with no legend.

MeaningColourSample
x axisred
y axisgreen
z axisblue
The thing under discussionaccent (orange)

This is the same red/green/blue mapping Blender, Unity, and every gizmo you have ever dragged uses. It is not arbitrary — xyz maps to rgb positionally, which is also why normal maps look lavender: a flat normal (0,0,1) encodes to (0.5, 0.5, 1.0). Module 6 pays that observation off.
